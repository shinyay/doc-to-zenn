> **一言で:** 本書は「**トークンとは何か**」を腑に落とし、その上で**なぜ重要か・どこに消えるか・どう健全に保つか・なぜ最適化に意味があるか**までを一気通貫で身体化するための、価格やモデル名に依存しない **first-principles ガイド**です。

---

## この本が存在する理由

LLM（大規模言語モデル）を使ったことがあるなら、あなたはすでに**何百万、ことによると何億**のトークンを消費しています。ChatGPT・Claude・Gemini・GitHub Copilot・社内に立てたオープンウェイトモデル — どれであっても、画面の裏側で**トークン**という単位がカウントされ、それに基づいて課金され、レスポンスが速くなったり遅くなったり、長いプロンプトでエラーになったりしています。

ところが「トークンとは何か」を**ちゃんと**説明できる人は少ない。その結果、

- 請求書が突然跳ねる理由が説明できない
- 「コンテキストを増やしたら賢くなる」と信じてしまう
- プロンプトを「短くする」べき場所と「動かしてはいけない」場所の区別がつかない
- 「キャッシュ割引が効いている／効いていない」が読めない
- アンチパターンを踏んでも気付けない

…といった事故が、現場で**毎日**起こっています。

この本は、そういった事故を**未然に防ぐ語彙とメンタルモデル**を提供します。テクニック集ではありません。「テクニック集を**読めるようにする**ための土台」です。

---

## 本書のスコープ

本書は **トークン中心**（token-centric）に書かれています。「最適化テクニックの完全カタログ」ではなく、

1. **トークンとは何か**（Part I, 第1〜5章）
2. **なぜ重要か** — 容量・お金・時間の3軸（Part II, 第6〜8章）
3. **どこに消えるか** — 1コールの解剖と隠れた消費者（Part III, 第9〜11章）
4. **どう健全に保つか** — 衛生・計測・アンチパターン（Part IV, 第12〜14章）
5. **なぜ最適化に意味があるか** — 戦略的な理由と道筋（Part V, 第15〜16章）

の **5部構成・全16本編** に、付録（用語集・クイックリファレンス）と巻末コラムを加えた **計20章** で構成されています。

最適化の**実装テクニック**そのもの（具体的なプロンプト圧縮ライブラリの使い方、特定の RAG フレームワークのチューニング、特定モデルのパラメータ操作など）は、本書の**範囲外**です。それは別のガイドの役割であり、本書はそのガイドを**批判的に読めるようにする**ための予備学習にあたります。

---

## 想定読者 — 3つの層

本書はあえて**3層の読者**を意識して書かれています。あなたがどの層に属していても、価値を持ち帰れるように設計されています。

### 🌱 はじめての方（newcomer）
- ChatGPT や Copilot を**使ったことはある**が、「トークン」という単語に登場するたびに何となく流していた
- 課金ページや「rate limit」エラーで初めて意識した
- 「subword」「context window」「BPE」と聞いて怯んだことがある

→ 各章の冒頭セクションだけでも読み進められます。専門用語は登場した場所で**毎回**定義します。

### 🛠️ 開発者（developer）
- LLM を使ったプロダクト・社内ツール・エージェントを実装している
- プロンプトの中に何が入っているかは大体知っているが、**トークン単価で見たときの内訳**を即答できない
- 「なんとなく重い」「なんとなく高い」を**測れる言葉**にしたい

→ 各章末の「**Going deeper**」セクションが本領発揮の場所です。コード例は Python (`tiktoken`, `transformers`) と TypeScript で提示します。

### 🧭 意思決定者（decision-maker）
- LLM の請求書にハンコを押す立場、もしくは社内ガードレールを設計する立場
- 「**何が**コストを駆動するのか」「**何が**良い／悪い使い方なのか」「**どこに**ガードレールを置けばよいか」を語彙化したい

→ 第7章（経済学）、第11章（品質 vs コスト）、第13章（計測）、第15章（なぜ最適化が重要か）が中心です。

## 読み方ガイド — 4つのパス

頭から順に読むのが一番自然ですが、目的に応じて以下の経路でも問題なく読めます。各章は**自己完結的**に書かれており、依存先の章へは本文中からリンクしています。

| あなたが… | 読む順序（章番号） |
|---|---|
| 🌱 はじめての方 | 1 → 2 → 4 → 6 → 9 → 15 |
| 🛠️ 開発者 | 2 → 3 → 5 → 7 → 9 → 10 → 12 → 14 |
| 🧭 意思決定者 | 1 → 7 → 11 → 13 → 15 → 16 |
| 🚨 トラブルシュート中 | 14 →（症状に応じて）9 / 10 / 13 へ |

「短時間で全体像」だけ掴みたい方は、**第1章 + 第15章 + 付録B（クイックリファレンス）** の3つを順に読むだけでも、本書の主張のおよそ7割が伝わるはずです。

## 本書の編集方針 — 5つの規律

本書は意図的に**書かない**ことを多く決めています。これは「手抜き」ではなく、**長持ちさせるための設計判断**です。

### 1. 具体的な vendor 価格を書かない
「現在 GPT-X は入力 $3/MTok」のような数字は数か月で陳腐化します。本書では `~3〜5×`、`~1/10`、`数倍程度` といった**形と桁**で書きます。読者が読む時点で価格が変わっていても、**構造は変わっていない**ように。

### 2. 特定モデルのコンテキスト窓サイズを書かない
「GPT-4o は 128K、Claude Sonnet は 200K」のような数字も同様です。本書では「数十万トークンクラス」「数百万トークンクラス」のような帯で表現します。

### 3. 特定モデル名・バージョン名を主役にしない
モデル名は**例として**しか登場しません。本書の主役は「トークン」という単位そのものです。

### 4. 「正解」を1つに決めつけない
最適化はトレードオフの連続です。「これが正解」と書ける場面はほとんどありません。本書は **「いつ・なぜ・何と引き換えに」** を語ることを優先します。

### 5. 出典の引用ではなく first-principles で書く
本書はどこかの記事や本の要約・翻訳・派生ではありません。**ゼロから組み立て直したメンタルモデル**を提示します。同じ結論にたどり着く理由は、根本のメカニズムが万人共通だからです（BPE は一通りしかありません）。

## 各章の構造

すべての本編章（第1〜16章）は、ほぼ同じ構造で書かれています。慣れるとパターン認識で速く読めます。

├───────────────────────────────────────────┤
│ ⑤ 具体例 / 実測 / コード                   │

├───────────────────────────────────────────┤

## 用語の表記について

本書は**日本語**を主体としますが、以下の理由で**英語のまま使う用語**が多数あります。

- `prefill` / `decode` — 訳語が定着していない
- `KV cache` — 訳すと逆に分かりにくい
- `context window` — 「文脈窓」「コンテキスト窓」両方使われるが、本書では **コンテキストウィンドウ** または **window** と表記
- `MTok` — Million Tokens の業界標準略記
- `BPE` / `WordPiece` / `Unigram` / `SentencePiece` — アルゴリズム名は原語
- `prompt` — 「プロンプト」と表記
- `system prompt` / `tool schema` / `chain-of-thought` — そのまま英語
- `always-on tax` / `prompt anatomy` / `heavy tail` — 本書独自の語彙、初出箇所で定義

すべての用語は **付録A（第18章）の用語集** に集約してあります。読みながら「あれ、この単語何だっけ」となったら、付録Aへ。

## サンプルコードと実測値について

本書のコード例は以下の方針で提示します。

- **Python** が中心。`tiktoken`、`transformers` (Hugging Face) を主に使用
- 動かすのに API キーは**不要**（トークン化の確認はローカルで完結する）
- TypeScript / Node.js のサンプルは `@dqbd/tiktoken` を併記する箇所あり
- 実測値（「英語1,000語のテキストは約Xトークン」など）は**目安**であり、tokenizer によって ±10〜30% 程度ぶれます

実測値の表は **本書の執筆時点の代表的な BPE tokenizer**（GPT-4 系の `cl100k_base` 相当）で測定しています。あなたの環境で必ずしも同じ数字が出るとは限らない点、ご留意ください。

```python
# 本書全編で使う最小サンプル
import tiktoken
enc = tiktoken.get_encoding("cl100k_base")
tokens = enc.encode("Hello, world!")
print(len(tokens), tokens)
# 出力例: 4 [9906, 11, 1917, 0]
```

このスニペットを動かせるなら、本書の実コード例はほぼ全て動かせます。

## 本書で**扱わないこと**

期待値調整のため、明確に書いておきます。

- ❌ 特定モデルのファインチューニング手順
- ❌ RAG パイプラインの実装ガイド（chunk size、embedding model 選定など）
- ❌ プロンプトエンジニアリングの「テクニック集」
- ❌ 各 LLM プロバイダの API 比較
- ❌ vector DB / agentic framework のベンチマーク
- ❌ 自社向け推論サーバの構築
- ❌ `tokopt` という測定 CLI ツールの使い方（**巻末コラム（第20章）で予告のみ**、別記事で扱います）

これらはどれも重要なテーマですが、**トークンの正しい理解**を前提とする話題です。本書を終えたあなたは、これらのテーマに進むときに「**何が起きているか**」を地に足をつけて読めるようになっているはずです。

## まとめ — この本のたった1つの主張

長々と書きましたが、本書のメッセージは1つに集約できます。

> **トークンに**流暢になってください**（Be fluent in the unit）**。

流暢になれば、最適化テクニックは**自明**に見えるようになります。流暢でなければ、最適化テクニックは**おまじない**にしか見えません。

おまじないでシステムを設計してはいけません。仕組みで設計してください。仕組みは、トークンから始まります。
