> **一言で:** トークンは、言語モデルが**読み・書き・数え・課金する**単位 — 文字でも単語でもなく、その**中間**にある何か。LLM を扱うなら、最初に身体化すべき**たった1つの単位**です。

---

## なぜこの章があるか

「はじめに」で読者は、本書が**トークンを最初に身体化する**本だと知りました。この章はその身体化の **第一歩** です。意図的に**短く・非技術的**に書きます。メカニズム（サブワード、BPE、UTF-8 のバイト効率…）は第2章以降で順を追って積み上げます。

ここで欲しいのは、本書を読み進める間ずっと「持ち歩ける」**働くメンタルモデル**です。それが手に入れば、後続の章はそのモデルの **詳細な解像度上げ** として読めるようになります。

## 持ち歩くメンタルモデル — 1枚の絵

```
┌────────────────────┐    ┌────────────────────┐    ┌────────────────────┐
│  あなたのプロンプト │ ─► │   モデル            │ ─► │  モデルの返答       │
│  (text in)          │    │  (固定サイズの      │    │  (text out)         │
│                     │    │   "思考パッド")     │    │                     │
└────────────────────┘    └────────────────────┘    └────────────────────┘
         │                          │                          │
         ▼                          ▼                          ▼
   トークンとして               一度に保持できる           トークンとして
   数えられ、課金される         最大トークン数             数えられ、課金される
   多いほど Prefill が          (= context window)         単価は通常入力の
   遅く、入る情報が増える       これを超えると             3〜5×。出力が長いほど
                                何かが落ちる               レイテンシも線形に伸びる
```

ここから読み取ってほしいのは **3つだけ** です。

### 1. **両側がカウントされる**
あなたが送るプロンプトも、モデルが書く返答も、**両方がトークンで測られ**、**両方に課金され**ます。直感に反するかもしれませんが、**出力側のほうが単価は高い**ことが多いです（理由は第7章・第8章で明らかになります）。

### 2. **「思考パッド」には上限がある**
すべてのモデルには **コンテキストウィンドウ**（context window）— 一度に保持できる最大トークン数 — があります。会話がそれを超えると、何かが落ちます。古いメッセージが silently 削除されるか、エラーが返るか、`AI`がフレームワークによって要約されるか — 振る舞いはプラットフォームごとに違いますが、**「無限には乗らない」**という事実だけは共通です。

### 3. **モデルは「文字」を見ていない**
モデルが見ているのは **整数 ID の列**（1トークン = 1ID）です。`Hello, world!` という文字列は、モデルの内部では `[9906, 11, 1917, 0]` のような数値列に変換されてから処理されます。これは訓練時も推論時も同じ。**トークンは表示用の概念ではなく、本当の入出力単位**です。「トークン化」は課金のために後付けされた抽象ではなく、**モデルのネイティブな読み書き単位**なのです。

これだけがメンタルモデルの全体です。残りの15章は、すべてこの絵の **どこかを拡大して説明している** だけ、と読むこともできます。

## 「トークン」が**意味しないもの**3つ

ソフトウェア業界では「トークン」という単語が複数の文脈で使われ、混同されがちです。本書での「トークン」は**以下のいずれでもありません**。

### ❌ セキュリティトークン
API キー、OAuth Bearer トークン、JWT、CSRF トークン — これらは「権限を表す秘密の文字列」です。本書のトークンとは**関係ありません**。同じ英単語が別分野で使われているだけ。

### ❌ コンパイラ／パーサのトークン
プログラミング言語処理系の文脈での「トークン」は、`if`、`==`、識別子、リテラルなど **構文単位** を指します。LLM のトークンはこの古い用法から名前を借りていますが、**単位の切り方も意味も別物**です。

### ❌ 単語・文字・形態素そのもの
本書のトークンは、単語と文字の**中間**に位置する独自の単位です。「1単語 = 1トークン」も「1文字 = 1トークン」も**両方とも誤り**です。

> [!NOTE]
> 本書で **修飾なしに「トークン」** と書いたら、それは常に **「言語モデルが読み・書きする単位」** を意味します。他の意味で使うときは必ず修飾を付けます（「セキュリティトークン」「構文トークン」など）。

## なぜ「文字」でも「単語」でもなく、その中間なのか？

詳しい話は第2章ですが、**1行サマリ**だけ先に。

- **文字を単位にすると**: 同じ文章を処理するのに5〜10倍の長さが必要 → コンテキスト窓を食い潰し、計算量が爆発する
- **単語を単位にすると**: 語彙サイズが百万単位に膨れ上がり、未知語（タイポ・新語・外国語）に対応できない

両方の弱点を**ちょうど中間**で吸収するのが**サブワード**（subword）という単位です。多くの場合は「単語より小さく、文字より大きい」、頻度の高い文字列に1つずつ ID を割り当てます。例えば `tokenization` という単語は典型的に `["token", "ization"]` の **2トークン** に分解されます。`token` も `ization` も多くの英文に頻出するので、独立したスロットを持つ価値がある、というのが BPE 等のアルゴリズムの判断です。

これが「ちょうど良い」と判明したのは経験則ですが、現代のフロンティアモデルはほぼ全てこの方針を採用しています。

## なぜ誰もが**トークン**を気にするのか — 3つのコスト

トークンが重要なのは、**3種類の異なるコスト**がトークンで支払われるからです。

### 💰 コスト1: お金（money）
LLM プロバイダはほぼ例外なく **per-token 課金** を採用しています。請求は使ったトークン数に**ほぼ線形**に比例します。

```
1コールあたりのコスト = 入力トークン数 × 入力単価
                       + 出力トークン数 × 出力単価
                       + (cached がある場合は cached × cached単価)
                       + (reasoning がある場合は reasoning × output単価)
```

「1日 100万コール × 平均 5,000 トークン削減」が、毎日数億トークンの削減 = **大きなお金の話**になります。

### ⏱️ コスト2: 時間（time）
トークンを生成するには、**実時間**（wall-clock time）がかかります。返答が2倍長ければ、生成にもおよそ2倍の時間がかかると思って構いません（第8章で詳述）。

これは UX 上の大事件です。「速い」は「賢い」と知覚されます。レイテンシが 200ms 改善するだけで、ユーザーの行動が変わります。

### 🪣 コスト3: 容量（capacity）
すべてのモデルに **固定サイズのコンテキストウィンドウ** があります。一度に乗せられる量に上限があり、超えたら何かが落ちます。

つまり**トークンは「価格」になる前に「予算」**です。お金の話を一旦忘れても、容量の制約は**避けられません**。

> [!IMPORTANT]
> 「**お金・時間・容量** の3つを **同じ単位で同時に語れる**」 — これがトークンに流暢になるリターンの本質です。CPU 使用率も、メモリ使用量も、ネットワーク帯域も、**1つの単位**で表せたら便利だと思いませんか？ LLM の世界ではそれが**実現しています**。その単位がトークンです。

## 「プロンプト」「リクエスト」ではなく「トークン」で考えるメリット

もう少し具体的に。**プロンプト数で考えるチーム**と**トークン数で考えるチーム**は、同じシステムを違うふうに見ます。

| 観点 | プロンプト数で考える | トークン数で考える |
|---|---|---|
| 「使用量が増えた」 | コール数を見る | 平均トークン × コール数を見る |
| 「コストが高い」 | 高機能モデルだから？ | input か output か cached か reasoning か |
| 「遅くなった」 | サーバーが詰まってる？ | prefill が伸びた？ decode が伸びた？ |
| 「品質が落ちた」 | モデルが劣化した？ | コンテキストが膨らんで lost-in-the-middle？ |
| 「コンテキスト長エラー」 | 長すぎたから減らす | どのセグメントが膨れている？ |

トークンで考えると、**問題が局所化**します。雲のように曖昧だった「重い」「高い」「遅い」が、**測れる量**になります。これは**設計判断の質を1段階上げます**。

## 「トークンはいずれタダになる」論への一言

これは第15章で本格的に扱う論点ですが、先回りして触れておきます。

「モデルはどんどん安くなる」「コンテキストもどんどん長くなる」「だからトークンの話は数年後には不要になる」 — これは**よくある反論**です。

短い反論を先に: **帯域・ストレージ・テレメトリで起きたことと同じ**です。これらすべてのリソースは、過去30年間で**桁単位で安くなりました**。にもかかわらず、誰も「ストレージ最適化はもう不要」とは言いません。なぜか？ **需要は供給の拡大に追従して必ず膨張するから** です。安くなった分、**もっと使う**ようになります。結果として「単価は下がる × 使用量は跳ね上がる」で**請求書は変わらない**か、むしろ増える、というのが歴史の教えるパターンです。

さらに、お金以外の3軸 — **時間（レイテンシ）・容量（コンテキスト）・品質（lost in the middle）** — はいずれも「単価が下がる」では解決しません。**構造の話**だからです。

詳細は第15章まで取っておきます。

## まとめ

- **トークンは LLM の読み・書き・数え・課金の単位**。文字でも単語でもない、独自の単位
- **両側がカウントされる**: 入力も出力も両方に課金され、**出力の単価のほうが高い**ことが多い
- **コンテキストウィンドウは固定**。トークンは**価格になる前に予算**
- **トークンは3つのコストを同時に駆動**: お金・時間・容量
- 「プロンプト数」ではなく「**トークン数で考える**」と問題が局所化し、設計判断の質が上がる
- 「いずれタダになる」論は**部分的にしか正しくない**: 単価は下がっても使用量は膨張する。さらに、構造的問題（容量・品質）は単価では解決しない

- 課金の単位は何ですか？（コール数？トークン数？月額？）
- そのサービスのコンテキストウィンドウのサイズを答えられますか？
- その**根拠**はどこにありますか？（ドキュメント？人から聞いた？うろ覚え？）

**演習 1.2**（10分）
お手元の Python 環境で `tiktoken` をインストールし、以下を実行してください。
```bash
pip install tiktoken
```
```python
import tiktoken
enc = tiktoken.get_encoding("cl100k_base")
samples = [
    "Hello, world!",
    "こんにちは、世界！",
    "def hello_world(): pass",
    "🚀🚀🚀",
]
for s in samples:
    tokens = enc.encode(s)
    print(f"{len(tokens):3d} tokens | {len(s):3d} chars | {repr(s)}")
```
予想と実測がどれくらいズレたか、感覚に焼き付けてみてください。これが第4章までで腑に落ちます。
