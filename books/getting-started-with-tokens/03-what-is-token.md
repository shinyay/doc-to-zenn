---

> **一言で:** トークンとは、LLM が入力を読み・出力を生成し・コストを数える際に**1つの不可分な単位**として扱う**テキストの塊** — 多くの場合は subword（単語の断片）、時に1文字、時に短い単語丸ごと。境界の切り方は **直感的でなく**、必ず**測ること**。

---

## なぜこの章があるか

第1章で「トークンは LLM の読み書き単位」というメンタルモデルが手に入りました。これはひとまずの会話を成立させるには十分です。しかし、

- なぜ200文字の英文と200文字の日本語文で、トークン数が3倍違うのか
- なぜ `tokenization` という1つの単語が、ある時は1トークン、ある時は2トークンに切れるのか
- なぜ絵文字 `🙂` が「1文字」なのに4トークンも消費するのか

を**説明できる**には、メンタルモデルだけでは足りません。この章はそのギャップを埋めます。

「文字でも単語でもない単位」が**なぜ採用されたか**、そして実際の文字列が**どう切られるか**を、コード実例つきで体得していきます。

---

## なぜモデルは「単位」を必要とするのか

ニューラルネットワークは**テキストを直接見られません**。見えるのは**数値**だけ — 正確には、数値ベクトルの並びです。

`Hello, world!` という文字列を処理する前に、それは**整数の列**に変換される必要があります。この整数は、モデルが訓練時に学習した **埋め込みテーブル**（embedding table）の行番号です。各行はベクトルで、その行が表す「1単位の意味」を符号化しています。

```
"Hello, world!" ──[トークナイザ]──> [9906, 11, 1917, 0]
                                        ↓ ↓ ↓ ↓
                                  [埋め込みテーブル]
                                        ↓ ↓ ↓ ↓
                            [vec₁, vec₂, vec₃, vec₄]  ← モデルが見る
```

「単位」をどう定義するかは、**他のすべて**に影響します。

- **シーケンス長** — 同じテキストを処理するのに、いくつ単位が必要か
- **語彙サイズ** — モデルが学習する必要がある単位の総数（埋め込みテーブルの行数）
- **未知入力への頑健性** — タイポ、新語、外国語、コード、絵文字に対応できるか
- **コスト・レイテンシの予測可能性** — 両方が単位数で線形にスケールする

「文字」と「単語」のどちらかにしたくなる — それは**自然な反応**ですが、両方とも**惨めに失敗**します。順に見ていきましょう。

---

## 素朴な選択肢 ①: 1トークン = 1文字

語彙は **数千エントリで足りる**（アルファベット・数字・主要記号・基本的な script で）。`<UNK>`（unknown）の概念は不要 — 入力できる文字なら何でも処理できる。

聞こえはきれいです。実際には**2つの大きな問題**で詰みます。

### 問題 ①: シーケンスが爆発する
中くらいの英語段落は **数百文字 ≒ 数十単語**。文字単位だとモデルの仕事が**5〜10倍**に膨らみます。アテンション機構の一部は**シーケンス長の二乗**でスケールするので、コストはさらに痛い。コンテキスト窓も瞬時に埋まります。

| 単位 | 「Hello, world!」のトークン数 |
|---|---|
| 文字 | 13 |
| サブワード（典型） | 4 |
| 単語 | 2 |

文字単位は、すべてのテキストでこの**5倍前後の重さ**を引き受けることになります。

### 問題 ②: スペリングをゼロから学ばないといけない
`cat` が3つの独立した単位 `c`, `a`, `t` だと、モデルは「この3つの並びが小さな毛むくじゃらの動物を意味する」ことを **共起パターンから1から学ぶ** 必要があります。`meow` `purr` `kitten` の近くに登場する3文字並びがある — そんな統計的ノイズから推論しなければなりません。

人間が小学校で済ませた問題に、モデルが**表現容量を浪費**することになります。

文字レベルのモデルは存在しますし、ニッチな用途で活躍もしますが、**汎用 LLM の主流ではありません**。

---

## 素朴な選択肢 ②: 1トークン = 1単語

逆の極端: **1単語丸ごとが1単位**。`cat` は1トークン。`tokenization` も1トークン。シーケンスは短く、人間の感覚に近い。

問題は **語彙です**。

### 問題 ①: 語彙が爆発し、しかも完成しない
英語だけで数十万語、しかも**毎日増えています**。新製品名、新しいスラング、新しい技術用語、昨日有名になった人の名前。
- 屈折形（run, runs, running, ran）
- 複合語（tokenization, pretokenization, detokenize）
- タイポ（teh, recieve）
- コード識別子（`get_user_by_id`）
- 他の全言語

これらすべてを足したら、ゆうに**100万単位**を超えます。

### 問題 ②: 語彙外は「のっぺらぼう」
純粋な単語レベルのトークナイザは、語彙にない単語をすべて `<UNK>` という特別なプレースホルダで置き換えます。新しい製品名も、誤字も、外国語の借用語も、**全部同じ `<UNK>`** に潰れます。元の文字列の構造（語幹、接頭辞、接尾辞）はすべて捨てられます。

```
入力: "I love using Anthropic's Claude!"
語彙外: "Anthropic's", "Claude"
モデルが見る: [I, love, using, <UNK>, <UNK>, !]
```

### 問題 ③: 一般化しない
`tokenize` を何千回も見たことのあるモデルが `tokenization` を見ると、**全く別物**として扱います。明らかな関連性が**生かされません**。

---

## 妥協: サブワード（subword）トークン

文字は単位が小さすぎる。単語は単位が大きすぎる。**両方の弱点を吸収する中間地点**が、サブワードです。

サブワードトークナイザは、**大量のテキストを観察して**、「どのバイト/文字の並びが、専用スロットを与える価値があるほど頻出するか？」を学習します。

結果として、おおむね以下のような**棲み分け**になります。

- **頻出する短い単語**は1トークン: `the`, `and`, `is`, `to`
- **頻出する長い単語**も1トークン: `because`, `function`, `important`
- **そこそこの単語**は2〜3トークンに分解: `tokenization → ["token", "ization"]`
- **稀な単語**は多くのピースに: `antidisestablishmentarianism → ["anti", "dis", "establish", "ment", "arian", "ism"]`
- **本当に未知の入力**は最終的に**バイト単位**まで落とせる（後述）

```
character-level                 subword                    word-level
   ┌────────────┐         ┌────────────────┐         ┌────────────┐
   │ 小さい語彙 │         │ 中くらいの語彙 │         │ 巨大な語彙 │
   │ 巨大な     │  ◄────► │ 中くらいの     │  ◄────► │ 短い       │
   │ シーケンス │         │ シーケンス     │         │ シーケンス │
   │ UNK 不要   │         │ UNK 不要(byte) │         │ UNK 多発   │
   └────────────┘         └────────────────┘         └────────────┘
                                  ▲
                                  │
                          現代 LLM はここに住む
```

語彙サイズはモデルによって違いますが、概ね **数万〜十数万** が現代の常識です。BPE 系であれば 50,000〜100,000 程度が典型的です。

具体的なアルゴリズム（BPE、WordPiece、Unigram、SentencePiece）は次の第3章で詳しく扱います。本章では「中間地点」という**形**だけを掴んでください。

## 実例で見るトークン分割

> [!NOTE]
> 以下の例は **GPT-4 系で使われている `cl100k_base`** で測定したものです。Tokenizer によって割り方は変わります。重要なのは**個別の数字**ではなく、**パターン**です。

### 例 ①: 短い挨拶

入力:
```
Hello, world!
```

`tiktoken` での分割:
```python
import tiktoken
enc = tiktoken.get_encoding("cl100k_base")
print(enc.encode("Hello, world!"))
# [9906, 11, 1917, 0]
print([enc.decode([t]) for t in enc.encode("Hello, world!")])
# ['Hello', ',', ' world', '!']
```

**4トークン**。気付くべき点:

1. **句読点（`,` と `!`）は独立したトークン**。短くて頻出するので。
2. **` world` の前のスペースは、`world` の一部として吸収されている**。これが現代の subword の重要な慣習です。`" world"`（先頭スペース付き）と `"world"`（スペースなし）は**別のトークン ID** を持ちます。文の先頭と途中で同じ単語が違うふうに分割される理由は、ここにあります。

### 例 ②: 中程度の長さの単語

入力:
```
tokenization
```

```python
print([enc.decode([t]) for t in enc.encode("tokenization")])
# ['token', 'ization']
```

**2トークン**。`token` 単体は頻出。`ization` も英語の頻出接尾辞（organization, civilization, modernization…）。**両方とも独自スロットを持つ価値がある**、という判断です。モデルは「見慣れた語幹 + 見慣れた接尾辞」と捉えるので、word-level トークナイザが捨ててしまった一般化を**保持**できます。

### 例 ③: 長く稀な単語

入力:
```
antidisestablishmentarianism
```

```python
print([enc.decode([t]) for t in enc.encode("antidisestablishmentarianism")])
# ['anti', 'dis', 'establish', 'ment', 'arian', 'ism']
```

**6トークン**（tokenizer によって増減あり）。長く稀な単語は、サブワードが**税金を取る**ところです。語彙にこの単語丸ごとは入っていない。代わりに、見慣れた**形態素**（morpheme）で組み立てます。

### 例 ④: コードスニペット

入力:
```
def hello_world():
```

典型的な分割:
```
['def', ' hello', '_', 'world', '():', ]
```

5〜6トークン。コードについて気付くべき点:

- **アンダースコア `_` は独立トークン**。`snake_case` を多用すると、毎回 `_` の分のトークンを払うことになります
- **`()` は1トークンになることが多い**（空括弧は code で頻出するため）
- **`:` は独立トークン**
- **インデント**（先頭の空白・タブ）も**独自トークン**として扱われる場合があります（4スペースまとめて1トークンになることも）

これが「コードはプロセより token 効率が悪い」と言われる理由です。記号と識別子だらけだから。

### 例 ⑤: 数字

入力:
```
12345
```

可能な分割:
```
["12345"]                — 5桁を1トークンとして持つ tokenizer
["123", "45"]            — 3桁ごとに区切る tokenizer
["1", "2", "3", "4", "5"] — 1桁ずつ
```

数字は**最もばらつく**カテゴリです。一部の tokenizer は 0〜999 や年（1990, 2024）のような頻出数値に専用スロットを持ちます。**算術タスクの精度**にも影響します — `123` を1トークンで持つモデルと、`1, 2, 3` の3トークンで持つモデルでは、数の内部表現が違うからです。

### 例 ⑥: 空白・改行・絵文字

```python
print(len(enc.encode("    "))) # 4スペース → 1トークン
print(len(enc.encode("\n")))   # 改行 → 1トークン
print(len(enc.encode("\n\n"))) # 2連続改行 → 1トークン (頻出なので)
print(len(enc.encode("🙂")))   # 絵文字 → 3〜4トークン
print(len(enc.encode("👨‍👩‍👧‍👦")))  # 家族絵文字（複合）→ 10トークン超
```

絵文字は実は**重い**入力です。1つの絵文字が複数バイトの Unicode シーケンスで表現され、tokenizer はそれを学習済み語彙に持っていなければ**バイト単位まで分解**します。

## よくある誤解（5つ）

ここまでくれば、以下の誤解がなぜ誤解なのか説明できるはずです。

### ❌ 誤解 ①: 「1トークン = 1単語」
✅ **誤り**。短い頻出単語は1トークンになりますが、長い・稀な単語は2〜6トークンに分解されます。**英文プロセでの経験則**は「1トークン ≒ 0.75 単語」ですが、ばらつきは大きく、コード・非英語・記号だらけの文では**この経験則は崩れます**。

### ❌ 誤解 ②: 「1トークン = 1文字」
✅ **誤り**、逆方向に。`because`（7文字）は1トークン。絵文字 `🙂`（1文字）は3〜4トークン。**文字数はトークン数のさらに悪い予測子**です。

### ❌ 誤解 ③: 「トークンはモデル間で同じ」
✅ **誤り**。各モデルファミリーは独自の tokenizer をパッケージしています。**同じ文字列でも、tokenizer 次第でトークン数が大きく変わります**。あるモデルで余裕で収まるプロンプトが、別のモデルでは溢れることがあります。第3章で詳述。

### ❌ 誤解 ④: 「文字列が短いほどトークンも少ない」
✅ **誤り**、特に多言語環境で。日本語・中国語・アラビア語の文は、英語より**少ない文字数**で同じ意味を表せることがありますが、**トークン数は2〜3倍多い**ことがざらです。第4章で詳述。

### ❌ 誤解 ⑤: 「空白を増やしても無料」
✅ **誤り**。スペース・タブ・改行は**それぞれトークンの一部**です。インデント、段落間の空行、整形済み JSON は**実際にトークンを消費**します。1回あたりは少量でも、大規模では効きます。

### ❌ おまけ ⑥: 「モデルは私が打った文字を見ている」
✅ **半分誤り**。モデルが見ているのは**整数 ID の列**です。`Hello` と ` Hello`（先頭スペース付き）は **異なる ID** に変換され、モデルからは**異なるもの**に見えます。プロンプト挙動を本気でデバッグするときは、「モデルが何を見たか」を tokenizer に問い合わせる必要があります。

## Going deeper

ここまでで実用的にはOKです。以下は「**なぜそうなっているか**」を知りたい人向けの追加情報。

### なぜ語彙サイズが「数万〜十数万」なのか

現代の tokenizer の語彙は、典型的に **30,000〜200,000** の範囲に収まります。なぜこの範囲か。

- **語彙が大きいほど**: 1トークンあたりのカバー範囲が広い → シーケンス短い → 推論速い、コンテキスト窓を有効活用。**ただし**、入力埋め込みテーブルと出力射影が肥大化（パラメータ数とメモリ消費）。さらに、稀なエントリは訓練中ほぼ見ないので**学習が進まない**。
- **語彙が小さいほど**: 1単語あたりのピース数が増える → シーケンス長い → 推論遅い、コンテキスト窓きつい。**ただし**、テーブル軽量、各トークンの再利用率が高く**学習が進みやすい**。

「数万〜十数万」は経験的なスイートスポットで、**正解は1つではありません**。モデル設計者の判断によって、別のポイントが選ばれることもあります。

### バイト単位フォールバックと `<UNK>` の死

古い tokenizer は、語彙外の単語を `<UNK>` で置き換えていました。これは**情報損失**です — 元のスペルを復元できない。

**バイト単位の tokenizer**（byte-level BPE）は、これを根絶しました。最小単位を **256通りのバイト値** に取ることで、Unicode のあらゆる文字、絵文字、誤って貼り付けられたバイナリ片 — **何でも表現可能**になります。頻出するバイト並びは大きなトークンにマージされ、稀なものはバイトに分解される。

```
"normal text" → 数トークン
"¡Hola!"      → 数トークン (頻出多言語ならマージ済み)
"中文"        → 数トークン
"🦄"          → 4トークン (バイト分解)
"\xfe\xff"    → 2トークン (生バイト)
```

これが、現代モデルが**任意の入力に grace を持って対応できる**理由です。「拒否される入力」は存在しない — 「**多くのトークンを使う入力**」があるだけ。

### Token ID と Token 文字列の違い

`tiktoken` が `["Hello", ",", " world", "!"]` のように文字列で表示するのは、**人間用**の表示です。内部的にはそれぞれが整数 ID（埋め込みテーブルの行番号）に対応します。**モデルが見るのは ID だけ**。

これには2つの実用的影響があります。

1. **表示形式は隠れた驚きを覆う**: 先頭スペース、不可視結合子、ノーブレークスペース — 見た目同じでも別 ID。
2. **ID はモデル間で互換性がない**: モデル A の ID 1432 と モデル B の ID 1432 は**全く無関係**。「トークン数を共有する」ときは**量の話**であって**識別子の話ではない**。

### 「先頭スペース慣習」の理由

なぜ ` world`（先頭スペース付き）が独自スロットを持つのか。

英語では、ほぼすべての単語が**前にスペースを伴って**現れます（文頭・段落頭を除く）。もしスペースを独立トークンにしたら、文書全体で**スペース1つにつき1トークン**を払うことになり、これは大きな固定税です。

先頭スペースを次のトークンに**畳み込む**ことで、「文中の `the`（` the`）」と「文頭の `the`」を別エントリにできて、**両方とも頻出**なので両方とも価値がある、という設計です。結果、通常の英文で**意味のあるトークン削減**になります。

慣習さえ知っていれば驚かなくなります。

### tokenization は「決定的」だが「非可逆ではない」

同じ入力に対して同じ tokenizer は**同じ出力**を出します（確率的ではない、決定的）。

そして、トークン列を `decode` すると**元の文字列が完全復元**されます（バイト単位フォールバックがあるなら）。**情報損失はない**。これは「`<UNK>` で潰す」古い時代の重要な改善です。

```python
text = "🦄 こんにちは, world! 中文"
tokens = enc.encode(text)
restored = enc.decode(tokens)
assert text == restored  # True
```

## まとめ

- **トークンはサブワード単位**。文字でも単語でもなく、頻度から学習された中間ピース
- 文字単位はシーケンス爆発、単語単位は語彙爆発と `<UNK>` 問題で破綻 — サブワードが**実用解**
- 現代のサブワード tokenizer は**バイト単位フォールバック**で `<UNK>` を根絶した
- 「文字数 → トークン数」「単語数 → トークン数」の予測は**当てにならない**。**測れ**
- **先頭スペース**は次のトークンに吸収される慣習。これを知らないと混乱する
- **トークン ID はモデル間で互換性がない**。「トークン数」を比較するときは**同じ tokenizer で測ったか**を確認する
- 5つの誤解（1単語=1トークン等）は**全て誤り**。本書を読み終わる頃には反射的に否定できるようになる
