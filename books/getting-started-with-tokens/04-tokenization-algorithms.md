> **一言で:** Tokenizer = **アルゴリズム + 訓練済み語彙** のセット。アルゴリズムは大きく **BPE / WordPiece / Unigram** の3種類しかなく、`SentencePiece` と `tiktoken` は**実装/ツールキット**にすぎない。同じ文章が別モデルで違うトークン数になる理由はここにある。

---

## なぜこの章があるか

第2章で「トークンは subword」と分かりました。次の自然な疑問は: **その subword は誰が決めているのか？** 

「BPE」「WordPiece」「SentencePiece」「tiktoken」 — 用語が乱立して混乱しがちですが、**整理すると驚くほどシンプル**です。本章でその地図を作ります。

実装する必要はありません。**仕組みの形**を知っていれば、第5章以降の「数える」「予算を組む」「最適化する」がぐっとクリアに見えるようになります。

---

## あなたが耳にする4つの名前

現代の LLM はほぼすべて、以下のどれかに当たります。

| 名前 | 種類 | 一行説明 |
|---|---|---|
| **BPE** | アルゴリズム | 小さな語彙から始め、頻出ペアを**マージ**していく |
| **WordPiece** | アルゴリズム | BPE 似だが、頻度ではなく**尤度**で選ぶ。BERT 系で有名 |
| **Unigram** | アルゴリズム | **逆方向** — 大きな候補語彙から始め、貢献の少ないものを**削っていく** |
| **SentencePiece** | ツールキット | BPE と Unigram の両方を実装。**空白を文字として扱う**ので CJK・コードに強い |

加えてよく聞く `tiktoken` は **新しいアルゴリズムではない** — GPT 系で使われている **byte-level BPE の高速実装**（Rust 製）です。

つまり、本当に覚えるべきアルゴリズムは **BPE / WordPiece / Unigram の3つだけ**。残りは**実装の詳細**です。

```
                    ┌─── BPE ────────┬─ tiktoken (GPT 系)
       アルゴリズム ─┤                ├─ SentencePiece-BPE
                    │                └─ Hugging Face tokenizers
                    │
                    ├─── WordPiece ──┬─ BERT 系
                    │                └─ DistilBERT / ELECTRA
                    │
                    └─── Unigram ────┬─ T5 / mT5
                                     └─ SentencePiece-Unigram (XLNet, ALBERT)
```

---

## 1. BPE（Byte-Pair Encoding）

最もシンプルで、最も広くデプロイされているアルゴリズムです。GPT 系・LLaMA 系・Mistral 系などフロンティア LLM のほぼ全てが BPE 系。**まず BPE を理解すれば 70% は終わり**。

### 訓練手順（5ステップ）

```
1. 目標語彙サイズ V を決める（例: 50,000）
2. 初期語彙: 全文字 (またはバイト 0〜255)
3. コーパスを単語に pre-split し、各単語を文字列に分解
4. 全コーパス内で**隣接ペア**の出現回数を数える
5. 最頻ペアをマージ → 新トークンとして語彙に追加 → コーパスを書き換え
   → ステップ4に戻る、|語彙| < V の間
```

**「貪欲（greedy）に、最頻ペアを毎回マージ」** — これだけです。

### 手計算ウォークスルー

語彙サイズの目標は気にせず、まず**動かしてみます**。コーパス:
```
low      ×5
lower    ×2
newest   ×6
widest   ×3
```

文字に分解（`_` は単語末尾マーカー）:
```
l o w _              ×5
l o w e r _          ×2
n e w e s t _        ×6
w i d e s t _        ×3
```

**初期語彙**: `{l, o, w, e, r, n, s, t, i, d, _}` (11 entries)

#### Merge 1
全隣接ペアをカウント:
- `e s`: newest×6 + widest×3 = **9**
- `s t`: newest×6 + widest×3 = **9**
- `l o`: low×5 + lower×2 = **7**
- ...

タイブレークでアルファベット順 → `e s` をマージ。

```
l o w _              ×5
l o w e r _          ×2
n e w es t _         ×6
w i d es t _         ×3
```
新トークン `es` 追加。

#### Merge 2
今度は `es t` が **9回**。マージ:
```
n e w est _          ×6
w i d est _          ×3
```
新トークン `est` 追加。

#### Merge 3
`est _` が9回。マージして `est_` を作る。

#### Merge 4
`l o` が7回（low×5 + lower×2）。マージして `lo`。

#### …続けると…

数十回のマージで:
```
low_     ×5    ← 一気に1トークン
lower_   ×2
newest_  ×6   ← 一気に1トークン
widest_  ×3
```

つまり、**訓練コーパスに頻出する単語は丸ごと1トークン**になります。それ以外は subword 分解されたまま残ります。

### 推論時（encode）の仕組み

訓練が終わると、**マージルールの順序付きリスト**（学習順 = 優先度）が手に入ります。新しいテキストを encode するときは、左→右の単純な1パスではなく、**繰り返し優先度の高いペアをマージする**手続きになります:

```
1. テキストを最小単位（文字 / バイト）に分解
2. 現在の系列の中で「学習済みマージ規則の優先度が最も高い隣接ペア」を1つマージ
3. 適用できる規則がなくなるまで 2 を繰り返す
4. 完了 → トークン列
```

ここで重要なのは「順番に適用していく」のではなく「**毎ステップ最優先のペアを1つマージ**」する点です。`tiktoken` のような実装は、優先度キューやトライ／FST を使ってこの反復を**ほぼ O(n)** で回します。

### Byte-level BPE

通常の BPE は「文字」から始めますが、**byte-level BPE** は「バイト」から始めます。違いは:

- **文字 BPE**: Unicode 文字 1 つを単位に。語彙外の文字（珍しい絵文字、稀な script）は `<UNK>` か事前の正規化が必要
- **byte-level BPE**: 256 バイト値を単位に。**任意のバイト列**を表現可能。`<UNK>` は構造的に存在しない

GPT 系（`tiktoken`）は **byte-level BPE**。これが「絵文字を入れても拒否されない」理由です。

## 2. WordPiece

BERT 系で使われる、BPE の**いとこ**です。

### BPE との違い

| 観点 | BPE | WordPiece |
|---|---|---|
| マージ基準 | 最頻ペア | **尤度向上**が最大のペア |
| 単語内の境界マーカー | なし（先頭スペース慣習） | **`##` プレフィックス** |
| 学習の数学 | カウント | 簡易な確率モデル |

### `##` プレフィックスとは

BERT で `tokenization` を encode すると:
```
['token', '##ization']
```

`##ization` は「これは単語の途中である」という印。BERT の語彙では `ization` と `##ization` は**別エントリ**です（前者は単独、後者は単語内の続き）。

これは**情報量の同じ**を別に表現しているだけで、BPE の「先頭スペース付き ` world` vs `world`」と本質的には同じ仕組み — **単語境界の符号化方法が違う**だけ。

### 尤度ベースのマージ

WordPiece は、各マージ候補について「マージ後にコーパスの確率がどれだけ上がるか」を計算し、**最も上がるペア**を選びます。式は:

```
score(x, y) = freq(xy) / (freq(x) × freq(y))
```

頻度が高くても、`x` と `y` が**それぞれ単独でよく出る**ならスコアが下がります。「真に意味のあるペアだけ」をマージしたい、という発想です。

実用上、BPE と WordPiece の出力差はそれほど大きくありません。最近のフロンティア LLM はほぼ BPE に収束しています。

## 3. Unigram

**逆方向のアプローチ**です。

```
1. 大きな候補語彙（数十万エントリ）を用意
2. 各エントリに「確率」を割り当てる
3. コーパスを最も高確率にする分割を選ぶ EM アルゴリズム
4. 各エントリを「もし削除したら全体確率がどれだけ下がるか」で評価
5. 貢献の少ないエントリを削る
6. 目標サイズまで縮小したら終了
```

特徴:
- **同じ文字列を複数の方法で分割可能**（BPE は決定的、Unigram は**確率的**）
- 訓練時に **subword regularization**（複数分割をサンプリング）で頑健性を上げられる
- T5、ALBERT、XLNet などで採用

実用上、ユーザー目線では BPE と区別しづらいですが、**「同じ文字列が違うトークン化を持ちうる」**という性質は知っておくと驚かなくて済みます。

---

## 4. SentencePiece

**アルゴリズムではなく、ツールキット**です。Google が公開しているライブラリ（C++ + Python バインディング）で、以下の特徴があります。

### 1. 空白を「ただの文字」として扱う

通常の BPE/WordPiece は、テキストを**先に単語に pre-split** してから動きます。これは英語のようなスペース区切り言語を前提にした設計です。

SentencePiece は **スペースを `▁`（U+2581 LOWER ONE EIGHTH BLOCK）に置換** して、ただの文字として扱います。結果:

- **CJK**（中国語・日本語・韓国語）、**タイ語**などのスペースを使わない言語で**均等に動く**
- **ソースコード**（インデント、整形済み JSON）を扱いやすい
- 「空白の入れ方」が訓練データと一致しなくても問題ない

### 2. BPE と Unigram の両方を実装

`--model_type=bpe` または `--model_type=unigram` で選べる。LLaMA 系は SentencePiece-BPE、T5 系は SentencePiece-Unigram。

### 3. **言語非依存**

英語前提の単語境界に依存しないため、**多言語モデル**で広く採用されています。

## 5. tiktoken（GPT 系）

OpenAI が公開している **byte-level BPE の超高速実装**です。

特徴:
- Rust 実装（Python から呼ぶ）
- GPT-2、GPT-3、GPT-3.5、GPT-4、GPT-4o などの語彙ファイルを同梱
- **同じファミリーでもモデルバージョンごとに語彙が違う**ので注意

```python
import tiktoken

# モデル名から取得
enc = tiktoken.encoding_for_model("gpt-4")          # cl100k_base
enc = tiktoken.encoding_for_model("gpt-4o")         # o200k_base

# エンコーディング名から直接取得
enc_gpt2     = tiktoken.get_encoding("gpt2")        # GPT-2
enc_p50k     = tiktoken.get_encoding("p50k_base")   # GPT-3 (Codex)
enc_cl100k   = tiktoken.get_encoding("cl100k_base") # GPT-3.5 / GPT-4
enc_o200k    = tiktoken.get_encoding("o200k_base")  # GPT-4o
```

**新世代ほど語彙が大きい**傾向があります（cl100k = 100K → o200k = 200K）。これは多言語・コードへの対応強化が背景です。

## アルゴリズム横断表

```
┌──────────────────────┬───────────────┬─────────────────┬──────────────────────────┐
│ アルゴリズム系統     │ 代表的な実装  │ 語彙サイズの目安│ 特徴                     │
├──────────────────────┼───────────────┼─────────────────┼──────────────────────────┤
│ byte-level BPE       │ tiktoken      │ 数万〜十数万     │ <UNK> 不要 / 多言語対応   │
│ BPE (SentencePiece)  │ SentencePiece │ 数万〜十数万     │ 単一バイナリで前処理込み │
│ WordPiece            │ HF tokenizers │ 〜数万          │ ## prefix / 主に BERT系   │
│ Unigram              │ SentencePiece │ 数万〜数十万     │ 確率分割 / 多言語頑健性   │
└──────────────────────┴───────────────┴─────────────────┴──────────────────────────┘
```

**実モデルとの対応はベンダー次第**: OpenAI 系は `tiktoken` で公開された語彙（`cl100k_base`, `o200k_base` など）、LLaMA 系は SentencePiece ベースの BPE、BERT 系は WordPiece、T5 系は Unigram、というのが代表例です。Claude や Gemini など**詳細を公開していないベンダー**もあり、その場合は**プロバイダの API トークンカウンタ**（例: `count_tokens` エンドポイント）で測るのが唯一の確実な方法になります。

> [!NOTE]
> tokenizer の対応は**モデル世代ごとに変わり得ます**。同じベンダー内でも世代をまたいで `tiktoken` のエンコーディング名が変わる（`p50k_base` → `cl100k_base` → `o200k_base`）ように、最新情報は**各ベンダーのドキュメント**で確認してください。

## なぜこれが請求書の話に直結するのか

**同じ文字列でも、tokenizer によってトークン数が変わる** — これが本書の最も重要な事実の1つです。

実例で確認:

```python
import tiktoken

text = """The quick brown fox jumps over the lazy dog.
これは日本語のテキストです。Tokenization は奥が深い。"""

for enc_name in ["gpt2", "p50k_base", "cl100k_base", "o200k_base"]:
    enc = tiktoken.get_encoding(enc_name)
    n = len(enc.encode(text))
    print(f"{enc_name:15s}: {n:4d} tokens")

# 出力は世代ごとに大きく変わる:
# - 古い世代（gpt2 / p50k_base）はトークン数が多くなりがち
# - 新しい世代（cl100k_base / o200k_base）は多言語効率が改善され、
#   同じ多言語テキストでも数割〜半分以下のトークン数で表現できる
# - 正確な数値は実行して測ること
```

**世代をまたぐと同じテキストでも数倍の差**が出ることもあります。新世代の tokenizer は多言語効率が大幅に改善されています。

これが意味するところ:
- 「**この文章は何トークン？**」という質問は**モデル指定なしには答えられない**
- 同じプロンプトを複数モデルで比較するときは **モデルごとに測り直し**
- **モデル乗り換え時の見積もり**は要注意（旧モデルでは収まっていたプロンプトが新モデルでは溢れることも、逆も）

## Going deeper

### BPE の「空白アンカー」問題と byte-level の救済

文字 BPE は通常、**単語境界（スペース）を超えてマージしない**前提で訓練されます。これは英語では合理的ですが、

- スペースのない言語（中国語など）で機能しない
- ソースコードのトークン化が不自然になる
- 訓練コーパスとデプロイ時で「単語境界の定義」が違うと壊れる

**byte-level BPE** はこの制約を捨て、**バイト並びとしての頻度**だけを見ます。byte-level の本質は「最小単位がバイト」であることであって、空白・改行・インデントの扱いは別の設計選択（pretokenization の有無や正規表現ベースの分割）で決まります。byte-level を採用すると、文字 BPE で必要だった「単語境界の決め打ち」を**前処理側に閉じ込められる**ため、空白も改行もインデントも、頻度に応じて自然にマージされる設計が組みやすくなります。これが**汎化性の本質**。

### Unigram と「subword regularization」

Unigram の確率的分割は、訓練時に **same input, different tokenization** を**ランダムに選択**するのに使えます。これにより:
- モデルは「同じ意味の異なる表現」に対する頑健性を獲得
- 推論時には best-decoding で1つに決められる

実用上、ベンチマーク差は小さいですが、**多言語タスクでの頑健性向上**が報告されています。

### tokenizer の語彙ファイル

`cl100k_base` の語彙ファイルは公開されており、各エントリを直接覗けます:

```python
import tiktoken
enc = tiktoken.get_encoding("cl100k_base")
print(enc.n_vocab)  # 100277

# 特定のトークン ID をデコード
for i in [0, 100, 1000, 10000, 100000]:
    print(f"{i:6d}: {repr(enc.decode([i]))}")
```

**最初のトークン**は記号類、後半に行くほど長い英語フレーズや多言語片が登場します。語彙の構成を眺めるだけでも、**何が「安い」か**の感覚が育ちます。

### special tokens

通常のテキストトークンに加え、tokenizer は **special tokens** を持ちます:
- `<|endoftext|>` — シーケンス終了
- `<|im_start|>`, `<|im_end|>` — chat template の役割境界
- `<|fim_prefix|>`, `<|fim_middle|>`, `<|fim_suffix|>` — code completion 用
- `<|tool_call|>` — function calling 用（モデル系統による）

これらは**通常のエンコードでは出てこない**（特殊な前処理が必要）。Chat API を呼ぶと、内部でこれらが**自動挿入**されます。「自分のメッセージは100トークンなのに、API では125トークン消費していた」のような差は、ここから来ます。

## よくある誤解

### ❌ 「tokenizer はモデルの一部ではない」
✅ **誤り**。Tokenizer は通常**モデル本体の事前学習より前に学習データから fit されて凍結**され、その後その語彙を前提に**埋め込みテーブルと出力層が学習**されます。語彙のサイズ・分布・特殊トークンの配置はモデル設計と密結合し、**勝手に別の tokenizer に差し替えると性能が壊れます**。

### ❌ 「BPE と WordPiece は明らかに違うアルゴリズム」
✅ **大筋は同じ**。マージ基準（頻度 vs 尤度）と境界マーカー（先頭スペース vs `##`）が違うだけで、出力も近似的に似ています。実用上の選択は**そのモデルが何を採用しているか**に従います。

### ❌ 「SentencePiece は速い／遅い」
✅ **アルゴリズムではなくツールキット**なので、速度はオプション次第。`tiktoken` は GPT 系の BPE を**専用に最適化**したのでとても速いですが、これはツールキット間の比較であってアルゴリズム間の比較ではありません。

### ❌ 「`tiktoken` は OpenAI モデルにしか使えない」
✅ 概ね正しい。語彙ファイルが OpenAI のモデル用に紐付いているので、**他社モデルでは正確な数を出せません**。Anthropic、Google、Mistral などは**それぞれ独自の tokenizer**（API か HF tokenizers 経由で取得）。

---

## まとめ

- 現代 LLM のトークナイザは **BPE / WordPiece / Unigram** の3アルゴリズムに集約される
- `SentencePiece` は**ツールキット**、`tiktoken` は**byte-level BPE の高速実装**
- BPE は「最頻ペアを greedy にマージ」 — シンプルで実用的、フロンティア LLM の主流
- WordPiece は BERT 系の親戚。`##` プレフィックスで単語内境界を表現
- Unigram は逆方向（大きな語彙から削る）。確率分割が可能で**多言語に強い**
- **同じ文字列でも tokenizer が違えばトークン数が違う** — 比較・移植の際は必ず測り直す
- 新世代の tokenizer ほど語彙が大きく、多言語効率が良い傾向
