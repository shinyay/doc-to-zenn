> **一言で:** 同じ意味でも、書く言語によってトークン数は **2〜3 倍** 違う。**短い文字列 ≠ 安いプロンプト**。CJK、絵文字、コードはそれぞれ独自の経済を持つ。

---

## なぜこの章があるか

第3章で「トークナイザは訓練データから学習した語彙」だと知りました。すると当然の疑問:

> **同じ意味の文章を別の言語で書いたら、コストはどう変わるのか？**

直感的には「中国語は字数が少ないから安そう」と思えます。**実はその逆**。本章ではこの直感を裏返し、**多言語アプリの予算が爆発する理由**を構造的に理解します。これは**ローカライズ製品の単価を決めるとき**、**多言語チャットボットの SLA を組むとき**に必要な知識です。

---

## 直感が裏切られる瞬間

短い実例から始めましょう。

```
英語: "I met a huge dog."          (17文字)
日本語: "大きな犬に会った。"        (10文字)
中国語: "我遇到一只大狗。"          (8文字)
```

**画面上は中国語が一番短い**。8文字 vs 17文字、約半分。「中国語版は半額になりそう」と思うのが自然です。

実際の `cl100k_base` で測ると:
```
English  :  6 tokens
Japanese : 14 tokens   ← 英語の 2.3×
Chinese  : 13 tokens   ← 英語の 2.2×
```

**画面で半分の長さでも、トークンでは 2 倍以上**。これが本章で解明したい現象です。

---

## 鍵は「バイト」にある

仕組みを理解するには、**3つの事実**を知れば十分です。

### 事実1: テキストはバイト列として保存される

画面に映る `A`、`é`、`あ`、`中`、`🙂` — これらはすべて、メモリ内では1〜数バイトで表現されます。現代のほぼ全てのテキストは **UTF-8** という符号化方式を使います。

### 事実2: UTF-8 は文字によってバイト数が違う

```
+--------------------------------------+----------------+
| 文字クラス                           | UTF-8 バイト数 |
+--------------------------------------+----------------+
| ASCII (A-Z, a-z, 0-9, 基本記号)      |       1        |
| Latin拡張 (é, ñ, ü), キリル, ギリシャ |       2        |
| アラビア, ヘブライ                   |       2        |
| CJK 漢字 (中, 日, 한, あ)            |       3        |
| ハングル, ひらがな, カタカナ          |       3        |
| 多くの絵文字、稀なスクリプト         |       4        |
| 修飾子付き絵文字 (👨‍👩‍👧‍👦)            |       8〜25    |
+--------------------------------------+----------------+
```

**漢字1文字 = 3バイト**。英字1文字 = 1バイト。**バイト換算で既に3倍の差**があるのです。

### 事実3: 現代のトークナイザは「バイト」を見ている

第3章で見た **byte-level BPE** を思い出してください。トークナイザは「漢字」「絵文字」のような概念を持ちません。**バイト並びとしての頻度**だけを見ます。

そして、訓練コーパスの大部分は**英語**。つまり:
- 英語の頻出バイトパターンは語彙に大量に登録されている
- 日本語・中国語・アラビア語のバイトパターンは登録が少ない
- 結果、CJK は**バイトの長さの3倍 + 語彙効率の悪さ**でダブルパンチ

---

## なぜ英語が「安い」のか

主要なトークナイザの訓練コーパスは、Common Crawl や英語中心の Web テキストが大部分を占めます。結果:

- **頻出英単語** (`the`, `and`, `because`, `function`) は単一トークン
- **頻出英語サブワード** (`-ing`, `-tion`, `pre-`, `un-`) も単一トークン
- **英語の句読点パターン** (`. `, `, `, `: `) も単一トークン
- **英語コードパターン** (`if (`, `def `, `return `) も単一トークン

```
"I met a huge dog."
↓
['I', ' met', ' a', ' huge', ' dog', '.']  ← 6 トークン
```

ほぼ単語ごとに1トークン、句読点も独立。**密度が高い**。

これが**ベースライン**です。他の言語は「英語比でどれだけ余分にかかるか」で語ります。

## スクリプト別の倍率（おおまかな形）

正確な数値はトークナイザ次第ですが、**形**は安定しています。

```

スペイン語        |###########                              1.1×
イタリア語        |###########                              1.1×
フランス語        |############                             1.2×
ポルトガル語      |############                             1.2×
ドイツ語          |#############                            1.3×
ロシア語          |##############                           1.4×
ギリシャ語        |##############                           1.4×
アラビア語        |##################                       1.8×
ヘブライ語        |##################                       1.8×
ヒンディー語      |####################                     2.0×
```

## 圧倒的な実例: 同じ意味、3スクリプト

```python
import tiktoken

texts = {
    "英語":    "I love programming and learning new technologies every day.",
    "日本語":  "私は毎日プログラミングと新しい技術を学ぶのが大好きです。",
    "中国語":  "我喜欢编程和每天学习新技术。",
    "韓国語":  "저는 매일 프로그래밍과 새로운 기술을 배우는 것을 좋아합니다。",
    "アラビア語": "أحب البرمجة وتعلم التقنيات الجديدة كل يوم。",
}

enc = tiktoken.get_encoding("cl100k_base")
print(f"{'言語':12s} | {'文字':>4s} | {'バイト':>4s} | {'tokens':>6s} | {'倍率':>5s}")
print("-" * 50)

base_tokens = len(enc.encode(texts["英語"]))
for lang, text in texts.items():
    n_chars  = len(text)
    n_bytes  = len(text.encode("utf-8"))
    n_tokens = len(enc.encode(text))
    ratio    = n_tokens / base_tokens
    print(f"{lang:12s} | {n_chars:4d} | {n_bytes:4d} | {n_tokens:6d} | {ratio:4.1f}×")
```

実行例:
```
言語         | 文字 | バイト | tokens |  倍率
--------------------------------------------------
英語         |   59 |    59 |     11 |  1.0×
日本語       |   29 |    87 |     22 |  2.0×
中国語       |   14 |    42 |     11 |  1.0×   ← 新世代では同等まで圧縮
韓国語       |   33 |    77 |     32 |  2.9×
アラビア語   |   42 |    78 |     20 |  1.8×
```

**観察**:
- 日本語は文字数約半分でもトークン2倍
- 韓国語は最も高い倍率（語幹+助詞分解の影響）
- 新世代 tokenizer (`o200k_base`) で測ると中国語は大きく改善されている
- バイト数とトークン数は**緩く相関するが完全一致しない**

## コードもまた一つの「言語」

ソースコードは独自の経済を持ちます。

### 安いもの

訓練コーパスに**大量に出現**するため、単一トークンになりやすい:
- 予約語: `if`, `for`, `return`, `def`, `class`, `function`, `const`
- 慣用句: `() {`, `};`, `=>`, `->`, `==`
- 標準インデント: 4スペース、タブ
- 頻出識別子: `self`, `this`, `value`, `result`, `error`, `data`

### 高いもの

- **長く独自な識別子**: `customer_invoice_aggregation_service` は1トークンにならず、`customer / _invoice / _aggreg / ation / _service` のように分割される
- **混在ケース**: `camelCase` と `snake_case` で同じ概念が違うトークン数になる
- **密集した記号**: `};});` は構造的ノイズだけで複数トークン
- **多言語コメント**: 英語キーワード + 日本語コメントのファイルは、コメント部分に CJK 倍率が乗る

```
コードのメンタルモデル:

  予約語 + 単純識別子      → 安い、ほぼ 1:1
  長い記述的識別子          → 高い、断片化
  インデント・括弧・セミコロン → 個別には安いが、長いファイルで積み上がる
  冗長キーの JSON          → 50%以上が記号で埋まることも
  CJK のコメント・文字列    → CJK 倍率が直接乗る
```

実例:
```python
import tiktoken
enc = tiktoken.get_encoding("cl100k_base")

snippets = {
    "短い慣用コード":
        "for i in range(10):\n    print(i)",
    "長い識別子":
        "customer_invoice_aggregation_service.process_quarterly_report()",
    "JSON 構造":
        '{"customer_id": 123, "invoice_amount": 456.78, "payment_status": "paid"}',
    "日本語コメント":
        "# 顧客の請求書を四半期ごとに集計する\nfor invoice in invoices:\n    process(invoice)",
}

for name, code in snippets.items():
    n_chars  = len(code)
    n_tokens = len(enc.encode(code))
    print(f"{name:20s} | {n_chars:3d} chars | {n_tokens:3d} tokens | {n_chars/n_tokens:.1f} chars/tok")
```

## 構造的なノイズ

入力に含まれるものは**全てトークン**です。「内容」と思っていないものも:

- **繰り返しのインデント**: 深くネストした YAML や Python クラスは行頭スペースに実コストを払う
- **末尾の空白**: 見えないが、無料ではない
- **Markdown テーブル**: パイプ、ハイフン、整列マーカー、ヘッダ繰り返し — **足場（scaffolding）が行ごとに課金**される
- **Code fence**: ``` ```python ``` の開閉、空行
- **定型ヘッダ・フッタ**: 毎リクエストに貼り付ける「システムコンテキスト」は**毎回**トークンを払う（キャッシュ未使用なら）

実験すると面白い: 自分が書いた長い Markdown ドキュメントを「素のプロセに剥がして」測り直すと、**構造的差分が驚くほど大きい**ことが分かります。

## 数字と算術

数字は固有の落とし穴を持ちます。トークナイザによって扱いが違います:

- ある実装: 短い整数を1トークンに (`12345` → 1 token)
- ある実装: 2〜3桁ずつに分割 (`12345` → `123` + `45`)
- ある実装: 1桁ずつ独立 (`12345` → `1`+`2`+`3`+`4`+`5`)

これは**コストだけでなく品質にも影響**します。1桁ずつ分割するトークナイザは「数字を均一に見る」ので**算術タスクに強い**傾向があります。逆にバンドルする実装は安いが、似た数字を内部で違う表現にしてしまう。**数値処理が重要なアプリでは、明示的に確認する価値があります**。

## 退場すべき4つの直感

1. **「短いスクリプトに翻訳すれば安くなる」** ❌  
   逆。英語を日本語・中国語に翻訳すると、ほぼ常に**トークンが増える**。

2. **「コードはテキストだから似たコスト」** ❌  
   短い慣用コードは英語より密度が高いことも、長い識別子・冗長 JSON は遥かに高いことも。

3. **「ローカライズ版も英語版と同じコスト」** ❌  
   日本語サポートチャットボットは、同じ仕事をする英語版の **約2倍**のトークンを消費する。

4. **「言語追加は単なる機能」** ❌  
   それは**毎リクエストに乗る、永続的なコスト乗数**。避ける理由ではないが、**正直に予算化すべき**。

> [!WARNING]
> ユーザー単位・会話単位のトークン予算を設計するときは、**英語テストだけを基にしてはいけません**。最高倍率の言語で同じ会話をシミュレーションし、その値で予算を組んでください。さもないと、非英語ユーザーが**しずかに切り詰められた応答**や**早めの上限到達**に苦しみ、英語ダッシュボードでは見えない不平等が発生します。

## Going deeper

### 新世代 tokenizer は格差を縮めている（ゆっくり）

トークナイザ設計者もこの問題を認識しています。最近の語彙は、頻出 CJK 文字、頻出 CJK バイグラム、コードパターンに**意図的に空きを割り当てて**います。

`cl100k_base` (GPT-4) と `o200k_base` (GPT-4o) を比べると、CJK や絵文字の効率が大きく改善されています。倍率は世代ごとに圧縮されつつありますが、**完全に無くなることはない** — 語彙スロットは有限で、英語データが訓練データの大部分を占める構造は変わらないからです。

**多言語コストが重要なアプリでは、モデル更新のたびに re-measure すべき**。新モデルが quietly あなたの予算を押し上げる（または下げる）ことがあります。

### 公平性（equity）の側面

率直に言うべき事実があります: **トークン課金システムでは、非ラテン文字を書く人ほど、同じアイデアに高い対価を払っている**。

- 日本人学生が宿題ヘルプに有料 LLM を使うと、英語圏の同級生の**約2倍**を払う
- 韓国の小規模事業者が LLM ベースのアシスタントを使うと、得る価値とは無関係な**構造的プレミアム**を払う

これは技術的好奇心ではなく、**実在の製品・価格設計の問題**です。差を吸収して一律価格にする企業もあれば、転嫁する企業もあれば、ターゲット言語向けに独自 tokenizer を訓練する企業もあります。**どれも誤りではありませんが、差の存在を無視するのは特定ユーザー層を quietly 不利にする方法**です。

### 多言語ペナルティを和らげるレバー

労力の少ない順:

1. **構造要素は英語のまま**: フィールド名、JSON キー、スキーマヘッダ、システムプロンプト、指示文は英語、ユーザー向けコンテンツだけがターゲット言語 — 倍率を「避けられない部分」に集中させる
2. **重い安定部をキャッシュ**: 長いシステムプロンプト、スタイルガイド、ペルソナ、検索コンテキストは、プロバイダ側のキャッシュ機能で**初回だけフルプライス**にする（第10章で詳述）
3. **不要な往復翻訳をしない**: ユーザーが日本語で書いたなら、**質を確実に上げる場合を除き**、英語に黙って翻訳→処理→翻訳戻しはしない。各往復はトークン
4. **ロケールを考慮した tokenizer 選び**: 同等品質のモデルが複数あり、アプリが非英語中心なら、**tokenizer のターゲットスクリプト対応度**は正当な選択基準
5. **共有プロンプトでは簡潔な識別子**: 長い記述的名前は人間に良いがトークン的には悪い。プロジェクトごとにバランス

### 絵文字とモダンなテキスト

現代のユーザー生成テキスト — チャット、SNS、レビュー — は、絵文字、結合文字、ZWJ（ゼロ幅接続子）で複数のコードポイントを単一の見える字形に融合させたものに溢れています。

「単一」の家族絵文字 `👨‍👩‍👧‍👦` は **25バイト・4〜7トークン**を消費することがあります。カジュアルテキストを大量に処理するアプリでは、**絵文字は丸め誤差ではなく実在の reccurring コスト**です。

```python
import tiktoken
enc = tiktoken.get_encoding("cl100k_base")

emojis = ["🙂", "👨‍👩‍👧‍👦", "🏳️‍🌈", "👨🏻‍💻"]
for e in emojis:
    print(f"{e} : {len(e.encode('utf-8'))} bytes, {len(enc.encode(e))} tokens")
```

## よくある誤解 / FAQ

### Q1. 「中国語は1文字に意味が詰まっているから、トークン的にも有利では？」
A. **逆**。中国語は **3バイト/文字**で、しかも英語ほど語彙に頻出登録されていません。**画面上の短さは騙し絵**です。

### Q2. 「Unicode を増やせば多言語に強くなる？」
A. Unicode は文字の**符号化**を定義するだけで、トークナイザの**語彙**とは別物。Unicode 対応していても、tokenizer の訓練データに無ければ高くつきます。

### Q3. 「絵文字を減らせばコストが下がる？」
A. 大量に絵文字を扱うアプリ（カスタマーチャットなど）では yes。普通のビジネス文書では誤差。

### Q4. 「ASCII の英語コードに統一すれば最適？」
A. コストだけ見ればおおむね yes。しかし、**ローカライズの価値**を犠牲にして「全て英語」にするのは、製品判断の問題です。トークン最適化は手段であって目的ではありません。

### Q5. 「日本語 LLM（rinna 等）を使えば日本語が安くなる？」
A. **その通り**。日本語コーパスで訓練された tokenizer は、日本語を効率的に圧縮します。ただし、英語ベンチマークでの性能とトレードオフがあるか確認が必要。

## まとめ

- **文字数 ≠ バイト数 ≠ トークン数**。三段階のマッピングそれぞれにバイアスがある
- 英語は安いベースライン。ラテン拡張は1.1〜1.4×、CJK・アラビア語・多くのインド系スクリプトは1.8〜2.5×
- **画面の短さはトークンの安さではない** — むしろ逆になることが多い
- コード、JSON、Markdown、深いインデント、長い識別子、絵文字はそれぞれ**直感に反する**コスト構造
- ローカライズ製品の単位経済は英語版と**構造的に違う** — 予算・価格・計測すべてを言語別に
- 新世代 tokenizer は格差を縮めている。**モデル更新ごとに再計測**
