> **一言で:** コンテキスト窓は、モデルが**一度の forward pass** で「頭に入れられる」トークンの**共有予算**。**まだ書いていない返事**を含めて、すべてが収まらなければならない。

---

## なぜこの章があるか

第5章で「数える方法」を学びました。次は**何と戦っているか** — トークンが住む**容器**の話です。

コンテキスト窓のサイズは、後の設計判断ほぼすべてを支配します:
- 履歴をどれだけ保持するか
- 検索結果をいくつ詰めるか
- モデルにどれだけ「考えさせる」か
- 余裕を残し忘れたとき何が壊れるか

**1ターン分のノート**だと思ってください。**ページ数は固定**。会話全体、全指示、検索片、**これから書く回答**すべてがその1冊に収まる必要がある。**ユーザーが明示的に使える「追加の作業領域」は無い**（モデルが内部的に reasoning 状態を持つ場合はあるが、それも結局はこの窓の中に収まる）。**モデルの「思考」に乗せたいものは、見えるページに書くしかない**。

---

## コンテキスト窓とは、本当のところ

> モデルが**1回の forward pass** で同時に attend できるトークンの最大数。

「forward pass」は、モデルが見えるすべてを取り込んで、**次の1トークン**を生成する瞬間。そのステップでアクセス可能なトークンは、すべて窓内に収まらなければなりません。

ここから2つの帰結 — 多くの人が混乱する点:

### 1. 窓は**共有**

「あなたのプロンプト用」と「回答用」に分かれていない。**1本のバー、1つの予算**。

### 2. 窓は**1リクエストごと**であって会話ごとではない

モデルを呼ぶたびに、**ノート全体が白紙から書き直される**。モデルは前のターンを「覚えていない」 — クライアント（または プラットフォーム）が**毎回履歴を再送している**。

この2点目が、**請求書の意外性**の主要源です。**長い会話が高いのは「モデルが覚えている」からではなく、毎ターン履歴を再送しているから**。

---

## 窓を共有するもの

窓を**1本のバー**として描き、各セグメントが同じ総予算を取り合う図:

```
┌───────────────────────── コンテキスト窓 ────────────────────────────┐
│ system │ tools │ history │ retrieved │ user │ <-- 出力予算 -->     │
└────────────────────────────────────────────────────────────────────┘
```

順に説明:

- **System prompt**: 常時オンの指示。スタイル、ロール、ガードレール、フォーマット規則。**毎呼び出し再送**
- **Tool / function スキーマ**: ツール呼び出しを許可するなら、各ツールの**定義**（名前、パラメータ、説明）が窓内のトークン。リッチに記述された数個のツールが**気付かぬうちに大きな塊**を消費
- **会話履歴**: 過去のユーザー/アシスタントターン。会話の**線形成長**
- **検索チャンク**: ベクトル DB、データベース、ファイル、検索 API から引いてプロンプトに貼り付けたもの
- **新しいユーザーターン**: ユーザーが今打ったもの
- **出力予算**: モデルがこれから書く回答用の**予約スペース**

最後のが**最も忘れられがち**。回答は窓の外で生成されるのではなく、**同じノートに1トークンずつ書き込まれる**。入力で埋めると、回答用のページがゼロになる。

> [!IMPORTANT]
> **出力は入力と同じ予算を取り合う**。「入力でちょうど収まる」リクエストは、モデルが回答の1トークン目を書こうとした瞬間に**失敗**します。

---

## なぜ天井があるのか

「メモリもストレージも安いのだから、巨大な窓を渡せばいいのでは？」 — もっともな疑問です。

答えは **attention 機構**にあります。各トークンが他のすべてのトークンとどう関係するかを決める部分。標準形では、**全ペアを比較** — 計算量とメモリコストは**シーケンス長の2乗**で増えます。入力を2倍にすると、コストは2倍ではなく**約4倍**。

現代の長コンテキストモデルは、これを和らげる仕掛けを多用:
- スパース attention パターン
- スライディング窓
- 階層的要約
- スマートなキャッシング
- ハードウェア対応カーネル

これらはとても効きますが、**3つの実用的な天井**は残ります:

1. **計算**: 仕掛けがあっても、長いほど1リクエストあたりの処理コストが大きい
2. **メモリ帯域**: モデルは大量の中間状態を限られたメモリチャネルに流す。長コンテキストは**パイプを飽和**させる
3. **精度劣化**: ある長さを超えると、モデルは**窓内にあっても上手く使えなくなる**（後述）

つまり、巨大な窓は存在するが、「**より大きい**」と「**より上手く使える**」は別物。

## ポジションエンコーディング（簡潔に）

トークン単独では順序のないバッグです。「the cat sat」と「sat the cat」を区別するには、**位置情報**が必要。これを **position encoding** が付与します:

- **Absolute**（絶対）: 各スロットに固定の信号
- **Relative**（相対）: 2トークン**間の距離**を符号化
- **Rotary (RoPE)**: 位置に依存する角度でトークン表現を回転 — 汎化性能の数学的に良い性質を持つ

数学を知る必要はなし。覚えるべき**1つの帰結**: **position encoding は特定の長さで訓練されている**。訓練長を大きく超えてモデルを使うと、**機構的には動いても品質が劣化**することが多い。「拡張コンテキスト」機能は助けになるが、**無料アップグレードではなくトレードオフ**。

---

## "Lost in the middle"（中で迷子）

多くのモデルで再現される実証的所見 — **早めに身に染み込ませるべき事実**:

> モデルは長いコンテキストの**冒頭**と**末尾**にある情報には上手く attend するが、**中間に埋もれた情報**は使い損ねやすい。

つまり、**長いコンテキスト ≠ よく使われるコンテキスト**。100個の検索ドキュメントを詰めて、答えが50番目にあると、モデルが見落とす確率が**有意に上がる**。

実用的な含意:

- **重要な指示は無視されない場所**に: 通常はシステムプロンプトの先頭、そしてユーザーの質問近くに**短い再掲**
- 検索チャンクは関連度でランクし、**強いものを両端**に。中間ではなく
- 「ドキュメントを与えたから答えは出るはず」と仮定しない。**与えるのは必要条件、十分条件ではない**

> [!NOTE]
> **長プロンプトの中間は低 attention ゾーン**として扱う。そこに置かなければならないものは、**端の近くで要点を再掲**する。

### 実例: needle in a haystack

「Lost in the middle」を自分で確認するシンプルな実験:

```python
# 大量の関係ない文章 + 真ん中に1文の「秘密」
# 注: 下の `pos` は文字位置（Python 文字列スライス）であってトークン位置ではない。
# 厳密にトークン位置で needle を移動したい場合は、filler を encode してから挿入位置を
# トークン単位で決めること。
needle = "The secret password is BANANA42."
filler = "This is filler text. " * 1000  # 数千トークン
char_positions = [0, 500, 1000, 1500, 2000]  # ← 文字位置として使う

for pos in char_positions:
    haystack = filler[:pos] + needle + filler[pos:]
    prompt = f"{haystack}\n\nQuestion: What is the secret password?"
    # モデルに送って回答を確認
    # → 中央付近で正答率が下がる現象が見られる
```

これを CI に組み込めば、**新モデルへの移行時に「中間の使い方」が改善/退化**したかを定量化できます。

## 窓を超えたとき何が起きるか

**統一された答えはない** — そしてそれが問題の一部です。プラットフォーム（または同じプラットフォームの異なる部分）ごとに**オーバーフロー処理が違う**:

1. **API のハードエラー**: 「prompt too long」と即拒否。**最も友好的な失敗** — 大声で、即座に、デバッグしやすい
2. **Silent truncation**: プラットフォームが古いメッセージを落とす、または片端からトリミングしてリクエストを実行。モデルは**短くされた版**で推論し、**あなたは気付かないかもしれない**
3. **クライアント側スライディング窓要約**: アプリコード（または使用中のフレームワーク）が古いターンを生成要約に置き換え、合計を上限以下に。モデルは**圧縮された履歴**を見る — 同意していない圧縮

各々が**違う種類のバグ**を隠します:
- ハードエラー → 派手に壊れる（直しやすい）
- Silent truncation → モデルが**見ていない指示**と矛盾する回答を生成
- クライアント側要約 → ユーザーが**3ターン前に言った事実**から drift する回答

第15章で**アンチパターン**として再訪します。

## 出力予算は**上限**であって**目標ではない**

モデル呼び出し時に通常、最大出力トークン数を指定します。これを「**モデルはその数字分生成する**」と読みたくなる — **それは違う**。**上限**です。モデルは多くの場合、もっと早く止まる（文が終わった、stop sequence にぶつかった、回答が完成した、と判断した）。

上限について重要な2点:

- **実際に生成された分しか課金されない**: 上限を「数千」にしても、課金は実出力量
- **上限はそのフルサイズ分、利用可能な入力予算を減らす**: モデルが100トークンしか書かなくても、プラットフォームは**事前にスペースを予約**する。その予約は同じ窓から出る

2点目が**意外**な部分。「念のため」に大きい出力上限を設定するのは**無料の安全マージンではない**。**送れる入力量からの実控除**です。「収まるはずのプロンプトでコンテキスト長エラー」が出たら、**最初に出力上限を疑う**。

---

## 予算の感覚を持つ

具体的な数字は変動するので**形**で覚える:

```
仮想モデル: コンテキスト窓 = 128K tokens

┌────────────────────────────────────────────────────────────┐
│ システム + tools (固定):       2K   ( 1.5%)                │
│ 短い履歴 (10 ターン):          5K   ( 3.9%)                │
│ 長い履歴 (50 ターン):         25K   (19.5%)                │
│ RAG コンテキスト (10 docs):   15K   (11.7%)                │
│ ユーザーメッセージ:            1K   ( 0.8%)                │
│ 出力上限予約:                 8K   ( 6.3%)                │
│ 安全マージン:                 13K   (10.2%)                │
│ ─────────────────────────────────                         │
│ 残り (使えない):              59K   (46.1%)                │
└────────────────────────────────────────────────────────────┘
```

上記は仮想ですが、**「半分以上を使い切ったら危険水域」**が一つの目安。**注意の質も劣化** — 単にエラーが出ないだけで「正しく使えている」わけではない。

## Going deeper

### KV キャッシュと、なぜ長コンテキストの **decode** が**メモリ束縛**か

LLM 推論には性質の違う2フェーズがあります:

- **Prefill**（プロンプト全体を一度に処理）: 大量の行列積を並列で回せるので **compute-bound**（GPU の演算性能で律速される）
- **Decode**（応答を1トークンずつ生成）: 各ステップで KV キャッシュを読み出すコストが支配的になり **memory-bound**（メモリ帯域で律速される）

モデルが回答を1トークンずつ生成するとき、毎ステップで attention 計算を最初からやり直しはしません。代わりに、**既に見たトークンの K（Key）と V（Value）の中間結果**を **KV-cache** に保存。新しい各トークンはこのキャッシュ + 自分自身に attend するだけ。

KV-cache は**速いが大きい**。サイズはシーケンス長 × モデル内部次元で増える。**長コンテキストの decode** での支配コストは**生 compute ではなく、このキャッシュをメモリに通す速度** — これが「memory-bound」の意味で、**decode 側の話**です。

これが、プロバイダが**キャッシュ済み入力トークン**に割引を提供できる理由でもある: 共有プレフィックスの prefill 結果（K/V）が既にメモリに乗っていれば、再構築より**本当に**安い。

### スライディング窓 attention vs フル attention

標準 attention は全トークンが全トークンを見る。**スライディング窓 attention** は、各トークンを**最近の固定数の隣接**だけに制限。これで2乗コストを破壊 — グローバル認識を捨てて、**スケーラビリティ**と引き換え。

多くの長コンテキストモデルは両者を組み合わせ: 一部の層ではフル、他では窓 — グローバル情報が**フル価格を払わずに**伝播するように。

トレードオフは実在: 純粋な窓モデルは機構的には超長シーケンスを扱えるが、窓の遠端からの情報は**中間トークンを介して跳ねて**現在に届く必要があり、信号は途中で劣化。「Lost in the middle」現象の一因。

### 長コンテキストモデル変種のトレードオフ

「長コンテキスト版」のリリースは、通常**同じモデルに大きな数字を付けた**ものではない。多くは**異なる position encoding、異なる attention パターン、またはその両方**で再訓練/ファインチューニングされ、長さで上手く振る舞うように調整。

短い兄弟と比べて:
- 1トークンあたりが遅い
- 一部の短コンテキストベンチで弱い
- サーブコストが高い

正しい選択は**ワークロード次第**で、「数字が大きいから良い」ではない。

### 窓を2倍にしても価格は2倍にならない

価格はトークンあたりで、**実際に送受信した分だけ払う**。窓が2倍のモデルが自動的に1呼び出しあたり2倍ではない — **追加の余地を実際に使った分だけ**。窓サイズは**容量**、価格は**消費**に従う。

ただし、極長リクエストは「同じものをもっと」ではない。**memory-bound 部分を多く使う**ので、一部のプロバイダは**長さで入力単価を変える**、または**キャッシュ済み vs 未キャッシュ**で違う料金を取る。第7章で経済を本格的に扱います。

## よくある誤解 / FAQ

### Q1. 「128K のモデルなら 100K のドキュメント全部入れて要約できる？」
A. **入力としては入れられる**が、**出力予算と「lost in the middle」**を考えると非推奨。チャンクして要約 → 階層的に再要約する設計の方が品質も予測性も高い。

### Q2. 「履歴を全部送らないとモデルが文脈を失うのでは？」
A. その通り。だから設計判断: 全履歴を送る（高コスト）vs 要約済み履歴を送る（劣化リスク）vs 必要な過去だけ抜粋（中庸）。**会話の長さに応じて切り替える**戦略が一般的。

### Q3. 「『コンテキスト窓 1M』のモデルは無敵？」
A. **窓のサイズと有効使用は別物**。長くなるほど「lost in the middle」「劣化」「速度低下」「コスト増」が効く。自分のユースケースで「needle in haystack」テストを実行して有効長を**実測**する。

### Q4. 「`max_tokens=4096` を設定したのに、3000 で打ち切られた」
A. 上限は**最大値**。モデルは EOS を出した時、stop sequence に当たった時、十分と判断した時に止まる。打ち切られたなら、**プロンプト側の指示**を疑う（「200字以内で」と書いていないか、Few-shot 例が短いか、など）。

### Q5. 「窓を増やすと品質が落ちると聞いたけど本当？」
A. **タスク次第**。短い質問なら窓サイズは無関係。長コンテキスト推論（複数ドキュメント横断、長い履歴）では、訓練長との乖離・「lost in the middle」で劣化する**ことがある**。新モデル乗り換え時は、**自分の代表タスクで**比較。

---

## まとめ

- コンテキスト窓は **system + tools + history + retrieved + user + 出力** の**共有予算**
- **出力は入力と取り合う**。`max_tokens` で予約した分は、入力で使えなくなる
- 天井は attention の **2乗コスト**と、訓練長を超えた**精度劣化**から来る
- **「Lost in the middle」**は実在: 重要なものは**端**に、中間に埋めない
- オーバーフロー時の挙動は**プラットフォーム依存**: ハードエラー、silent truncation、クライアント側要約 — 各々違うバグを隠す
- 窓サイズは**容量**、価格は**消費**
