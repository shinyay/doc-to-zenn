> **一言で:** トークンは**計算の単位**であると同時に**価格の単位**。input/output の非対称、キャッシュ済みプレフィックスの割引、reasoning トークンの隠れコスト — この3つを理解すれば、請求書は不思議でなくなる。

---

## なぜこの章があるか

ここまでトークンを**技術的な量**として扱いました。本章ではレンズを切り替えます。トークンは**請求書の単位**、**予算の単位**、**キャパシティ予測の単位**でもある。具体的な単価は数ヶ月で変わりますが、**価格の「形」**は安定しているので、それを身体化します。

1つだけ覚えるなら:

> **モデルの請求書はリクエスト数ではなく、リクエストを通じて流れた**トークン数**で決まる。そしてすべてのトークンが同じ単価ではない**。

---

## 価格の単位: per million tokens

ホスト型モデルプロバイダはほぼ全員、同じ形で価格を提示します: **per million tokens（MTok 単位）**。任意のモデルに対して**2つのレート**が並びます:

- **input レート**: モデルに送る各トークンの単価（system + history + retrieved + user + tools + 添付ドキュメント）
- **output レート**: モデルが生成する各トークンの単価

per token ではなく per million なのは**読みやすさのため**（per token だと先頭ゼロが多すぎる）。基本式:

```
cost_for_a_call = (input_tokens  / 1_000_000) × input_rate
                + (output_tokens / 1_000_000) × output_rate
```

これがキャッシュなし・reasoning なしの単一呼び出しの**全式**。本章の残りはこの基本式の修正子です — 一部の入力に効く割引、出力に隠れた割増、多数呼び出しでの集約効果。

> [!NOTE]
> モデルを比較するときは常に**ペアで**比較する。input が安く output が高いモデルは、ワークロードの形次第で魅力的にも不利にもなる。

---

## input/output の非対称

価格に関する**最重要事実**、初心者を最も驚かせる点:

> **output トークンは input トークンの数倍高い**。倍率はプロバイダ・ティア次第で、業界全体で**3〜5×** が一般的。フロンティアモデルではさらに大きいことも。

なぜか？ 恣意的な価格設定ではなく、**裏の compute の反映**です。

- **input は並列処理**: 10,000 トークンのプロンプトを送ると、モデルは**1回の forward pass** で全部消費 — 効率的にバッチ化された行列演算1つで。GPU が密に使われるので**トークンあたりが安い**
- **output は autoregressive（逐次）**: 各 output トークンは**モデル全体を通る独自の forward pass** が必要。先に生成したトークンに依存。「未来の並列化」は不可能 — トークン1を生成し、それに条件づけてトークン2を生成し、トークン1, 2に条件づけてトークン3を生成 — と続く

```
INPUT (並列、トークンあたり安い)         OUTPUT (逐次、トークンあたり高い)

  [t1 t2 t3 t4 t5 ... tN]                 [o1] -> [o1 o2] -> [o1 o2 o3] -> ...
   \  \  \  \  \      /                    ^         ^            ^
    \  \  \  \  \    /                     |         |            |
     >>> 1 forward pass <<<            1 fwd     1 fwd        1 fwd
                                        pass      pass         pass
```

100万トークンを**読む**のは1つの大きな安いオペレーション。100万トークンを**書く**のは100万回の逐次の高いオペレーション。

### 実用的含意

この非対称はプロンプト編集の判断基準を変えます:

- system prompt に **100 input トークンを足す**変更（小さな指示、明確化の例） — **ほぼ無料**。毎呼び出しで走っても、生成される output に比べたら微々たる増分
- 全応答に **500 output トークンを足す**変更（「推論を段階的に説明して」「常にサマリを含めて」） — **本当に高い**。output は高い方の側で、追加された各文は永続的に毎呼び出し高レートで支払う

教訓: **読ませることには寛大でも、書かせることには質素であれ**。

## キャッシュ済みトークン: 安定プレフィックスの割引

現代のホスト型 API のほとんどが **prompt caching / prefix caching** をサポート。

仕組み: モデルが入力を処理するとき、**Transformer 各層の K/V cache**（内部表現）を構築。**全く同じプレフィックスで始まる別のリクエスト**が来たら、プロバイダはこの内部表現を**再利用**できる — ゼロから再計算する代わりに。

その節約をユーザーに還元するため、**キャッシュ済み input トークンは大幅割引** — 多くの場合**フルレートの ~10%**（プロバイダにより varies）。**桁としては正しいメンタルモデル**。

```
fresh input  :  ████████████████████  (full)
cached input :  ██                    (~10%)
output       :  ████████████████████████████████████████████████████████████  (3-5×)
```

### 「同じプレフィックス」の意味

キャッシュは通常**プレフィックスキー**: 新しいリクエストの先頭が、過去リクエストの先頭と**バイト単位（またはトークン単位）で一致**している限りヒット。**乖離した瞬間**、それ以降はすべて fresh 扱い。

プロンプト構造への含意は巨大:

- **安定コンテンツは前**: system prompt、tool 定義、永続ロール記述、ターン間で変わらない大きな参照ドキュメント — すべて入力の**先頭**へ
- **揮発コンテンツは後ろ**: ユーザーの最新メッセージ、最新ツール結果、ターン毎に変わるもの — 末尾に置き、上の長く高価なプレフィックスを**キャッシュ可能に保つ**

毎呼び出しで同一の 50,000 トークンの system prompt は、定常状態で、毎呼び出し regenerate される 50,000 トークンの system prompt より **桁違いに安い**。トークン数は同じでも、請求では1桁近い差。

> [!TIP]
> プロンプト再設計時に問うべき: 「**1行動かしてキャッシュ可能プレフィックスを壊していないか？**」 上部の1文を並び替えるだけで、その下のすべてのキャッシュが無効化されることがある。

### キャッシュ可能プレフィックスのパターン

```python
# 良い例: 安定 → 揮発
messages = [
    {"role": "system", "content": LARGE_STABLE_SYSTEM_PROMPT},  # キャッシュされる
    {"role": "system", "content": REFERENCE_DOCS},              # キャッシュされる
    *conversation_history,                                      # 一部キャッシュされる
    {"role": "user", "content": new_user_message},              # 揮発
]

# 悪い例: タイムスタンプを毎回先頭に入れてキャッシュを無効化
messages = [
    {"role": "system", "content": f"Current time: {now()}\n{LARGE_SYSTEM_PROMPT}"},  # 毎回違う！
    ...
]
```

時刻のような**揮発情報を上部**に入れると、その下の長いプレフィックスが**すべて再計算**される。タイムスタンプが必要なら、**末尾**に入れるか、**ツール経由で取得**させる。

## Reasoning トークン: 見えない出力

新しい複雑性: 多くの新しいモデルは**回答前に「思考」**するように設計されている。内部で reasoning トークンのストリーム（**プライベートな scratch pad**）を生成し、それが完了してから可視応答を生成。

ユーザーから見えるのは**最終回答だけ**かもしれない。請求の観点からは、**隠れた reasoning トークンは実トークン**で、autoregressive にフル output コストで生成される。**ほぼ常に output トークンとして課金** — ユーザーには表示されないのに。

```
ユーザーに見える:    [final answer tokens]
output として課金:   [reasoning tokens] + [final answer tokens]
                     \________________/
                      見えないが、払っている
```

「reasoning effort」「thinking budget」のつまみを露出するモデルでは:

- **低 effort**: 数百トークン程度の reasoning trace — 可視回答への小さな割増
- **高 effort**: **数千〜数万トークン**の reasoning trace — 可視回答自体より遥かに大きいこともある。難問では reasoning が**output コストを支配**

実用的帰結:

1. 「reasoning モデル」のコストを測るときは**reasoning トークンを必ず含める**。可視回答だけ数えると**請求を大きく過小評価**
2. 「念のため」reasoning effort を上げるのは、**気付かず output コストを倍々にする**最も簡単な方法。reasoning effort は**タスクごとに調整するつまみ**であって、デフォルトを上げるものではない

## モデルファミリ内のティア

ほとんどのプロバイダが、**性能と価格の段階**でモデルファミリを公開:

| ティア | 典型用途 | トークンあたり相対価格 |
|---|---|---|
| 🟢 Small / cheap | 分類、抽出、ルーティング、簡単なドラフト | 最低 |
| 🟡 Mid-tier | 一般チャット、要約、多くのアプリ | 中間 |
| 🔴 Frontier | 困難な推論、複雑なコード、高ステークス判断 | 最高 |

最安と最高ティアの差は、**多くの場合1桁、時にそれ以上**。フロンティア reasoning モデルは、思考トークンの傾向を考慮するとさらに高くなる。

これが「**常にベストモデルを使う**」が滅多に正解ではない理由。単純で繰り返しの多いコールが支配するワークロードは、**安いモデルで 1/10 のコスト**で（測定可能な品質低下なしに）走れることが多い。少数の難判断が肝のワークロードは、**それらだけフロンティア**、残りは安いティア — このパターン（**リクエスト種別をティアにルーティング**）は本書全体で繰り返し登場します。

## 2つの課金スタイル

トークンコストが裏側の現実ですが、ユーザーには**2つの違う形**で届きます。

### 1. Pay-per-token（API 直、使用量ベース）

期末に、消費した input / cached / output の**正確な数 × 各レート**で請求。月額固定なし、quota なし、上限は自分で設定するしかない。

- **Pros**: 線形、透明、モデル化容易。トラフィック2倍 → 請求2倍
- **Cons**: 自然な天井がない。**暴走ループ**、**設定ミスのエージェント**、**バイラル製品**で前日の桁違いの請求が、見るまで分からない

ほぼ全ての直接 API オファリング。これに乗るなら**自前の可観測性と予算ガード**が必須（第14章）。

### 2. Request-based / quota-based（サブスク、リクエスト単位）

エンドユーザーが見るほぼ全て。期間あたり flat な額（月額サブスク、シートライセンス、含まれた割当）を払い、引き換えに「リクエスト」「メッセージ」「premium requests」「クレジット」を得る。

- **Pros**: 買い手にとって予測可能、上限あり、予算化容易
- **Cons**: **課金単位（リクエスト）と消費単位（トークン）に固定関係なし**。1リクエストが1行の質問のことも、50ターンのエージェントセッションで内部で数百万トークン燃やすことも。quota の中からは**区別できない**

```
pay-per-token:                   quota:

bill = Σ tokens × rate           bill = flat_monthly_fee
                                 allowance: N "requests" / month
コールごとに予測可、              月単位で予測可、
月単位で青天井                    コール単位で不透明
```

実製品の多くは**両方を組み合わせ**: 無料/込みのリクエスト割当 → 超過分はトークンレートで課金。**ビルダーは両側を理解**する必要がある — ユーザーが見るもの (quotas / requests) と、プロバイダが請求するもの（トークン）。

### Free-tier と fallback UX

裏のコストがティアごとに大きく違うため、製品はしばしば:

- **「Auto」モデル選択**: システムが各リクエストを「成功する見込みのある最安モデル」に黙ってルーティング
- **Fallback chain**: 安いモデルで失敗したリクエストを、より強いモデルでリトライ
- **予算枯渇時の graceful degradation**: フロンティアから mid-tier に切り替え、リクエストを拒絶しない

これらは**UX 選択**ですが、本章の経済学に完全に駆動されています。**どのティアがどのリクエストを処理するか**を真剣に考えない製品は、過剰支出するか、提供不足になる。

## 計算例

シナリオ:
- system + tools = **5,000 tokens**
- 全コールに添付の reference doc = **45,000 tokens**
- セッション中に **100 ターン**会話
- 平均ユーザーメッセージ **200 tokens**、平均回答 **300 tokens**

input コストだけ追いかける、まず**キャッシュなし**:

各ターンで送るもの: 5,000 (system) + 45,000 (doc) + これまでの会話 + 新メッセージ。会話の伸びを無視しても、**50,000 トークンの常時オンコンテキストを 100 ターン送る**:

```
50,000 tokens × 100 turns = 5,000,000 input tokens
```

これは**500万トークンの単一リクエストと同じ input 請求**。**繰り返しがコスト**。500万トークン打ったわけではなく、5万トークンを1回打って99回再送した結果。

キャッシュをオンに。50,000 トークンのプレフィックスはターン2以降同一なので:

- ターン1: 50,000 fresh input
- ターン 2-100: 50,000 **cached** input + 新メッセージのわずかな fresh

cached がフルレートの 1/10 とすると、99 ターンの input 請求は **約 90% 削減**。output 請求は不変（キャッシュは input のみ）。**支配コストだった input 側が劇的に縮む**。

長セッションの中心経済学的洞察:

> キャッシュなしの世界で本当に払っているのは、**同じコンテキストを何度も読み直す行為**。キャッシュはまさにこのコストを破壊するための機構。

## Going deeper

### Volume discount と committed spend

意味のあるスケールに達すると、**公示価格は実払価格ではなくなる**。プロバイダは一般に:

- **Volume discount**: 月次しきい値を超えると per-MTok レートが下がる
- **Committed-spend agreement**: 期間にわたる最低月額の保証と引き換えに割引

公示価格は**天井**であって**床ではない**。大規模ワークロードを予測するなら、**交渉前提のレート**で計画。

マルチベンダー戦略の隠れコスト: トラフィックを分散すると、**どのベンダーでも volume tier に届かない**ことがある。

### リージョン別の価格差

一部プロバイダで、per-MTok レートが**呼び出すリージョンに依存**。差は通常控えめだが非ゼロ。**データレジデンシー要件**と相互作用する: 最安リージョンが**使用許可されたリージョンとは限らない**。リージョン選択は**コストとコンプライアンスの結合判断**として扱う。

### Hosted vs self-hosted の経済学

- **Hosted**: $/token（**消費した分だけ**）
- **Self-hosted**: 本質的に **$/GPU-hour**（**ハードを持っている時間**）

これらは同じ通貨ではない。換算は思うより難しい。

```
effective_$_per_token = (gpu_hours_consumed × $_per_gpu_hour) / tokens_processed
```

Self-hosted は**よく利用されている時のみ**トークンあたり安い。10% 利用率では、有効 per-token レートが hosted を**大きく超える**。これが、self-hosting が**持続的高ボリューム**でしか経済的に勝たない理由。

### キャッシュ込みの**有効レート**

キャッシュが絡むと、**公示 input レートはもはや実 input レートではない**。本当に追跡したい数字は **effective input rate**:

```
effective_input_rate = (fresh_input × fresh_rate + cached_input × cached_rate) / total_input
```

大きな安定プレフィックス + 小さな揮発尾部のワークロードでは、定常状態の有効レートは**フルレートの 15-25%** に落ちることがある — **公示価格だけ見ていると見えない構造的節約**。逆に、毎ターンプレフィックスが変わるワークロードでは、価格表に何と書いてあろうと**有効レートはフルのまま**。

プロバイダ・モデルを実ワークロードで比較するときは、**自分のアクセスパターン**での有効レートを計算 — 公示じゃなく。

## よくある誤解 / FAQ

### Q1. 「input が安いから、巨大コンテキストを毎回送って output を短くすれば最適？」
A. キャッシュが効くなら正解に近い。ただし**キャッシュ可能性**を壊さない設計が前提。安定プレフィックスを上、揮発を下、というルールを死守。

### Q2. 「reasoning モデルで `reasoning_effort=low` にすればコストが下がる？」
A. **下がる**。ただし**精度も下がる**ことが多い。タスク種別ごとに**最低必要 effort** を実験で見つけるのが王道。

### Q3. 「キャッシュは何分間有効？」
A. プロバイダにより 5分〜24時間。**頻繁にアクセスされる**プレフィックスは延長されることが多い。長時間アイドルなプレフィックスは破棄。

### Q4. 「output rate が高いから、JSON で簡潔に返させれば安い？」
A. **概ね正解**。JSON 構造分（`{`, `}`, `"key":` などの記号）を払う代わりに、**冗長な散文の節約**になる。ただし JSON のキー名が長いと相殺される（`"customer_invoice_id"` vs `"id"` の差）。

### Q5. 「reasoning トークンを `usage` で確認できる？」
A. プロバイダによる。OpenAI o-series は `usage.completion_tokens_details.reasoning_tokens` で見える。Anthropic extended thinking も同様の breakdown。**監視に組み込む**べき。

## まとめ

- 価格は **per million tokens** で公示、input と output で別レート
- **output は input の 3-5× 高い** — 並列 vs 逐次の compute 反映
- **キャッシュ済み input は大幅割引**（~10%）。安定プレフィックスは揮発と桁違いに安い
- **reasoning トークンは output 課金**（見えないのに）。難問では支配的になりうる
- ティア間価格差は**しばしば1桁以上**。簡単な仕事を安いモデルにルーティングは**最高レバレッジの最適化**
- 2つの課金スタイル: pay-per-token（線形、青天井、透明）と quota（flat、上限、不透明）
- 長セッションのコストは**何を言ったか**より**同じコンテキストを何回再送したか**で決まる。キャッシュはこれを破壊する機構
- スケールでは **volume discount、リージョン、hosted vs self-hosted、有効レート**を直視
