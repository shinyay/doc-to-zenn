> **一言で:** トークンは金だけでなく**時間**も食う。モデルが各 ms をどう使うかを理解することが、「**瞬時に感じる**アプリ」と「**壊れて感じる**アプリ」の違いを生む。

---

## なぜこの章があるか

第7章でトークンを**請求書の項目**として扱いました。本章では別の側面: トークンは**ハードが実際に行う仕事の単位**でもある。アプリが送受信する各トークンは、どこかの GPU で**壁時計時間**を消費する。それが:

- ユーザーへの**レイテンシ**
- バッチジョブの**スループット制限**
- 同時接続数の**天井**

として現れます。**金額がほぼゼロのレスポンスでも、間違った頼み方をすれば遅く感じる**。**金額が高いレスポンスでも、上手くストリームすれば軽快に感じる**。金と時間は相関するが**同じ通貨ではない**。

---

## 2つのフェーズ、2つのコスト形

推論は**2つの異なるフェーズ**で起こります。外から見ると似ている（トークンが入って出る）が、**性能プロファイルが全く違う**。本章で1つだけ覚えるなら、この2つの曲線の形を覚えてください。

### Prefill — プロンプトを読む

プロンプトを送ると、モデルはまず**全体を読み込む**。これが **prefill**。各 input トークンをネットワークの各層に通し、最初の output トークンを生成するために必要な内部状態を計算。

prefill の決定的性質: **decode に比べてはるかに並列化しやすい**。モデルは開始前に全 input トークンを既に知っているので、**1つのバッチ pass** でアクセラレータ全体を使って処理。GPU はこれが得意で、トークン**あたりの**コストは decode より大きく下がります。ただし**入力長を 2 倍にしてもプリフィルの壁時計時間は半分になりません** — 実際にはプリフィル時間も入力長に応じて伸びる（attention は超線形、それ以外も線形以上）し、長くなるほど memory も帯域も食う、という意味で **prefill は安いのではなく「decode より per-token が安い」**だけです。

```
PREFILL
input tokens:  [t1][t2][t3][t4][t5][t6][t7][t8] ... [tN]
                |   |   |   |   |   |   |   |       |
                v   v   v   v   v   v   v   v       v
               +-----------------------------------------+
               |   1つの大きな並列 forward pass          |
               +-----------------------------------------+
                                 |
                                 v
                        (token 1 を出す準備完了)
```

### Decode — 応答を生成

prefill が終わると、output トークン生成が始まる。これが **decode**（または "generation"）。decode は prefill の**正反対**: 本質的に**逐次**。

output トークン N を生成するには、output トークン 1〜N-1 が**既に生成されている**必要がある（各新トークンは前の全てに条件づけられる）。つまり**output トークン1個につきネットワーク全体の forward pass 1回**。output 次元を**並列化できない**。一度に1ステップしか進めない。

```
DECODE
              forward pass -> output token 1
              forward pass -> output token 2
              forward pass -> output token 3
              ...
              forward pass -> output token M
```

| フェーズ | スケールする量 | 並列? | 支配コスト |
|---|---|---|---|
| Prefill | 入力長 | ✅ 1つのバッチ pass | 幅の広いテンソルでの compute |
| Decode | 出力長 | ❌ トークン毎に1 pass | メモリ帯域幅、繰り返し |

この1つの区別が、**生成 AI を使った時の感触のほぼすべて**を説明します。

## 重要な2つのレイテンシ数値

LLM 呼び出しの体感速度を表す2つの数字。**どこでも見るので慣れる**。

### Time-to-first-token (TTFT)

**TTFT** = ユーザーが「送信」を押してから、応答の最初の文字が現れるまでの待ち時間。主に prefill の関数: **プロンプトが長いほど prefill が長く、TTFT が長くなる**。

短いプロンプトは軽快に感じる。非常に長いプロンプト（肥大したシステムメッセージ、詰まった会話履歴、大きなドキュメント貼付）は、**テキスト表示前から**すでに重く感じる。

> **最初の1秒の沈黙は、対話全体で最も高価な1秒** — ユーザーが「壊れているか？」と判断する1秒だから。

> [!TIP]
> プレフィックスが安定していれば、サービングスタックは過去呼び出しの仕事を再利用できる（**KV-cache ヒット**）。キャッシュヒットでは prefill のほとんどがスキップされ、**TTFT はネットワーク往復時間 + サーバ側の最低限のオーバーヘッド（スケジューリング、キャッシュ参照、最初の decode ステップ）まで縮みます**。完全に RTT だけになるわけではなく、最低限のサーバ処理は残る、という点に注意。

### Inter-token latency と tokens-per-second (TPS)

最初のトークン出現後、モデルは output トークンの**安定的な滴下**に落ち着く。**Inter-token latency** = 連続トークン間のギャップ、**TPS** = その逆数。これがモデルの「**タイピング速度**」。

TPS は主に依存:

- **モデルサイズ**: 大きいモデルは1トークンあたりの仕事が多い → decode が遅い
- **サービングスタック**: バッチング、quantization、attention カーネル、ハードウェアすべて
- **サーバの並行度**: あなたのリクエストは GPU を他と共有

TPS は**短〜中程度のプロンプト長に対しては比較的安定**ですが、**プロンプトが長くなると低下します**。decode の各ステップでは過去のトークン全てに対する KV キャッシュをメモリから読み出す必要があり、プレフィックスが長いほど 1 トークンあたりに読むキャッシュが増えるため、memory-bandwidth bound な decode の時間は伸びます。「全く影響しない」ではなく「短〜中ではあまり感じないが、長コンテキストでははっきり遅くなる」が正しい直感です。

### 両者を組み合わせる

単一リクエストのエンドツーエンドレイテンシ:

```
total_latency  ≈  TTFT  +  (output_tokens / TPS)
                  ^^^^      ^^^^^^^^^^^^^^^^^^^^^
                  prefill   decode
                  (input)   (output)
```

しばらく見つめてください。多くの本番対話で、**第二項が支配的**。中程度の TPS で decode するモデルは、数千 input トークンを読むよりも、**数百 output トークンを生成する方に遥かに長い時間**を費やします。これが「短いプロンプト = 速い応答」が**半分しか正しくない**理由 — TTFT には効くが、生成が長ければユーザーは依然 decode の尾を待つ。

もっと露骨に言えば:

> **入力長は応答が始まる速さに、出力長は応答の続く長さに効く**。

## ストリーミング: 何も買わず全てを買う UX トリック

decode は逐次なので、モデルは生成した瞬間に各 output トークンを知っている。**全応答完了を待つ理由がない**。現代 API は生成と同時にトークンを **stream** で返します。

ストリーミングは**1トークンも節約しない**。総 compute は同一。しかしユーザー視点の体験は反転:

```
NON-STREAMING                    STREAMING
                                  
[send] ............ [whole]      [send] [t1][t2][t3][t4][t5]...
       <-- silence -->                  <-- 単語が現れる -->
       perceived latency                perceived latency
       = total_latency                  ≈ TTFT
```

最初のトークンが届いた瞬間にユーザーは読み始める。最初の1文を読み終わる頃には、モデルは次の数文を生成済み。**完了に10秒かかる応答が、ストリーミングで decode の尾をユーザーの読書速度の裏に隠せば、ほぼ瞬時に感じる**。

実用的帰結:

1. **対話 UI では常にストリーミング**: チャット、autocomplete、コード提案、人間が見るすべて。請求は同じ、体験は劇的に違う
2. **ストリーミングは TTFT を「重要な体感レイテンシ」にする**: プロンプト長と prefix caching の最適化が**製品の応答性に直結**

```python
# OpenAI ストリーミングの例
stream = client.chat.completions.create(
    model="gpt-4o",
    messages=[...],
    stream=True,
)
for chunk in stream:
    delta = chunk.choices[0].delta.content
    if delta:
        print(delta, end="", flush=True)
```

## スループット vs レイテンシ: バッチングのトレードオフ

API エンドポイントの裏で、プロバイダはあなたのリクエストを**専用 GPU で単独実行しない**。多数の並行リクエストを**バッチ化**。ハードは幅の広い行列乗算を、狭い行列乗算とほぼ同じ速度で実行できるので、**多くのリクエストを詰めると総スループット（全ユーザー合計の TPS）が上がる** — リクエストごとのレイテンシを犠牲に。

```
LOW BATCH (少ユーザー)              HIGH BATCH (多ユーザー)
+---------+                        +---------+---------+---------+
| req A   |                        | req A | req B | req C | ... |
+---------+                        +---------+---------+---------+
A は速い、                         A は遅い、
GPU 未使用                         GPU 飽和、トークンあたり安い
```

ユーザーとして**バッチを見ずに**結果を見る:

- **負荷でレイテンシが変動**: 同じリクエストが午前3時と peak で違う速度に感じる
- **安いティア・モデルはより熱いバッチで動く**ことが多い: それが**安い per-token 価格の一部**。レイテンシの一貫性と引き換え
- **スループット指向ワークロード（夜間バッチ、ドキュメント処理パイプライン）はバッチを抱きしめる設計に**: 多くの並行リクエストを小さく速いモデルに → 1リクエストを大きく遅いモデルに、より**done-work-per-minute** が高いことが多い

正しい問いは「**1コール最速のモデルはどれ？**」ではなく「**与えられた時間と予算で全仕事を片付ける構成は？**」。

## KV-cache、プレーンな英語で

「KV-cache」は既に2回出てきました。理解する価値があります — prefix caching が強力な構造的理由であり、第10章でも再登場。

Transformer の各層は、シーケンスの**各位置に対して2つのテンソル**を保持: **K**（keys）と **V**（values）。Attention 機構が「各トークンが何に attend すべきか」を決めるのに使う。トークン N を生成するとき、モデルはトークン 1〜N-1 の K と V が必要。

キャッシュなしだと、各新トークン生成のたびに**過去全トークンの K, V を再計算** — 2乗で支払えない。サービングシステムは生成しながら K, V を保存 — これが **KV-cache**。各 decode ステップは**新しい1トークンの K, V だけ計算**し、過去はすべて再利用。

```
DECODE 中の KV-CACHE

  過去:  [K1,V1][K2,V2][K3,V3] ... [K_{n-1},V_{n-1}]   <-- 既にキャッシュ
  新規:  [Kn,Vn]                                        <-- これだけ計算
         -----------------------------------------------
         attention は上記全体に対して走る
```

同じアイデアをプロンプトに拡張: **2つのリクエストが同じプレフィックスを共有**するなら（例: 同じ長い system message）、そのプレフィックス用の KV テンソルは同一。**prefix caching を持つサービングスタック**はこれを認識し、再計算する代わりに**キャッシュ済み K, V を再利用**。

これがキャッシュ済みプレフィックスが**速い**（キャッシュ部分の prefill compute なし）かつ**安い**（プロバイダがキャッシュヒット入力に大幅割引を提供）理由。**同じ機構、2つの恩恵**。

裏返し: **キャッシュは有限**で、負荷下で evict されうる。キャッシュキーは**正確なプレフィックスバイト**に敏感 — プロンプト前部の1文字を変えるだけで、それ以下が**すべて無効化**。キャッシュ可能性のための設計とは、**各リクエストの前部を安定に、可変部を末尾に**。

## 実用パターン

### 対話チャット UX 用

- **プレフィックスを安定化**: system prompt、instructions、persona、再利用コンテキストを前へ。ターン間で同一に。ユーザーの新入力は**一番末尾**
- **出力をストリーム**: 常に。やらない理由がない
- **最大出力長を妥当な予算でキャップ**: 暴走長生成は、長プロンプトより壁時計時間を悪化させる
- **TTFT を製品メトリックとして監視**: ユーザーが実際に感じる数字

### バッチジョブ用

- **総スループット最適化、リクエストごとのレイテンシではなく**: 一晩で1万ドキュメント要約 → 問いは「1要約の速さ」ではなく「総壁時計時間と総コスト」
- 品質が許す限り、**遅く高いモデルでの1リクエストより、安く速いモデルでの並行リクエスト**を優先
- **全アイテムで安定プロンプトテンプレートを再利用**: 同じプレフィックス、末尾だけ違うペイロード。prefix caching が**ここで巨大な配当**

### ループするエージェント用

エージェント系（計画、ツール呼び出し、繰り返し）は、性能とコストの直感が**最速で間違う**場所。コストは第11章だが、性能側も同様に深刻:

- **各ループ反復が独自の prefill 請求を払う**: 毎ステップ 0からプロンプトを再構築すると、モデルは**毎回全部読み直す**。10ステップ × 太いコンテキストで、10回の prefill を伸びるプロンプトに対して
- **ループを同じプレフィックス上で動くよう設計**: 新しい観察を末尾に append。前部の履歴を並び替えない
- **キャッシュヒットがエージェントを成立させる**: prefix caching なしの長エージェントループは遅く高い。あれば、追加ステップの限界コストが妥当に保てる

> [!IMPORTANT]
> エージェントの壁時計時間がステップ数より**悪く**スケールするなら、**反復間で変異するプレフィックス**を疑え。プロンプト先頭の小さな変更が、その下のすべてのキャッシュを無効化している。

## Going deeper

### Speculative decoding

decode が遅いのは逐次だから（1トークン1 pass）。**Speculative decoding** はシーケンスを破壊する巧妙なトリック。**小さく速い「draft」モデル**が複数トークンを一度に提案。**大きな「target」モデル**が**1つの並列 pass で検証**。draft が target が出すものと一致すれば全て受け入れ → **1 big-model pass の価格で複数トークン decode**。draft が間違っていればそこで fallback。

出力は**big model 単独と数学的に同一** — **品質損失なし**。壁時計時間だけ短い。多くの本番サービングスタックがこれを使用。ユーザーは機構を見ない、ただ TPS が raw decode 速度より高いことに気付くだけ。

### Mixture-of-experts (MoE) モデル

伝統的な dense モデルは**全パラメータを全トークンに使う**。**MoE** は各トークンに対して**パラメータの一部 — 多くの中の数 expert** だけを活性化。**総パラメータ数は巨大、トークンあたり compute は小さい**。

性能への含意:

- MoE は**集合的に巨大でもトークンあたりは安く速い**
- 負荷下の性能が**ぎこちない**ことがある（routing がどの expert が叩かれるかに影響、batching が trickier）
- サービングコストは**total ではなく active な compute / memory / 通信負荷**で決まる傾向があり、「巨大」モデルでも total ベースの想像より per-token は安く済むことがある。ただし**API 価格そのものは結局プロバイダのポリシー**なので、「active 数 × 物理コスト = 価格」と一対一で考えないこと

### 公開ベンチマーク vs 本番現実

ベンチマーク TPS はほぼ理想条件: 暖かいキャッシュ、短いバッチキュー、よくチューニングされたハード。**本番レイテンシは乱雑**: 並行度変動、キャッシュ evict、ネットワーク fluky、long-tail がキューを伸ばす、cold start とリトライがオーバーヘッド。

教訓: **自分のトラフィックで自分のレイテンシを測る**。ヘッドラインは ballpark 比較に有用、容量計画には不十分。第14章で計測を扱います。

## よくある誤解 / FAQ

### Q1. 「短いプロンプトにすれば応答が速くなる？」
A. **TTFT には効く**。total latency は output 長次第。長い回答が予想されるなら、出力長制御の方が効く。

### Q2. 「ストリーミングはトークン代を節約する？」
A. **1トークンも節約しない**。体感を変えるだけ。請求書は同一。

### Q3. 「TPS が高いモデル ≒ 良いモデル？」
A. 速度は1次元。**品質、コンテキスト、コスト、キャッシュ対応**すべて見るべき。「ベンチマーク TPS」は本番再現性の保証ではない。

### Q4. 「KV-cache は私が制御できる？」
A. プロバイダ依存。一部はメッセージに `cache_control: {"type": "ephemeral"}` のようなマーカーで明示制御。多くは**自動**で、安定プレフィックスを認識。**プロバイダのキャッシュ命名規則を読む**。

### Q5. 「並行リクエストを増やせばスループット線形に上がる？」
A. **ある点までは**。サーバ側のバッチ深度限界、レート制限、queue 待ちで頭打ち。**実測でスイートスポットを見つける**。

## まとめ

- 推論は2フェーズ: **prefill**（並列、入力長で scale）と **decode**（逐次、出力長で scale）
- **TTFT** は prefill 支配、**TPS** はモデルサイズとサービングスタック支配
- total latency ≈ TTFT + output_tokens / TPS。**長生成では output 長が壁時計時間を支配**
- **ストリーミング**はトークン節約しないが、体感を激変させる。**対話 UI では常に**
- プロバイダは**バッチ**する。可変速度として現れる
- **KV-caching** がキャッシュ済みプレフィックスを速く安くする構造的理由
- **チャット**: プレフィックス安定化、ストリーム、出力キャップ。**バッチ**: 総スループット最適化。**エージェント**: ループ間でキャッシュを守る
- **Speculative decoding** が TPS を上げる、**MoE** が「大きい = 遅い」ルールを破壊
- **本番レイテンシはベンチマークレイテンシではない**。自分で測れ
