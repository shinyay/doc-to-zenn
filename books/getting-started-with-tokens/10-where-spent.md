> **一言で:** あなたのユーザーメッセージは**ほぼ常に請求書の中身ではない**。請求は**毎コールに同乗する**指示・ツール・履歴・検索コンテキストに支配されている。

---

## なぜこの章があるか

ここまでで、トークンが**仕事・金・レイテンシの単位**だと理解しました。次の自然な問い: **どこに行ってる？**

コーディングアシスタントに1行の質問をして1段落の回答が返ったとき、なぜメーターがあれほど回るのか？

答え: 「単一の LLM 呼び出し」は**ほぼ常にあなたの質問だけではない**。それは**7つの異なるセグメント**から組み立てられたパッケージで、モデルは毎回**全部**取り込む必要がある。エージェント設定（モデルがツールを呼んでループする）では、**そのパッケージ全体が毎ステップ再送される**。

本章ではパッケージを開けます。中に何があり、各部分がどれくらいの大きさで、なぜ50トークンの質問が**気付かぬうちに数万トークン**を食うのかが見えてきます。

---

## 単一コールの解剖学

現代の LLM 呼び出しは、チャットメッセージでもエージェントステップでもツール使用ターンでも、**同じビルディングブロック**から作られる。プロバイダ・フレームワークで命名は違えど、構造は驚くほど一貫。

7つの正典 input セグメント + output、リクエスト内のおおよその順:

### 1. System prompt

モデルに「あなたは何者で、どう振る舞うか」を伝える指示ブロック: 「あなたは役に立つアシスタントです...」+ トーン規則、フォーマット規則、拒否ポリシー。

- **毎ターン送られる**
- 通常はアプリ開発者が一度書く
- 簡単なアシスタントで数百トークン、ルールが多い本番では**数千**に膨張も

チャット UI には現れないが、**モデルが最初に読むもの**。

### 2. 常時オンの指示ファイル

ユーザーが**存在自体を知らない**ことが多いセグメント。多くのエージェント系ツールは、**プロジェクトレベルの指示ファイル**を毎コールに自動注入: リポジトリレベルのガイドライン、コントリビュータ指示、ツールが自動発見するプロジェクト規則ファイル。

- セッション中の**毎ターン、毎ステップ**で送られる
- あなたまたはチームが書く — **しばしばトークンコストを考えずに**
- まじめなプロジェクトで簡単に **1,000-5,000 トークン**

危険性: これらは**ドキュメントのように感じる、コンテキストとは感じない**。一度書いて忘れる。モデルは**毎コール永久に再読する**。

### 3. Tool / function スキーマ

アシスタントがツールを使えるなら（ファイル読み書き、シェル、Web 検索、DB クエリ）、各ツールは JSON-Schema 風の記述を伴う: 名前、docstring、型付きパラメータリスト。モデルはこれを**全て**知る必要がある。

- **毎ステップ送られる** — ツールを使わないステップでも
- 各ツールは通常**100-500 トークン**（パラメータ数と記述の詳細度次第）
- 10-15 ツールの中規模エージェントで、**仕事開始前にスキーマだけで 2,000-5,000 トークン**

**滅多に使われないツールを1つ減らす**のは、最高レバレッジのトークン最適化の1つ。第13章で詳述。

### 4. 会話履歴

現セッションの**過去の全ユーザーメッセージと全アシスタント返信**が、新しい呼び出しのたびにモデルに再生される。**モデルにはコール間の記憶がない** — トランスクリプトが記憶。

- セッション中**単調増加**
- あなたのメッセージとアシスタントの**フル返信**両方を含む — 冗長なツールコールトレースも
- 数十ターン後には**他のすべてを支配しうる**

これが、長い会話が遅く高くなる理由 — 個別メッセージが短くても。第14章で実測。

### 5. 検索コンテキスト

このターン特定で、アプリは**追加素材**を引き込む: 参照したファイル内容、検索結果、ベクター DB の RAG ヒット、最近のエラーログ、開いているエディタタブ。これらは**ユーザーメッセージの前**に追加される。

- 原則 **このターンのみ** — しかし実際には、検索チャンクは**会話履歴に紛れ込み永久に再生**されることが多い
- サイズは大きく変動: 数百トークン（タイトな検索）から数万（「ファイル全部投入」）まで

**最も注意深い設計が報われる**セグメント — **最も自分の制御下にある**から。

### 6. 現在のユーザーメッセージ

あなたが実際に打ったもの。

- **通常リクエスト全体で最小**のセグメント
- 典型的な質問で数〜数百トークン

**意図を表す部分が送信物の 1% 未満**ということが多い、という事実は身に沁みる。

### 7. Reasoning hint / scratchpad / chain-of-thought

該当する場合、リクエストに reasoning scaffold が含まれる: モデルが「考える」ための隠れ scratch、継続する prior reasoning、内部推論の構造化指示。**全モデルが対応するわけではない、全アプリが使うわけでもない**。

- アプリが opt-in した時送られる
- サイズは技法次第 — 無視できる量から実質的まで

### そして: 出力

モデルの返信。一部のモデルでは、**請求はされるが表示されない隠れ reasoning トークン**を含む。

- **トークンごと**に生成 — output が遅い部分の理由
- per-token、output は**全対話で最も高価**なことが多い

## スタックして可視化

実際的な相対サイズで描いた、エージェントの中盤ターンのパッケージ:

```
┌─────────────────────────────────────────────────────────────┐
│  1. System prompt                                           │ 小
├─────────────────────────────────────────────────────────────┤
│  2. 常時オン指示ファイル (AGENTS.md, project rules)         │ 中-大
├─────────────────────────────────────────────────────────────┤
│  3. Tool / function スキーマ (ツール毎に1ブロック)          │ 中-大
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  4. 会話履歴 (毎ターン成長)                                 │ 大
│                                                             │
├─────────────────────────────────────────────────────────────┤
│  5. 検索コンテキスト (ファイル内容、RAG ヒット)             │ 大
├─────────────────────────────────────────────────────────────┤
│  6. 現在のユーザーメッセージ                                │ 極小 ← あなた
├─────────────────────────────────────────────────────────────┤
│  7. Reasoning scaffold (該当時)                             │ 変動
├─────────────────────────────────────────────────────────────┤
│  ──── モデルが今から生成 ────                              │
├─────────────────────────────────────────────────────────────┤
│  8. 出力 (アシスタント返信、+ 隠れ reasoning の可能性)     │ 小-中
└─────────────────────────────────────────────────────────────┘
```

2点が際立つ: 第一に、**ユーザーメッセージは細い切片**。第二に、その上のセグメント（あなたが打たず、おそらく見ていないもの）が**ほぼ全ての仕事をしている**。

## 実例で歩く

数字を入れましょう。中規模リポジトリの単一関数を refactor するよう、コーディングアシスタントが頼まれたと仮定。ユーザー入力:

> 「`lib/parser.js` の `parsePayload` を refactor して、空文字列ケースを処理するようにして」

可視の質問。実際にこのターンで送られるもの:

| # | セグメント | 概算 tokens | 注 |
|---|---|---|---|
| 1 | System prompt | ~300 | 標準「あなたはコーディングアシスタント」規則 |
| 2 | 常時オン指示ファイル | ~2,000 | プロジェクトの contributor + コーディングスタイル規則 |
| 3 | Tool スキーマ (12 ツール) | ~3,000 | read/write/shell/search/edit/diff/test runner... 平均 250 tokens 各 |
| 4 | 会話履歴 (過去 6 ターン) | ~5,000 | この refactor に至るまでの議論 |
| 5 | 検索ファイルコンテキスト | ~4,000 | `lib/parser.js` と関連 2 ファイル |
| 6 | 現在のユーザーメッセージ | ~50 | 実質の質問 |
| 7 | Reasoning scaffold | ~0 | 未使用 |
|   | **入力小計** | **~14,350** | |
| 8 | 出力 | ~800 | 短い説明 + コード編集 |
|   | **合計** | **~15,150** | |

50 トークンの質問が、**~14,350 トークンのコンテキストの波**に乗って届いた。実質の言葉は input 請求の **約 0.3%**。

これは**病的なケースではない**。注意深く設定されたエージェントの**正常運転時**の姿。カジュアルに設定されたもの（5,000 トークンの指示ファイル、25 ツール、リポジトリ全部投入）は、**同じ質問で容易に5倍**の数字に。

> [!NOTE]
> 正確な比はあなたのセットアップ次第だが、breakdown の**形は普遍**: 小さなメッセージ、小さな出力、それより遥かに大きい補助コンテキストの封筒。**ユーザーメッセージだけ最適化するのは間違ったものを最適化している**。

## マルチステップエージェントループ: 請求書、再請求

ここまで1ターンを見ました。エージェント系アシスタントは滅多に**1ターンで仕事を終えない**。反復します。

```
┌──────────────────────────────────────────────────────────┐
│  Step 1: モデルがフルコンテキストを読み、ツールコール書く │
│          → ツール実行、出力返却                           │
├──────────────────────────────────────────────────────────┤
│  Step 2: モデルがフルコンテキスト + ツール出力を読み、    │
│          別のツールコール書く                              │
├──────────────────────────────────────────────────────────┤
│  Step 3: モデルが過去全ツール出力 + フルコンテキスト読み...│
├──────────────────────────────────────────────────────────┤
│  ...                                                      │
├──────────────────────────────────────────────────────────┤
│  Step N: モデルが最終回答を生成                          │
└──────────────────────────────────────────────────────────┘
```

**決定的細部**: 各ステップは**完全な LLM コール**。モデルにはステップ間の記憶もない。Step 2 では Step 1 の全部 + ツール出力が送られる。Step 3 では Step 1, 2 の全部。等々。

つまり**各ラウンドが全コンテキスト請求を再支払い**、新しいツール出力分だけ追加。

上の例: 1ターン ~14,350 input。実際に refactor を解くのにモデルが:

1. ファイルを読む
2. 関連ファイルを読む
3. テスト実行
4. テスト失敗を検査
5. 編集を提案
6. 編集を適用
7. テスト再実行
8. 確認して回答

…おおよそ 8 ステップ。各ステップが system prompt、常時オンファイル、ツールスキーマ、伸びる履歴、累積ツール出力**全て再送**。10 ステップエージェントループは、**per-turn input コストを ~10×**（あるいはそれ以上、履歴がループ中に伸びるため）に倍増させうる。

50 トークンの質問が、**10万 input トークン超の仕事**を駆動した。

## セッション横断の複利効果

さらにズームアウト。実セッションは多ターン:

```
total_input_tokens  ≈  Σ_t (system + always_on + tools + history_t + retrieved_t + msg_t) × steps_t

total_output_tokens ≈  Σ_t output_t × steps_t
```

この式から落ちてくるもの:

- **履歴は伸びる**: `history_t` は毎ターン増加 → **後のターンは系統的に前のターンより高い**
- **ツール重ターンは加算でなく乗算**: 1エージェントステップ追加 = 固定コストの追加ではなく、**もう1つのフルコンテキスト封筒**の追加
- **固定部分が複利で効く**: system + 常時オン + tools の per-step「税」が、N ターン × M 平均ステップで `N × M` 回支払われる。**この固定税から1,000トークン削るのは、単一検索から10,000トークン削るより効くことが多い**

**キャッシュは助ける**。多くのプロバイダがリクエストのプレフィックス（コール間で変わらない部分）をキャッシュ → system prompt を毎ステップフルプライス再請求はされない。**しかしキャッシュは割引であって免除ではない** — トークン数自体は変わらない、安くなるだけ。問題の形は変わらない。

## 入力/出力比、実務での着地点

入力/出力比について、**反対方向に引っ張る2つの事実**を頭に同時に保持:

- **量で**: 入力が支配。エージェントワークロードでは input が output の **10× 〜 100× 以上**が普通
- **金額で**: **output が支配することが多い**。output は input より遥かに高い per-token レート → 少ない出力トークンが多い入力を請求書で上回る

実用的含意: **片方だけ最適化はできない**。

- 入力だけ見る → **請求書で驚く**
- 出力だけ見る → **レイテンシとコンテキスト窓で驚く**

良い計測（第14章）は**両方を追う**。

## Going deeper

### Chat テンプレートは「見えない」トークンを使う

モデルは実際には `{role: "user", content: "..."}` のクリーンなリストを見ていない。裏で chat はテンプレートで**単一トークンストリームに描画**され、各メッセージ間に role marker、ターン区切り、特殊制御トークンが挿入される。これらは**実トークン** — コンテキスト窓スペースを消費し、課金されるが、メッセージテキストには現れない。

短いターンの多い長会話で、**テンプレートオーバーヘッドが総入力の数%**に積み上がりうる。最大の項目ではないが**実在する**。トークン数がメッセージ長の素朴な合計と**正確に一致しない**理由。

### ツール結果は次ターンの入力として数えられる

ツールが出力を返すとき、その出力は何らかの方法でモデルに戻る必要がある。多くのプロバイダは**ツール結果を特殊メッセージとして会話に追加** → 次ステップの**入力の一部**になる。**5,000 トークンのファイル内容を返すツールは、その履歴部分が削除/要約されるまで、以後の各ステップで5,000 input トークンを買い続ける**。

エージェントコスト分析で**最も一般的な驚き**: セッション初期の単一「この大ファイルを読む」ツールコールが、**セッションの残りの間、フル課金され続ける**。

### Reasoning トークンは隠されていても実在

一部のモデルは、可視回答を生成する前に内部推論 pass を行う。これらの reasoning トークンは**生成され、課金され、コンテキスト窓に対して数えられる** — 応答に表示されないかもしれないのに。

コスト視点で reasoning トークンを **output として**、容量計画視点で**コンテキスト窓消費**として扱う。reasoning モデルからの「短い返信」は欺瞞的: 可視部分が短くても、裏で**大量のトークン生成**が起きていた可能性がある。

> [!TIP]
> 意外な請求書や遅い応答を調査するとき、**7セグメントを順に走り、「どれが伸びたか？」**と問う。答えはほぼ常に**特定の1セグメント**で、ユーザーメッセージのことはまずない。

## よくある誤解 / FAQ

### Q1. 「ユーザーメッセージを短くすればコスト下がる？」
A. 効果は限定的。**全体の 1% 程度**しか改善しない。**システム + tools + 履歴**を見直す方が**桁違いに効く**。

### Q2. 「履歴を全部送らないとモデルが文脈を失う？」
A. 概ねその通り。だから設計判断: 全送信（高コスト）vs 要約（劣化リスク）vs 必要部分抜粋（中庸）。**会話の長さでスイッチ**する戦略が一般的。

### Q3. 「ツール定義を Markdown で書くより JSON Schema の方が安い？」
A. **書き方次第**。冗長な description、長い列挙、深いネストは双方で重い。**最小限の必須情報**にするのが最大効果。

### Q4. 「キャッシュがあるなら何も心配いらない？」
A. **誤り**。キャッシュは無効化される（プレフィックス変更、TTL、evict）。**設計でキャッシュ可能性を守る**必要が常にある。

### Q5. 「system prompt を毎回違う動的内容にしたい — 妥協案は？」
A. **動的部分を末尾**に。たとえば: `[安定 system] + [安定 tool 定義] + [動的 user-specific 設定]` の順。安定プレフィックスはキャッシュ。

## まとめ

- 各 LLM コールは **7 input セグメント + 出力**のパッケージ
- ユーザーメッセージは**最小**、補助コンテキストが支配
- 各ツール定義 ~100-500 tokens、会話履歴は毎ターン成長、常時オンファイルは毎ステップ支払い
- **エージェントループは毎ステップで全コンテキスト請求を再支払い**: 10ステップ → ~10× per-turn コスト
- セッション横断で**固定セグメント (system + always-on + tools) は `N × M` 回支払う税**。**ここを削ると効果絶大**
- 量は入力支配、金額は出力支配することが多い。**両方を計測**
- Chat テンプレート、再生されるツール出力、隠れ reasoning トークン — メッセージテキストに現れない**実消費**
