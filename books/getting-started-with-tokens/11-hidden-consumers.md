> **一言で:** あなたが払うトークンの**ほとんどはプロンプトに現れない** — 入力前に黙って読み込まれ、毎ステップに自動添付され、表示されることなく課金される。

---

## なぜこの章があるか

第9章で**可視の予算**を見ました。本章ではその下、**氷山の水面下**を扱います。実ツール（プロジェクトルール付きアシスタント、ツールカタログ持ち IDE Copilot、ループするエージェント、毎ターン fetch する RAG）を使う瞬間、画面に見えるのは氷山の頂上だけ。**請求書はあなたが打たず、読まず、存在も知らないトークンに支配**されます。

本章は、**実害の大きさ順に**それらの最大の消費者を名指します。怖がらせるのが目的ではなく、**見えるようにする**こと。後の hygiene と計測の章が**具体的に対処できる対象**を持つために。

---

## 氷山の形

メンタルモデル: 各リクエストには**2層**ある — **あなたが書いた部分**と、**ランタイムがあなたに代わって書いた部分**。

```
あなたが見るもの              モデルが実際に受け取るもの
------------                  --------------------------------
                              ┌────────────────────────────────┐
                              │  chat template / role markers  │
                              ├────────────────────────────────┤
                              │  system prompt (プラットフォーム)│
                              ├────────────────────────────────┤
                              │  常時オン指示ファイル            │
                              ├────────────────────────────────┤
                              │  tool / function カタログ       │
                              ├────────────────────────────────┤
                              │  検索コンテキスト                │
                              ├────────────────────────────────┤
                              │  過去ターン (履歴)              │
                              ├────────────────────────────────┤
"この関数を refactor して"───►│  あなたのメッセージ              │
                              └────────────────────────────────┘
                                          │
                                          ▼
                              ┌────────────────────────────────┐
"完了 — はい、こちら。"  ◄──── │  隠れ reasoning + 返信          │
                              └────────────────────────────────┘
```

1行書いた。ランタイムは数千トークン送った。モデルは短い回答を出す前に**さらに数千生成**したかも。**図のすべての層が課金**。**ほとんどが見えない**。

5大消費者を歩きましょう。

## 1. 常時オン指示ファイル

現代のコーディングアシスタント・チャットツールは**プロジェクトレベルのルール**をサポート: リポジトリ内のファイル（ツール名で命名されることも、generic な agents ファイルのことも）が、そのプロジェクトに関する全会話に注入される。**個人ルール**も同様。

これらは一貫性に素晴らしい。**しかし黙った無駄の最大源の1つ**でもある — 3つの性質が悪く複利:

- **自動**: 一度書いて、考えなくなる。送られ続ける
- **per-turn**: 会話の最初だけでなく、**全メッセージに同乗**
- **per-step**: エージェントループで、ツール呼び出し → 結果 → 継続のたびに**ルールファイル再送**。5ツール呼び出し = 請求書に5コピー

控えめな例: プロジェクトルールが 1,500 トークン（ガイドライン1〜2ページ）。アシスタントとの長作業セッション: 100 ターン。各ターン中に複数ツールステップ。控えめに見て、そのファイルを**200〜300回**送った。**1回書いて忘れたファイルに数十万 input トークン**。

ファイルは**送られるごとに有用になっていない**。モデルは最初のコピーで既にルールを知っている。50回目で賢くなったりしない。

> [!TIP]
> 常時オン指示ファイルを書くとき、「これは**毎ターン、永久に、私のアカウントに課金される**」と想像してください。そして問う: **この段落にそれを払うか？** 多くの「重要に感じる文」がこの問いを生き残らない。

## 2. ツール / function-calling カタログ

モデルにツール呼び出し能力を与える時（function-calling API、MCP サーバ、IDE 統合）、**実際にモデルへ送られるツール記述はあくまで「クライアント／ホストアプリがプロンプトに含めると決めた分」**です。例えば MCP は「サーバが提供する `tools/list` をクライアントが取りに行ける」ことを定義しているだけで、**それを毎ターンモデルプロンプトに何件・どう連結するかはホスト側の実装の責任**です（必要なツールだけ抜粋することも、全件貼ることもできる）。つまり「MCP を有効化しただけで自動的に何万トークンが乗る」のではなく、「**接続先のクライアント／IDE が取得したツールセットをそのままモデルに渡す挙動なら**何万トークン乗りうる」というのが正しい言い方です。

各ツール定義は通常:

- 名前
- 「何をするか・いつ使うか」の自然言語記述
- パラメータの構造化スキーマ（型、デフォルト、enum、各フィールド記述）
- 時にコール例

よく文書化された単一ツールはしばしば数百トークン。リッチなネストパラメータを持つツールはもっと。**カタログはツール数で線形成長**:

```
   1 tool   ≈   小さな段落
  10 tools  ≈   1ページ
  50 tools  ≈   1章
 150 tools  ≈   小さな本 — 毎ターン送られる
```

気付かぬうちに上端に達するのは簡単です。十数の MCP サーバ、各々8〜15ツールを **クライアント／IDE がそのまま全件モデルプロンプトに貼る設定**だと、ユーザーメッセージ到達前に**数万トークン分のツール定義が乗る**ことがあります。**モデルが使うかどうか・現タスクが関係しうるかどうかに関わらず**乗る、というのは「クライアントが選別せずに全部渡している」場合の話で、賢いクライアントは関連ツールだけを渡すこともできます。**ここでのコストは MCP プロトコル自体ではなく、ホストアプリのプロンプト構築方針が決めている**点を押さえておいてください。

そして**エージェントループの全ステップに存在** — カタログがモデルに有効ツールコールを emit するよう conditioning する一部だから。

ここでの**発見可能性問題**は深刻: ほとんどの製品で**カタログが見えない**。**数ヶ月前に有効化した統合**が、まだロードされている。

## 3. 検索コンテキスト (RAG bloat)

RAG — 知識ソースから関連スニペットを fetch しプロンプトに貼る — は現代 LLM アプリで最も有用なパターンの1つ。**同時にトークンが最も漏れやすい場所**。コストはチャンク数とサイズに比例し、両方とも**時間と共に上向きに drift**。

```
あなたの質問
     │
     ▼
┌─────────────┐    top-K chunks    ┌──────────────────┐
│   retriever  │ ─────────────────► │  prompt builder  │ ──► model
└─────────────┘                    └──────────────────┘
                                        │
                                        │  K チャンク × N tokens
                                        ▼
                                    K × N tokens、毎ターン
```

K = 20、各チャンク 500 tokens なら、retriever は**毎ターン 1万トークン追加**。長会話で sliently 積み上がる。さらに悪いことに、**retriever は recall 寄りに誤る**: 20 チャンクのほとんどは実は質問に関連していない。**送るのに払い、モデルは無視するのに attention 予算を払う** — 第8章で見たように、これも**レイテンシと品質**コスト。

ユーザー視点では「アシスタントが何かを調べた」というフレーズに**全てが隠れる**。**実際のレシート**: アシスタントは20検索を実行し、結果をプロンプトに貼り付け、次のターンも、次のターンも同じ。

## 4. 隠れ reasoning トークン

多くの新しいモデルは、可視回答を出す前に**内部 chain of reasoning** を生成するよう設定可能。ランタイムはこのチェーンをあなたから隠す（最終返信のみ表示）が、**チェーンはモデルが生成し、通常 output トークンとして課金**。

これが効く理由:

- Reasoning chain は**可視回答より遥かに長い**ことが多い。2行の返信が**ページ単位の内部思考**の上に乗る
- Reasoning effort はしばしば**ダイアル** (low/medium/high)。上げれば隠れ output が乗算 — 難プロンプトで**1桁以上**
- Output トークンは**より高い種類**（第7章）。隠れ reasoning は**請求書の高い側**を叩く

罠: 可視返信は**短く安く見える**。質問 → 段落。**小さな対話に感じる**。実生成量（あなたが払っているもの）は**数十倍大きい**かも。

> [!NOTE]
> 隠れ reasoning は難問では**本当にコスト分の価値**があることが多い。「悪い」と言っているのではない。**コストが design上見えない**ので、**いつ適切かを能動的に判断する必要がある**。デフォルトを「high」にするのは、**金の電気を点けっぱなし**バージョン。

## 5. マルチステップエージェントループ

**最大の隠れ乗数はエージェントループ自体**。

「失敗テストを直して」のような**単一質問に感じる**リクエストが、裏で多ステップに分解: テストファイル読む → 実装読む → 関連コード検索 → テスト実行 → 失敗読む → ファイル編集 → テスト再実行 → 新失敗読む → 再編集... **各ステップがモデルへのフル forward pass**。各 pass はそれまでの全コンテキストを受け取る: system prompt、指示ファイル、ツールカタログ、過去ツールコール、過去ツール結果、元質問。

```
turn 1: ユーザーが質問
        ├── pass 1: context + 質問                        → tool call A
        ├── pass 2: context + tool result A               → tool call B
        ├── pass 3: context + results A, B                → tool call C
        ├── pass 4: context + results A, B, C             → tool call D
        └── pass 5: context + results A, B, C, D          → 最終回答
```

「1質問」に対して **5 passes**。**コンテキストは各 pass で大きくなる** — 各 pass が前のツール結果を append。**5番目の pass が最も高価**。15ステップなら、最後の数 pass は**非常に**高価。

これが**エージェントフレームワークの驚き請求書の機構**。UX には「30回の累積コンテキスト往復に今から払う」とは書いていない。製品はスピナーと結果を見せるだけ。

## 名指すべき小さな隠れ消費者

5大が大ダメージだが、**高ボリューム使用で積み上がる小漏れ**もある:

**Chat テンプレートトークン**: 全ての chat 風 API がメッセージを role marker、区切り、特殊トークンで包む。1メッセージあたり数トークン — しかし**全会話の全ターンの全メッセージに掛けると**、無視できない。

**Tool result echo**: ツールが返ると出力は会話に append され、**残りのセッション履歴に居続ける**。長 JSON を返すツールがセッション初期に呼ばれると、**履歴が削除/会話終了するまで以後の毎ターンで再課金**。

**貼り付けられたエラー blob とスタックトレース**: ユーザーが1000行のスタックトレースを debug のために投入。問題は3ターン後に解決。**スタックトレースは次の40ターンの会話履歴に残り、毎リクエストに同乗**。

**礼儀的な filler**: 「もちろん、お手伝いします。共有いただいた内容を見てみますね...」 単一コールでは些細。**デプロイ済み製品の数百万コール**で実金、しかも**output トークン（高い側）**。

## 発見可能性の問題

これら消費者が危険な理由は**個別コールでの大きさではない**。**請求書で見えない**こと。

ほとんどのプラットフォームは**集約**する。月次合計、おそらく日次・API キー別の breakdown。「**今ターン送った 5万 input のうち、47,000 が system + プロジェクトルール + 6ヶ月前に有効化したツールカタログ、3,000 が実質の質問**」とは見えない。**breakdown なしには行動の signal がない**。請求が上がる、どのつまみが動いたか分からない。

これがまさに**第14章 (計測)** が存在する理由。**計測なしの hygiene は推測**。計測の最初の仕事は、**氷山を分解しレイヤーにラベルを貼る**こと。

中間の有用な演習: 今日使うどの LLM ツールについても、**プロンプトがモデルに到達する前に何がロードされているか**を一旦止まって自問してください。見えないプラットフォーム system prompt？ 数ヶ月前に書いたルールファイル？ 有効化したことを忘れた統合からのツールカタログ？ 5で十分なところに20ヒットを fetch するベクター DB？ 毎日使うツールについてこれに答えられないなら、**そのコストを制御していない**。

## Going deeper

### Prompt caching を**部分的緩和**として

多くのプロバイダは何らかの形の prompt prefix キャッシングを提供しています。**ただしキャッシュヒットは自動ではなく、プロバイダ固有のルール**（最低長、ブレークポイントの置き方、キャッシュの TTL、料金体系など）に従って初めて成立します。プロンプト先頭（system prompt、指示、ツールカタログ）が次コールと**バイト／トークン単位で**同一で、かつそのプロバイダのルールを満たすなら、ランタイムは**それら反復トークンを大幅に低レート**で扱うことがあります — 既に処理済みだから。条件が揃えば**本当に有用**です。

**しかし氷山は小さくならない**。請求は依然として送ったトークン数で scale; **キャッシュ部分の単価が下がるだけ**。キャッシュは「**安定で重複排除されたプレフィックスを持つこと**」を報酬する; 「**小さなプレフィックスを持つこと**」は報酬しない。**そしてキャッシュ済みトークンも依然コンテキスト窓に対して数えられる** — $ コストは下がるが、**attention とレイテンシのコストは依然支払う**。

重要な区別を生む:

- **コンテキストサイズ**: 収まるもの — 1コールでモデルが attend できるトークン数
- **課金トークン**: 払うもの — キャッシュ適用時はコンテキストサイズより**低**く、隠れ reasoning output を数えると**高**くなる

両者はキャッシュ下で発散し、**「tokens in, tokens out」の単純推論を誤導する**。キャッシュ済み 5万トークンプレフィックスはキャッシュなしより安いが、**モデルは依然 5万トークン全体で考える必要がある**。

### Aggressive な履歴 pruning

一部のエージェントランタイムは、ループ進行と共にコンテキストをスリムに保つため、**古いターンと古いツール結果を意図的に破棄/要約**。少しの recall を、大量のコスト・レイテンシと交換。**最も効果的な隠れ消費者緩和策の1つ**だが、**慎重さが必要**: 間違ったものを刈ると、エージェントが筋を見失う。第13章で再訪。

組み合わせ — **安定なものはキャッシュ、古いものは刈る** — が、長エージェントセッションを手頃にする state of the art。**どちらも、最初から不要なトークンをロードしないことの代替ではない**。

## よくある誤解 / FAQ

### Q1. 「キャッシュがあるなら指示ファイルを大きくしても安全？」
A. **誤り**。コンテキスト窓スペース、attention、レイテンシは依然コスト。キャッシュは $ だけ。**「lost in the middle」**も悪化。

### Q2. 「ツールカタログを動的に絞れる？」
A. プラットフォーム次第。一部は「このターンで必要そうなツールだけ」をロードする mechanism を提供（"tool selection" 機能）。**手動で OFF にする**だけでも大きな効果。

### Q3. 「reasoning effort はどう調整？」
A. タスクごとに実験で**最低必要 effort** を見つける。デフォルトを「high」にせず、難タスクのみ昇格する設計に。

### Q4. 「履歴を全部要約に置き換えれば安くなる？」
A. **コストは下がるが、品質劣化リスク**。**Recent N ターンは生で残し、それ以前を要約**のような hybrid が一般的。

### Q5. 「polite filler は消すべき？」
A. ユーザー対面では UX とトレードオフ。**バッチ処理・エージェントの内部出力**では削れる（プロンプトで明示的に「no preamble」と指示）。

## まとめ

- 可視プロンプトは**氷山の頂上**。請求はランタイムが代わりに追加したトークンに支配される
- **5大隠れ消費者**: 常時オン指示ファイル、ツール / function カタログ、検索コンテキスト、隠れ reasoning output、マルチステップエージェントループ自体
- 各々がターン・ステップ・両方で乗算。**小さな per-call コストが大きな per-session コストに**
- 小漏れ — chat テンプレート、tool result echo、貼られたエラー、礼儀的 filler — も**スケールで積む**
- ほとんどのプラットフォームが**ソース別 breakdown を提供しない**。テレメトリなしでは**どこに行ったか見えず、行動できない**
- **Prompt caching** は安定プレフィックスの単価を下げるが**氷山を縮めない**。**履歴 pruning** は氷山を縮めるが慎重さが必要
- 使う各 LLM ツールへの最初の問い: **「プロンプトが到達する前に、毎回、何がロードされているか？」**
