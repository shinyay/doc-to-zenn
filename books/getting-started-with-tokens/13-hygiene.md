---
title: "第12章 — トークン hygiene の原則"
---

> **一言で:** トークン hygiene とは、特定の最適化技法に手を伸ばす前に、**請求書を黙って健全に保つ小さな習慣セット** — レイアウト、スコープ、表現、出力、コンテキスト、モデル選択、セッション、ツール、計測 について。

---

## なぜこの章があるか

ここまでの章は領土の地図を描きました: トークンとは、どう数えるか、何が高くつくか、どこで消費されるか、多すぎると何が起きるか、品質とコストのトレードオフ。**本格最適化を始める前に、最適化の下層に位置する、どの単一トリックよりも重要な層**があります — それが **hygiene**。

Hygiene は、一貫適用すればトークン使用が問題に drift するのを防ぐ、**小さく、ほぼ退屈な習慣セット**。**何も clever ではない**。**新しいツールやフレームワークを要求しない**。ほとんどは**一度決めて考えなくなる判断**。しかし**複利**で効く — それらを尊重するプロジェクトは、後で英雄的最適化を必要としない傾向、なぜなら**そもそも無駄遣いしたことがないから**。

本章は **10 の原則**を歩きます。各々は **原則 (principle)** であり、recipe ではありません。Recipe ファミリ — caching、retrieval、summarization、batching、distillation — は第16章でプレビュー。ここでは**それら recipe が乗る基盤習慣**を与えます。

**2〜3 採用するだけでも違いに気付きます**。**10 すべて採用すれば、「趣味プロジェクト」から「驚きなしにスケールできるシステム」**へラインを越えます。

---

## 1. Stable prefix, volatile suffix

現代の推論スタックは、**2 リクエストの prefix が完全一致するとき計算を再利用**。共有 prefix の内部表現はキャッシュされ、同バイトで始まる次リクエストは再計算不要。**キャッシュ部分は通常入力価格の数分の1**で課金。

このキャッシングは**位置的・厳密**です。**プロンプト先頭から forward** に働き、**最初に異なるトークン（あるいは serialize 後のプロンプトセグメント）で止まる**のが基本動作で、プロンプト先頭側の **1 文字変更で結果として以後すべてが invalid 化**されることがあります。実装上はバイト単位ではなくトークン化／メッセージ整形後の表現でマッチングされる点に注意してください（プロバイダごとに breakpoints の置き方や粒度のルールがあります）。

そこから派生する hygiene ルールは単純: **変わらないものを先頭に、変わるものを末尾に**。

```
[ system prompt        ]   ← 安定、プロジェクトに1度設定
[ project rules        ]   ← 安定、編集稀
[ tool definitions     ]   ← 安定、リリースで変更
[ retrieved context    ]   ← クエリ毎変動
[ conversation history ]   ← 時間で増加
[ current user message ]   ← 常に異なる
```

**間違ったレイアウト** — タイムスタンプ・セッション ID・「今日の日付は…」を**先頭に置く** — は**全リクエストでキャッシュを無効化**。**正しいレイアウト**はプロンプトを**層ケーキ**として扱う: 下層は1度焼かれ再利用、上層が新鮮。

キャッシング機構を詳しく考える必要はない。**揮発部分を底に保つ**だけ。

## 2. Scope load on demand, not always-on

**自動でロードされるものは毎ターン、永久に課金される**。**気付かれぬトークン支出の最も一般的な源**。

パターンは多所に: 全プロンプトに append される global 指示ファイル、全リクエストでプロジェクトを説明する preamble、アシスタントが**必要かもしれない**全ツールを含むカタログ、関連性関係なくユーザー設定を注入する memory ストア。**これら各々が毎ターン課金される — 重要な特定ターンだけでなく**。

Hygiene ルール: **デフォルト load when relevant, not load always**。プロンプトに添付される各テキストブロックに対し、**1つの問い**:

> **モデルはこれを毎ターン必要か、特定のファイル・タスク・トピックで作業する時だけか？**

答えが「X で作業する時だけ」なら、**global path にあってはならない**。**条件付きロード** — ファイルが触れられる時、トピックが上がる時、ツールが実際に発火しそうな時に。ほとんどのプラットフォームは**スコープ付き/条件付き指示ロード**をサポート; **使え**。

数学は容赦ない。**1,000-token 常時オン指示ファイル** = 1,000 × 毎ターン × 毎セッション × 毎ユーザー。**1 ファイルパスにスコープされた 1,000-token ファイル** = 週1〜2回 1,000。**内容は同じ。請求書は違う**。

## 3. Declarative over imperative phrasing

**2 プロンプトが同じ意図を、極めて違うトークン数で表現できる**。

```
Imperative (冗長):
"もしデータを返す機会があれば、JSON として整形してください、
そして適切な JSON 構文を使うのを忘れず、JSON にコメントを
含めず、全文字列が適切に quote されていることを確認し、…"

Declarative (簡潔):
"常に有効な JSON で応答。コメントなし。"
```

Declarative 版は**短く、明確で、実用上より信頼性高い**。**長く、hedge され、qualified された指示は高価なだけでなく、モデルがフォローするのも難しい**。各「please」「remember to」「make sure that」は**トークンと指示フォロー明確性の両方で支払い**。

Hygiene ルール: **スタイルガイドが書かれるように書け、ナーバスなメールが書かれるようにではなく**。制約を**1度、命令現在形で**述べ、モデルに従うことを信頼。従わなければ、修正は稀に「もっと言葉」。**通常は鋭く短いルール、または重要な部分の近くにルールを移動**。

この習慣だけで、**初回パスでシステムプロンプトが半分**になる傾向。

## 4. Output discipline

**請求書の出力側は、トークンあたり通常より高い側**です。生成トークンは入力の数倍値段で、入力と違って context window で cap されるわけでもなく、明示的に `max_tokens` のような上限を設けない限り、**出力はモデル（とプロバイダ側のデフォルト max_output 上限）が許す限り伸び続けうる**。つまり「モデルが決めるだけ」ではなく「**モデル × プロバイダの上限 × あなたが指定した上限**の組み合わせで決まる」のが正しい言い方です。

これが「**Code only, no explanation**」「**Reply in three sentences or fewer**」のような 1 行指示を、**プロンプトに加えられる最高 ROI 追加の1つ**にする。**1 度数トークン払って、毎レスポンス何百も節約**しうる。

```
Output discipline なし:
  request:  ~500 tokens
  response: ~1,800 tokens   ← モデル説明、要約、hedge、recap

Output discipline あり:
  request:  ~510 tokens     ← 制約に +10
  response: ~250 tokens     ← モデルは答えて止まる
```

**逆もまた真**: **本当に欲しくない限り「thorough, complete, exhaustive」な回答を要求しない**。「complete」「comprehensive」「detailed」のような語は**出力 amplifier**。長尺成果物が**実際に**必要なら適切。**単に答えが欲しかっただけなら無駄**。

> [!TIP]
> Chat 風アシスタント向けの便利デフォルト: 「**Be concise. Use short paragraphs. Skip recaps.**」 3 短句、平均レスポンス長への大効果。

Output discipline は**UX も改善**。**速くタイトなレスポンスは長いものよりほぼ常に好まれる** — そして**生成すべきものが少ないので早く出荷**。

## 5. Minimum sufficient context

第11章は品質視点でケースを作りました: **より多くのコンテキストは常に良いとは限らず、ある点を超えると能動的に害す**。Hygiene 視点はコスト側から見た同じ点。**毎ターン、モデルの前に置いたものに払っている**。

Hygiene ルール: **モデルが良く回答するのに必要なものを与えよ**。**利用可能なものではなく**。「**potentially useful**」ではなく。「**interesting background**」ではなく。**Minimum sufficient context**。

実用上、各 retrieval/include の前に問う:

- これは*このクエリ*に関連するか、それともトピックに関連するだけか？
- この**全文書**が必要か、**1 セクション**だけか？
- この会話履歴はまだ load-bearing か、内容は superseded されたか？
- **1 行要約が full text と同じ仕事**をするか？

**答えはしばしば「少ない方が機能した」**。「念のため全部詰める」パターンは**快適だが高価** — 第11章が示した通り、**品質カーブは直感より早くピーク**。

この原則は**原則1 (stable prefix) と composes**。**Minimum sufficient context は揮発コンテキストの傾向** (per-query retrieved chunk) — それを刈ることは**二重に良い**: 請求書が小さく、キャッシュ済み prefix が荷重をより担う。

## 6. Right-size the model

モデルは**段階的**。最安段と最強段の**コスト差はトークンあたり通常 1 桁** — 時にそれ以上。能力差は実だが、**全タスクに均一に適用されない**。

**小さなモデルが通常 fine**:

- 分類・ラベリング
- 構造化データの再フォーマット
- 整形済みテキストの要約
- 適切な下流ハンドラへのリクエストルーティング
- より能力高いモデルが編集する前のドラフト

**フロンティアモデルが価値を稼ぐ時**:

- 多コンテキストにわたる多ステップ推論
- 出力が高ステークス (security、正確性、ユーザー対面の声)
- タスクが小段階で失敗することが観察された

Hygiene ルール: **タスクに許容品質を生む最小モデルを選び、ラインナップ変更時に再評価**。**全コールを最強モデルにデフォルトするのは、製品システムで単一最高価な習慣** — そして**多数派トラフィックにはほぼ常に不要**。

一般的・健康的パターン: **2 段セットアップ**。安いデフォルトがバルクを処理、強いモデルはタスクが「難」フラグされた時または安いモデルの出力がチェック失敗した時のみ起動。

---

## 7. Right-size the reasoning effort

多くの現行モデルは reasoning-effort knob を露出 — 時に明示 (パラメータ)、時に暗黙 (「thinking」モード)。**Knob を上げると、モデルは回答前に隠れ reasoning トークンを生成。それら隠れトークンは課金**。

Knob を最大に置くと、**reasoning トークンがしばしば出力請求を支配** — 時に可視レスポンスの 1 桁上。第10章で記述したコストパターン。

Hygiene ルール: **デフォルト medium (またはその等価)**。**真に必要なタスク** — 証明、多ファイルデバッグ、多ステップ変更計画 — のみ上げる。**ルーチン** — 形式変換、単一ファイル編集、短回答、分類 — では下げる。

```
high reasoning, simple task
  ┌────────────┬──────────────────────────────┬─────┐
  │  prompt    │  hidden reasoning tokens     │ ans │
  └────────────┴──────────────────────────────┴─────┘

medium reasoning, simple task
  ┌────────────┬──────────┬─────┐
  │  prompt    │ thinking │ ans │
  └────────────┴──────────┴─────┘
```

**両方が同じ回答を生んだ。1 つだけ払った**。

## 8. End sessions on topic change

長会話は履歴を蓄積。**毎ターン、前ターンのメッセージが次リクエストに送られる prefix に追加**。キャッシングで安くはなるが、無料ではなく、**請求が抑制されても会話は窓ターム的に成長**し続ける。

Hygiene ルール: **トピック変更時は新セッションを始める**。「**fresh start**」のコスト (system prompt 再確立、プロジェクトルール、現フォーカス) は、**無関係な前トピックを残りの日中引きずり続けるコストよりほぼ常に少ない**。

特にアシスタント風使用で真。「この機能名付け手伝って」で始まり今は「このスタックトレースをデバッグ」になった会話は、**長く無関係な prefix が毎デバッグターンで再支払いされている**。スタックトレースは命名議論から利益を得ていない。**新セッション開始でこのバラストを落とす**。

**セッションは安い。トピックは安くない**。

> [!NOTE]
> 「**End on topic change**」は「**毎メッセージ後に終了**」ではない。**モデルがターンにわたって本当に有用な状態を構築している時** — デバッグトレイル、進化する設計、長編集 — セッションは価値がある。**その状態が load-bearing でなくなった時に終わる**。

## 9. Treat tools as inventory

**モデルに公開する各ツール** (名前、記述、パラメータスキーマ、使用ノート) は、**ツールが利用可能な毎ターン、プロンプトに含まれる**。50 ツールカタログは、**モデルがどれかを使うかどうかに関わらず、50 ツール分のトークン**を支払わせる。

第10章で機構をカバー。Hygiene ルール: **ツールカタログを在庫として扱え**。定期的に audit。各ツールに問う:

- このツールは**実際に呼ばれている**か？ (ログが教える)
- 2 つの類似ツールを**より豊かなパラメータで1つ**にマージできるか？
- **稀使用ツール**を、関連時のみロードする**別のスコープ付きカタログ**に移せるか？
- 記述は**実仕事**をしているか、**半分にできる boilerplate** か？

カタログにあるが**呼ばれないツールは家賃を払い価値ゼロ**。**60 トークンと同じ仕事をする 200 トークン記述は払いすぎ**。**時間と共にツールを蓄積し、刈らないカタログは、ほぼ常に必要より大きい**。

**節約が毎ターンに適用されるので、これは利用可能な最高 leverage audit の 1 つ**。

---

## 10. Measure before optimizing

**見えないものは改善できない**。本章の他の全原則は、**トークンが実際にどこに行くかへの基本可視性**があるとき、適用も検証も容易。

Hygiene ルール: **per-call 使用を log、breakdown して**。最低限:

- input tokens (uncached)
- input tokens (cached、プロバイダが露出すれば)
- output tokens
- reasoning tokens (モデルが別途露出すれば)
- 使用モデル
- セッション/リクエスト識別子

**少量の plumbing**。**Payoff は大きい**。**初週内にしばしば**、請求が**少数のパターン**に集中していることを発見: chatty なシステムプロンプト、常時オンツールカタログ、暴走会話、最強モデルにデフォルトする単一エンドポイント。**測定なしには見えず、見えれば全て quick fix**。

第13章でこれを深堀り。Hygiene の目的では、原則は: **instrument first, optimize second**。**測定なしの最適化は推測** — そして**推測するものは、実際にコストかかるものとは稀に一致**。

---

## これらの原則は複利

各原則を単独で読むと**控えめ**。短いシステムプロンプト、スコープされた指示ファイル、「no preamble」ルール、易しい path での小さなモデル、トピック変更時の新セッション。

**一緒に適用すると乗算**。**長さ半分・prefix キャッシュ・小モデル・出力 bounded・新セッション**のプロンプトは、代替の 5× 安いではない — **同タスクでしばしば 20× 〜 50× 安い**。**そして品質は通常より高い**、なぜなら各 move もモデルが分別すべき noise を減らすから。

10 全てを一度に採用する必要はない。**2〜3 採用で経済を materially シフト**。**10 全採用は、小規模で動くプロジェクトと、運用驚きなしに持続的にスケールするプロジェクトの差**。

```
        cost per task
          ▲
          │   ●  hygiene なし
          │
          │      ●  + output discipline
          │
          │         ●  + scoped loading
          │
          │            ●  + right-sized model
          │
          │               ●  + cache-friendly レイアウト
          │                  ●  + tool audit
          │                     ●  ...
          └──────────────────────────────────► 適用原則数
```

各習慣は床を少し下げる。**複利効果が大きく下げる**。

## Going deeper

これら原則を **last-mile fix ではなく design constraint** として考える。

**トークン経済の最大節約は、early に作られ、稀に再訪されるシステムレベル判断から来る**:

- どのツールがデフォルトで公開されるか
- どの指示が global vs スコープか
- どのモデルがリクエスト種別ごとのデフォルトか
- セッションがどう bound されるか
- 揮発内容がプロンプトのどこに住むか
- デフォルト reasoning effort は何か
- デフォルト output cap は何か

**これらはアーキテクチャ選択**。プロジェクト開始時に作るのは安く、**数千ユーザーが毎日叩くプロダクションに入った後で undo するのは高い**。

対照的に、**後期プロンプト微調整** — ここで一文を削り、そこでツール記述を絞る — は**収穫逓減**。実仕事だが、**天井仕事であって床仕事ではない**。**アーキテクチャ仕事をしたプロジェクトは、残るプロンプト微調整がほとんどない**; **省いたプロジェクトは微調整で戻れない**。

第16章の recipe ファミリ — caching、retrieval、summarization、batching、distillation — は**全てこれら原則の上に座る**。**先頭に揮発タイムスタンプがあるプロンプトに clever な caching 戦略を適用しても助けない**。**既に global に全てロードするプロンプトに retrieval を適用しても助けない**。**原則が先、recipe が増幅**。

> [!IMPORTANT]
> 本章から1つだけ覚えるなら: **トークン無駄のほとんどは、最適化が試みられる前に決まっている**。デフォルトロード、デフォルトモデル、デフォルトツールカタログ、デフォルト出力長 — **これらデフォルトが床を設定**。**最適化は床に対して働き、床より下では働かない**。

## よくある誤解 / FAQ

### Q1. 「キャッシングが効かないモデルなら原則1は無関係？」
A. 大きく効くのはキャッシング時だが、**lost-in-the-middle 緩和** (重要情報は先頭/末尾) や**メンテナンス性**の理由でも layered prompt は良い設計。

### Q2. 「declarative にすると指示フォロー精度が落ちることもあるのでは？」
A. 稀。落ちるなら**ルールが曖昧**か**矛盾**してるサイン。冗長化ではなく**より鋭い定式化**で対処。

### Q3. 「output cap で答えが切れたら？」
A. **タスクに必要な長さ +α** で cap。切れた時のみ後処理 (continuation 要求)。デフォルト無制限は逆方向の罪。

### Q4. 「small モデルがどこで失敗するか事前に分からない」
A. **評価セットを作って測る**。失敗パターンが見えれば**ルーティングルール** (難フラグ、信頼度しきい値) を作れる。

### Q5. 「セッション分けたら過去履歴の文脈が失われる」
A. その文脈が**新トピックに load-bearing なら短い summary で渡す**。多くの場合**文脈は実は不要**だったことに気付く。

## まとめ

- **Stable prefix, volatile suffix**: 不変内容を先頭に、リクエスト毎内容を末尾に
- **Scope load on demand**: 常時オン内容は毎ターン永久課金。条件付きロードをデフォルトに
- **Declarative over imperative**: 短く直接のルールが安く、明確で、よくフォローされる
- **Output discipline**: 出力は高い側; 長さ・形式の 1 行制約は ROI 大
- **Minimum sufficient context**: 利用可能ではなく、よく回答するのに必要なものを
- **Right-size the model**: デフォルトは小モデル; 最強段は本当に利益あるタスクに予約
- **Right-size the reasoning effort**: デフォルト medium; 難問のみ上げる
- **End sessions on topic change**: 長セッションはバラスト蓄積; 新セッションが落とす
- **Treat tools as inventory**: カタログの全ツールが毎ターン課金。Audit と prune
- **Measure before optimizing**: per-call 使用ログが請求の実所在を明かす — 通常予期しない場所
- これら習慣は**複利**: 2〜3 で請求が顕著にシフト、10 でプロジェクトの viability がシフト
- **Last-mile fix ではなく design constraint**として扱う。最大節約は early のシステムレベル判断から

あなたのアプリのプロンプトレイアウトを書き出し、**どこに揮発要素が混入しているか**特定してください。並び替えれば**キャッシュヒット率がどう変わる**か推定してください。

**演習 12.2**（15分）  
あなたのシステムプロンプトを **declarative ルールセット**に書き直し、トークン数の差を測定してください。同タスクの出力品質は維持されますか？

**演習 12.3**（チャレンジ）  
あなたのツールカタログを audit。**過去 30 日で呼ばれていないツール**、**記述が boilerplate なツール**、**マージ候補ペア**を特定してください。
