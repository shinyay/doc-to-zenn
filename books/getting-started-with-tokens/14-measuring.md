title: "第13章 — トークンを計測する"
---

> **一言で:** 見えないものは管理できず、見るべきは「使ったトークン数」ではなく **コストミックス全体**を**ユーザーとタスクで切り**、**分布が本当にどれほど heavy-tail か**を尊重したもの。

---

## なぜこの章があるか

チームが**初めて自分のトークンテレメトリを正しく見る**時、反応はほぼ常に同じ: **請求書が実際にどこに行くかへの驚き**。**高価ワークフローは予測したものとは稀**。「**小さい**」機能が支配的だと判明。「**重い**」機能はキャッシュがよく効いて安い。**少数のユーザーが支出大半を占める**。

これらの洞察を得るには、**早期に instrument し、誠実に instrument し、平均を見たい誘惑に抵抗**するしかない。

本章は **instrumentation 層**について。**何を log し、どう slice し、どの数字をダッシュボードに置き、どれを無視するか**。

---

## 第一原則

**見えないものは管理できない**。

トークンは CPU やメモリと違います。**カーネルレベルビューは無料で来ない**。リクエストが正確に何トークン消費したかを知っているのは**モデルプロバイダだけ**で、その情報を最も低レイテンシかつ呼び出し単位で確実に取れる場所が **API レスポンス**です（プロバイダのダッシュボードや usage API、請求書 export からも事後に取れますが、リクエスト単位の構造化データに紐付けるには **API レスポンスでの捕捉が最も信頼できる**）。**そこで捕捉しなければ、コール単位で構造化データに残すのは難しくなります**。

ルールは単純: **全モデルコールが usage record を生まねばならない**、そして**全 usage record はクエリ可能な場所に永続化**されねばならない。**ログだけでなく、トレースだけでなく — group・filter・aggregate できる構造化ストアに**。

**トークン使用を、財務チームが請求書を扱うように扱え**。**請求書はサンプリングしない**。

---

## トークンカウントの 4 つのカテゴリ

現代のプロバイダは複数のトークン数を返します。**フィールド名はプロバイダごとに変わり、全てのフィールドが常に露出されているとは限らない**ものの、概念的には次の 4 つのカテゴリに分類できます:

```
┌──────────────────────────┬──────────────────────────────────────────────┐
│ カテゴリ                 │ 意味                                          │
├──────────────────────────┼──────────────────────────────────────────────┤
│ prompt / input tokens    │ このリクエストで送った fresh input。          │
│                          │ 通常入力レートで課金。                        │
├──────────────────────────┼──────────────────────────────────────────────┤
│ cached tokens            │ プロバイダが prefix キャッシュから供給した    │
│ (キャッシュ対応時)       │ 入力部分。深く割引（プロバイダが露出時のみ）  │
├──────────────────────────┼──────────────────────────────────────────────┤
│ completion / output      │ モデルが生成した可視出力。                    │
│ tokens                   │ 出力レートで課金。                            │
├──────────────────────────┼──────────────────────────────────────────────┤
│ reasoning tokens         │ 可視回答前にモデルが生成した隠れ chain of     │
│ (reasoning モデル/該当時)│ thought。請求扱いはプロバイダ依存（後述）     │
└──────────────────────────┴──────────────────────────────────────────────┘
```

**プロバイダが返す usage フィールドは、見えたものを全部そのまま保存してください**。「常に 4 つ揃っている」と仮定せず、**そのコールで実際に返ってきたフィールドだけ**を構造化ログに保存し、欠落しているものは null として扱うのが安全です。

理由: **集約「使用トークン」数はコストには無意味**。**同じ総トークン数の 2 リクエストが、キャッシュ量と reasoning 量によって実コストで数倍〜十数倍異なりうる**。

usage logger が**単一整数しか保存しないなら**、**重要な質問に答えられない logger** を持っている、ということになります。

## 内部で使うべき単位

自分のシステム内では、**トークンで価格付けしない**。**コストで価格付け**。

**コストはリクエスト・モデル・テナントにわたって単一・比較可能な意味を持つ唯一の数**。**極めて違うトークン形状の 2 リクエストが同じコストを持ちうる**; **同じトークン総数の 2 リクエストが極めて違うコストを持ちうる**。

数式は単純です。ただし**プロバイダによって reasoning トークンの請求方法は異なる**ため、ここでは reasoning を独立カテゴリとして書きますが、実プロバイダではしばしば output トークンと同じレートで一括課金されたり、別ラインに出たり、無料扱いだったりします。**プロバイダの請求セマンティクスを実装側で扱う薄い関数（例: `provider_billing_semantics(provider, model)`）に閉じ込め、ここでの数式はその関数が返すレートを使って計算する**のが現実的です:

```
# provider_billing_semantics(provider, model) は
# {input_rate, cached_rate, output_rate, reasoning_rate} を返す。
# 例: あるプロバイダでは reasoning_rate = output_rate（合算課金）、
#     別のプロバイダでは reasoning_rate を別フィールドで返す等。

rates = provider_billing_semantics(provider, model)

cost = rates.input_rate     × input_tokens
     + rates.cached_rate    × cached_tokens
     + rates.output_rate    × output_tokens
     + rates.reasoning_rate × reasoning_tokens
```

**ログ時に計算し、列として保存し、全ダッシュボードの分母として使う**。「リクエストあたりコスト」「ユーザー・日あたりコスト」「解決チケットあたりコスト」 — **これら全て答えられる**。「解決チケットあたりトークン」は答えられない、**4 つの違うものを混同するから**。

> [!TIP]
> コールに有効だったレートを、計算済みコストだけでなく**コストと一緒に保存**。レートが変わっても、**元レコードを失わずに歴史的数値を再導出**したくなる。

## Per-X ビュー

**dimension のない usage record は、行動できない数**。**実際に必要な切り方ができるよう、各レコードに十分なメタデータを持たせる**:

- **Per call**: 単一 API リクエスト。**原子単位**。これが log される
- **Per turn**: 1 ユーザー対面対話。複数内部コール (ツールステップ・retrieval ラウンド・リトライ) を含むかも。エージェントでは 1 ユーザーメッセージ = ダース単位コール
- **Per session**: 完全会話。**会話長でコストがどう増えるか**理解に有用 (第9章)
- **Per user**: アカウントまたはエンドユーザー
- **Per team / workspace**: 最終的に請求書を見るべきコストセンター
- **Per task type**: codegen vs chat vs RAG vs 分類 vs バックグラウンドバッチ

これら全てを得る最安方法: **捕捉時に各 usage record に小識別子セット (call id、turn id、session id、user id、tenant id、task tag) を添付**。**後でログから join を再構築しようとすると、subtle な方法で失敗**する。

## Heavy-tail 観測

プロダクションのトークン使用について**最重要事実**: **正規分布ではない**。

ユーザーにわたって、セッションにわたって、タスクにわたって — **分布は heavy-tail**。**少数のユーザー (またはセッション、タスク種別) が総コストの大部分を消費**。**平均はテイルに引き上げられ、実ユーザーを代表しない**。

**ユーザー・コスト平均だけ見るなら**、ほぼ全てについて**自信を持って間違える**: 価格設定、容量計画、最適化場所、誰と話すか。

典型ユーザーコストヒストグラム:

```
users
  │
  │ █
  │ █
  │ █ █
  │ █ █ █
  │ █ █ █ █
  │ █ █ █ █ █
  │ █ █ █ █ █ █ █ █
  │ █ █ █ █ █ █ █ █ █ █  █  █     █                       █
  └──────────────────────────────────────────────────────────────►
    low                                                       high
                          cost per user per day

      ↑ ほとんどのユーザーはここ      ↑ 細いテイルがここに住む
        (中央値)                        (請求書の大半を払う)
```

**この分布の平均はバルクの右、ほぼ誰も実際に住んでいない領域**にある。**平均を引用すれば、架空ユーザーを記述している**。

代わりに引用すべき:

- **中央値 (median)**: 典型ユーザー。バルクについて誠実
- **p90**: 90% より重いユーザー。「power user」が始まる場所
- **p95, p99**: テイルが住む場所。**請求書の出所**
- **Max**: その日の最高ユーザー1人。しばしば驚き。**時にバグ**

**良いダッシュボードは中央値・p90・p99・max を並べて見せる**。**それらの間の形状が、どの単一数より多くを語る**。

## 実用的に track すべき指標

**小さく規律ある指標セット**で遠くまで行ける。**全てを track したい誘惑に抵抗**。

### Cost per user per day
中央値・p90・p99 を track。**中央値 = 典型ユーザーが時間と共により高くなっているか**。**p99 = テイルが太っているか** — 通常**より緊急なシグナル**。

### Cost per task / cost per correct answer
第11章で議論した**実生産性指標**。**品質なしの cost per task は半分の絵**; **cost per correct task が完全な絵**。

### Cache hit ratio
`cached_tokens / (cached_tokens + uncached_input_tokens)`。**prefix 規律 (第12章) が機能しているかの直接測定**。プロバイダによって `prompt_tokens` がキャッシュ込みかキャッシュ抜きかの定義が異なるので、必ず**「キャッシュから供給された入力 ÷ 入力全体（キャッシュ込み）」**の形に揃えてから比率を出してください。比率が下がる = **何かが prefix を invalidate し始めた** — 通常システムプロンプトかコンテキスト組立て方の最近の変更。

### Output / input ratio
ここの突然上昇 = しばしば**冗長性へのプロンプト回帰** — 誰かがコストを考慮せず「explain your reasoning」「be thorough」を追加。

### Reasoning ratio
`reasoning_tokens / output_tokens`。突然上昇 = しばしば**難度スパイク** (ワークロードが難しくなった) または**reasoning-effort 設定ミス**。**どちらも調査価値あり**。

### Tool-call count per turn
エージェントワークフローで**最安の早期警告シグナル**。**高値はほぼ常にエージェントがループ**していることを意味 — 同ツールリトライ、ツール間振動、収束失敗。

**6 指標**。**フロントページにこれ以上不要**。

## 予算、cap、アラート

**指標は何が起きたかを語る**。**予算とアラートはいつ行動するかを語る**。

合理的開始セット:

- **Per-user soft cap**: 通知発火 (ユーザー、あなた、両方) するが**ブロックしない**警告閾値。**健全使用の概ね p95** に設定。**多くは false alarm**; **真陽性が見たいもの**
- **Per-user hard cap**: ブロック閾値、または安いモデル / レート制限段にダウングレード。**soft cap よりかなり上、単一ユーザーがユニット経済を materially 損ねうるレベル**に
- **Per-team monthly budget with daily burn-rate alert**: チームが月予算を 18 日目までに超える track なら、**12 日目に知りたい、30 日目ではなく**。**Burn-rate アラートは閾値アラートに勝つ**
- **Anomaly alerts on percentile shifts**: p99 が一夜で倍 = **何かが変わった** — デプロイ、新機能、新ユースケース。**アラートはどの dimension がシフトしたかを伝える**べき、単に「コストが上がった」ではなく

> [!IMPORTANT]
> **支払いユーザーを驚かす hard cap は、節約するコストより悪い**。**Cap は常に明確メッセージとアップグレードパスとペア**で。**信頼を損ねるコスト制御はコスト制御ではない**。

## サンプリングと PII

**2 ストリーム、2 ポリシー**。

### Usage メタデータ
トークン数、コスト、識別子、レイテンシ、モデル ID、cache hit ratio。**全コールで全て log、サンプリングなし**。**このストリームは小さい** (コールあたり数百バイト)、**保存安く、不可欠**。

### 完全プロンプト・レスポンス本体
実テキスト。**安全で明確なニーズがある時のみ log**: 失敗プロンプトのデバッグ、評価セット構築、品質回帰調査。**保持を短く**。**機密フィールドを strip / redact**。**データに適用される規制制約を尊重** — 多くのドメインがユーザーコンテンツの限定目的を超えた保存を禁じ、**いくつかは第三者送信を完全に禁じる**。

避けるべき間違い: **2 ストリームを混同**すること。「**body log が大きすぎる**」から usage log をサンプリング = **別途・安く保存できたものを節約してコスト可視性を失った**。

## テレメトリ hygiene

**コスト&使用ストリームをアプリケーショントレースストリームと分離**。

アプリケーショントレース (span、log、性能指標) は通常プロダクションで**積極サンプリング**。**トレースには fine**; レイテンシ理解に全 span 不要。**しかしコストデータのサンプリング = 回復不能な金を失う**。**サンプリングされた請求書は間違った請求書**。

アーキテクチャ的に:

```
   ┌──────────────────┐         ┌────────────────────────┐
   │   model call     │────┬───►│  usage stream          │
   │                  │    │    │  (never sampled)       │
   │                  │    │    │  → cost warehouse      │
   │                  │    │    └────────────────────────┘
   │                  │    │
   │                  │    │    ┌────────────────────────┐
   │                  │    └───►│  trace stream          │
   │                  │         │  (sampled, short TTL)  │
   │                  │         │  → tracing backend     │
   └──────────────────┘         └────────────────────────┘
```

**2 ストリームはインフラを共有してよいが、ポリシーは違うべき**。**コストストリームは財務記録**。**そう扱え**。

## Going deeper

### マルチテナントシステムでのコスト attribution
1 アプリで複数顧客にサーブするなら、**全コールがクライアント離脱の瞬間からテナント ID を運ぶ**必要。**事後にサーバログからテナント attribution を再構築するのは痛く、しばしば不可能**。**正しい場所はリクエスト境界**、そして**全内部 hop を伝播**すべき。

### CI でのプロンプト回帰検出
代表タスクスイートがあるなら、**プロンプトまたはコンテキスト組立てパスに触れる全変更で実行**し、**タスクあたり平均トークンを記録**。**その数の non-trivial 変化** (上下どちらも) **がレビュー価値あるシグナル**。誰かがシステムプロンプトに段落を追加してタスクあたり入力トークン 30% ジャンプ = **出荷前に捕まえたいもの**。**インストール可能な最安トークン節約制御**。

### Effective rate per million tokens
キャッシュ健康の有用 proxy 指標: **総コスト ÷ 総入力トークン**、百万あたり表現。**cache hit ratio が改善するとこの数は下がる**、入力ボリュームが伸びても。**劣化すると** (例えば prefix invalidate するプロンプト変更後) **数は急上昇**。**時間追跡で同ワークロードでより/より少ない効率を得ているかの単一・鈍い view**。

### Rate-limit テレメトリ
厳密にはコストではないが隣接。**どのコールが throttle、retry、reject されたか log すれば、コストスパイクをプロバイダ側プレッシャーと相関付け**できる、そして**「より多くやってるからより多く支出」と「リトライしてるからより多く支出」を区別**できる。

## よくある誤解 / FAQ

### Q1. 「provider のダッシュボードで十分では？」
A. 集約・遅延・粒度不足。**自分のディメンション (テナント、機能、ユーザー)** で切れない。**自前ストアが必須**。

### Q2. 「全コール log は重くない？」
A. usage record は数百バイト。月数百万コールでも GB スケール。**ストレージは安い、可視性消失は高い**。

### Q3. 「平均を見たくなる衝動はどう抑える？」
A. **ダッシュボードに平均を載せない**。中央値・p90・p99・max のみ表示。**見えなければ参照されない**。

### Q4. 「コスト計算がモデル変更で複雑」
A. **rate テーブルをコード化**し、log 時にスナップショット保存。モデル切替時もテーブル更新だけ。

### Q5. 「prompt body も log すべき？」
A. デバッグ・評価構築には欲しい。**短保持・redact・サンプリング OK** で別ストリームに。

## まとめ

- 見えないものは管理できない。**全コールに usage record、構造化ストア、サンプリングなし**で捕捉
- **常に 4 トークンカウント log** — fresh input、cached input、可視 output、隠れ reasoning。**単一「使用トークン」数はコストミックスを隠す**
- **内部はトークンではなくコストで価格付け**。ログ時にコスト計算・保存
- **全レコードを per call/turn/session/user/team/task type で slice 可能に**
- **トークン使用は heavy-tailed**。**平均は誤導**。**中央値・p90・p99・max でリード**
- **小規律セット**を track: cost per user、cost per correct task、cache hit ratio、output/input ratio、reasoning ratio、tool-calls per turn
- **soft cap、hard cap、月予算と burn-rate アラート、percentile-shift 異常アラート**をペア
- **コストストリームをトレースストリームから分離**。後者は自由にサンプリング、**前者は決してサンプリングしない**
- **CI でタスクあたりトークン平均**をプロンプト変更の回帰シグナルとして watch
