---
title: "第14章 — アンチパターンと落とし穴"
---

> **一言で:** 実システムのトークン浪費のほとんどは exotic ではない — **何度も再発する小さなアンチパターンセット**が黙って複利し、**チェックリストで見つけられる**。

---

## なぜこの章があるか

ここまでで、トークンがどこから来て、何にコストし、品質にどう影響し、どう測るかのモデルを持ちました。**本章は診断対応物**。実デプロイで何度も現れる**特定の失敗モード**、外から見える形、なぜ高くつくか、**同じものの健康版**を名指します。

end-to-end で読めますが、**スキム可能**にも設計。各アンチパターンは同じ **4 部構造**:

- **症状** — ログ・ダッシュボード・挙動で観察するもの
- **なぜ痛いか** — コスト・品質・レイテンシ機構
- **どう見つけるか** — 実行できる具体チェック
- **健康版は何か** — トークンを浪費しない版

自分のセットアップで 2〜3 個以上見つけたら**正常**。**特定システムを恥じさせるのが目的ではなく**、**起きていることを名付ける語彙を与え、修正の優先付けを助ける**こと。

```
        アンチパターン            コストの形
        ─────────────────         ──────────────────────
        kitchen-sink prompt   →   毎ターンの flat tax
        history bloat         →   ターン数で成長
        tool overload         →   flat tax + per-step
        retrieval over-pull   →   per-query スパイク
        reasoning leakage     →   per-call 乗数
```

---

## 1. Kitchen-sink システムプロンプト

**症状**: 何ヶ月もの「これについてもルール追加しよう」で**数千トークンに成長**したシステムプロンプト。ミッション宣言、トーンガイド、出力形式、禁止フレーズリスト、**前四半期に削除したツールへの指示**、いつ確認質問するかについて**3つの矛盾ルール**を含む。

**なぜ痛いか**: システムプロンプトは**毎ターン課金**。**bloated プロンプト = 全対話 (些細なものを含む) への固定税**。さらに悪く、**矛盾ルールはモデルに、質問に答える代わりに矛盾解決へ attention を使わせる** — コストが上がるのに**品質が下がる**。

**どう見つけるか**: システムプロンプトのトークンを数える。**数百を超えるなら、声に出して end-to-end で読む**。正当化できないルール、互いに矛盾するルール、もう存在しない能力のルールがあれば、kitchen-sink プロンプト。**別のサイン**: チームの誰も自信を持って中身を言えない。

**健康版**: **編集された** (ただ append された、ではない) プロンプト。ルールはグループ化、重複排除、定期的に剪定。**能力固有指示は能力と一緒に住む**、global preamble ではなく。**誰かがプロンプトを所有し、コードと同じく diff をレビュー**。

> [!TIP]
> 有用な演習: **システムプロンプトを完全に削除**し、小評価を実行。**削除が品質を計測可能に害したルールだけ再追加**。多くは cut しない。

## 2. 自動生成された冗長指示ファイル

**症状**: リポジトリに**長い機械生成コンテキストファイル** — プロジェクトを exhaustive だが unfocused に文書化。「init」コマンドが書いた。**全セッションでエージェントコンテキストにロード**され、エージェントの挙動はなぜか**前より少し悪い**。

**なぜ痛いか**: **長コンテキストは無料 attention ではない**。低 signal 記述を数千トークン詰め込めば、モデルは**現タスクを助けない素材の処理に容量の一部を使う**。古典的「more context, worse answer」効果。**毎ターン同じ無情報トークン**にも払う。

**どう見つけるか**: ファイルを開く。各セクションに「**新しい人間チームメイトはこれを読む必要があるか？**」と問う。No なら**モデルにも不要**。**別チェック**: 小固定テストセットで**ファイルあり/なしで品質比較**。なしが悪化なし = ファイルは純コスト。

**健康版**: **短く、手 curate されたコンテキストファイル** — 有能な新参が実際に必要とするもの (プロジェクト目的、主要規約、重要エントリポイント、既知 footgun)。**生成内容は出発点だが、verbatim ではなく heavily 編集**して commit。

## 3. MCP / ツール過剰

**症状**: エージェントが**長い統合リスト**で構成 — 全コネクタ、全内部 API、全 DB。任意セッションで**ほとんど使わない**が、**全ステップでスキーマがモデルに広告**される。

**なぜ痛いか**: ツールスキーマ = トークン。各統合は通常、各公開関数の名前・記述・パラメータスキーマを貢献、**1 統合が容易に数千トークン**。十数統合有効化で、**per-step オーバーヘッドが他全てを支配**しうる。コスト超え、長ツールリストは**選択精度も下げる**: モデルは 50 から選ぶ時 5 からより**間違ったツールを選びやすい**。

**どう見つけるか**: 典型エージェントステップで、**ユーザーコンテンツ前に消費される入力トークン数**を測定。ツール定義が全入力の小さな fraction を超えるなら**過剰プロビジョン**。**ツール使用頻度** track: セッションの数 % 未満で発火するツールは削除/lazy load 候補。

**健康版**: ツールは**手元タスクにスコープ**。エージェントは小デフォルトセット、追加ツールは会話が必要を示す時のみロード。**スキーマは terse に書く** — 短記述、最小パラメータ doc — モデルが毎ステップ読むから。

```
    bad:                          good:
    ┌──────────────────┐          ┌──────────────────┐
    │ 30 ツール常時    │          │ 5 デフォルト     │
    │ ロード           │          │ + lazy ロード    │
    │ ~6000 tokens     │          │ ~800 tokens      │
    │ per step         │          │ per step         │
    └──────────────────┘          └──────────────────┘
```

## 4. Reasoning leakage

**症状**: 全レスポンス (「2 + 2 は？」を含む) が**実答前に「ステップバイステップで考えましょう」数段落**を伴う。または、**reasoning effort ダイアルが永久に最高設定** — 誰かが「品質を上げる」と読んだから。

**なぜ痛いか**: Reasoning トークンは課金。内部 reasoning trace を出すモデルで、**ユーザーが見なくてもこれらは出力予算に数えられる**。Trivial 質問では、**reasoning が実答の 10〜20 倍サイズ**になりうる。**何百万コールにわたって、利益ゼロの大請求書**。

**どう見つけるか**: 最近数百レスポンスをサンプル、**reasoning トークン (または可視 thinking 前置き) と最終回答トークンの比**を見る。易質問でこの比は小さくあるべき。**易と難が同じに見えれば reasoning が leak**。

**健康版**: **Reasoning effort はタスク難度に合わせる**。易質問は preamble なしの短回答、難質問は full reasoning。**システムが対応すれば、予測難度でルーティング**; なければ「**問題が必要としない限り preamble をスキップ**」とモデルに指示。

## 5. 病的な history 成長

**症状**: 長セッションが速く始まり、**毎ターン遅くなる**。50 ターン目で、ユーザーの質問が同サイズなのに新メッセージが顕著に長く・高く。最終的に**品質崩壊** — モデルは会話冒頭を忘れ、自己矛盾し、最新メッセージへの engage を拒否。

**なぜ痛いか**: **会話履歴は毎ターン replay**。アシスタントが冗長返信 → 次ターンの入力。50 連続冗長返信 → 51 ターン目の入力に全て。**ターンあたりトークン数は monotonic に成長、コンテキスト窓に当たるまで** — そして silently truncate か break。

**どう見つけるか**: 長セッションの**ターンあたり入力トークンをプロット**。健康セッション = ほぼ stable な線、新素材導入時に時々 bump。**病的セッション = steadily 上昇する線**。傾きが「**各ターンが追加する history debt**」を語る。

**健康版**: **古いターンは閾値超過後 summarize / drop / 圧縮**。アシスタント自身の出力を terse に保ち、history を膨らませない。

## 6. 出力の polite filler

**症状**: 全レスポンスが「**もちろん、お手伝いします！ご要望は以下です:**」で開き、「**他にご質問があればお知らせください！**」で閉じる。**実答は中央にサンドイッチ**。

**なぜ痛いか**: 出力トークンは通常**システムで最高価**、**これらフレーズは情報目的なし**。単一レスポンスでは些細。**1日数百万コール**で意味ある line item に。**レイテンシも追加** — モデルが emit する全語 = ユーザー待ち時間。

**どう見つけるか**: レスポンスサンプルで、**ユーザー質問に実際に answer する最初の文の前のトークン数**を測定。一貫して数を超えれば filler 問題。

**健康版**: **システムプロンプトが冒頭の社交辞令と末尾の追加質問オファーを明示的に禁止**。モデルは substantive 内容に直行。単一例では curt に響くが、**一貫スタイルとして読み手の時間を尊重するプロフェッショナルに読める**。

---

## 7. 巨大ログ・スタックトレースの貼り付け

**症状**: ユーザーが**1 万トークンのスタックトレース**を貼って 1 特定エラーを質問。エージェント回答。50 ターン後、**そのスタックトレースはまだ history、毎コールで replay**。

**なぜ痛いか**: **入力の一回スパイクが reccurring tax に**。ユーザーは関連時に 1 度払い、**現議論と無関係なのに以後毎ターン払い続ける**。**予期せぬ高価セッションの最一般的原因の1つ**。

**どう見つけるか**: 早期 1 ターンが残りより劇的に大きく、その後 input-per-turn が高いまま留まるセッションを探す。**シグネチャ = 戻らない per-turn トークン数のステップ関数**。

**健康版**: **大貼り付け artefact は使ったターン後に summarize**。要約は関連洞察 (「エラーは foo.py 42 行目の null 参照」) を保ち、生テキストを破棄。**Agent ランタイムはサイズ閾値で自動化可能**; ユーザーは履歴編集/剪定で手動。

> [!NOTE]
> **対話的コーディングツールで最初に修正する価値のある単一パターン**。「ユーザーが大出力を貼る」「セッション長期実行」の組み合わせがコストを特に速く成長させる。

## 8. ツールコール ping-pong ループ

**症状**: エージェントがツール A 呼び → 結果見る → B 呼び → 結果見る → **同パラメータで A 再呼び** → 同結果 → B 再呼び... ユーザーは spinner を見る; ダッシュボードは API コール stream を見る; **何も progress しない**。

**なぜ痛いか**: 各ループステップが**前ツールコール+結果の成長 history を含む full 入力コスト**を払う。誰かが気付くまで 10 ステップループ = **1000 通常ターン分**コストしうる。**最終終了時 (通常ステップ制限)、ユーザーはその金で有用回答を得ない**ことが多い。

**どう見つけるか**: ユーザーメッセージあたりツールコール数を track。合理的天井超過セッションにアラート。**単一セッションで同パラメータの繰り返し同一ツールコール = ループの smoking gun**。

**健康版**: **ランタイムが同パラメータ繰り返しコールを検出**し、キャッシュ結果で short-circuit するかループを破る。**エージェントは progress していないと認識し、grind の代わりにユーザーに助けを求めるよう促される**。**ターンあたりツールコール hard cap** が最悪ケースを保護。

## 9. 取りすぎる RAG

**症状**: RAG システムが「**念のため**」全クエリで 30 チャンク pull。チャンクはプロンプトに詰める。**モデルは自作 haystack で needle を探さねばならず、しばしば見逃す** — 5 良選チャンクで得たであろう回答より悪い。

**なぜ痛いか**: コストは直接 — 全クエリで含めた全チャンクに払う。**品質コストは subtler**: モデル attention は有限、**30 のほぼ無関係チャンクに range させると、実重要少数への focus を希釈**。**より高価でより不正確**になりうる。

**どう見つけるか**: **検索チャンク数を変える評価**を実行。少数を超えて精度が plateau / 劣化なら**過 retrieval**。「**実際に質問に答えたチャンク**」のランク分布も見る — ほぼ常に top 数なら**他は何も払わない**。

**健康版**: Retrieval は**少数の高品質チャンク** + **最関連を昇格する re-ranking**。**Recall は labelled 評価セットに対してチューン**、盲目的最大化ではなく。長尾「念のため」チャンクは、**主層が不十分時のみ参照される第二層**に住む。

## 10. 段階を間違えたモデルデフォルト

**症状 (over-sized)**: 全対話が**最高能力・最高価モデル**へ — 「今日の日付？」「この段落 1 行要約」を含む。**ダッシュボードは難度関係なく flat な高コスト/コール**。

**症状 (under-sized)**: 全対話が小さく安いモデルへ、しかし**リトライ率高く、ユーザーが frustrated に再質問、より大きなモデルへの escalation パス頻発** — 小モデルが**実は金を節約していない**。

**なぜ痛いか**: **両極端が負ける**。Over-sized = タスク不要の能力に大乗数払い。Under-sized = コール毎は小乗数だが**リトライと escalation でコール数を乗算**、結局**より高価**になることが多く、**遅く、煩わしい**。

**どう見つけるか**: cost-per-successful-task をプロット (cost-per-call ではなく)。**big-model セットアップが小代替より成功率で顕著に良くなければ over-pay**。**small-model セットアップがリトライ/escalation 率高ければ under-pay**。

**健康版**: **ルーティング層が易タスクを小モデル、難を大モデルへ**、「**何が hard か」明示ポリシー**で。**ポリシーは仮定ではなく measured outcome に対してレビュー**。

## 11. Markdown table 濫用

**症状**: レスポンスが**ダース列・多行の wide markdown table**。下流システムが pipe split で parse。**値に pipe 文字が含まれて parse 破綻**することも。

**なぜ痛いか**: Markdown table は**句読点トークン密** (pipe、dash、alignment marker、escape) — データに何も貢献しない。**JSON や CSV なら数百トークンの table が markdown では容易に倍**。**alignment 形成・維持にも追加で払い、brittle parsing 問題も負う**。

**どう見つけるか**: 構造化出力が markdown table で受信側で parse なら、このアンチパターン。**同データを table、JSON、CSV でレンダー比較** — table がほぼ常に最大。

**健康版**: 構造化出力は**構造化形式**で要求。ネスト/型付きデータには JSON、tabular なら CSV、**markdown table はチャットクライアントの人間消費者向けのみ**。

---

## 12. 忘れられたデバッグ出力

**症状**: システムプロンプトが「**全中間変数を常に print**」「**全レスポンス末尾に full reasoning を含める**」と言う。**開発中デバッグ用に追加、削除されなかった**。プロダクションで**モデルは毎コール忠実にコンプライ**。

**なぜ痛いか**: 出力トークンはモデルに作らせる量と線形 scale。**毎コール full 中間状態要求は通常出力サイズを大係数で乗算、エンドユーザー価値ゼロ**。デバッグ出力は通常 discard / parse away / UI で隠される — **誰も読まない内容に払っている**。

**どう見つけるか**: プロダクションシステムプロンプトを「explain」「show your work」「include all」「print every」等で読む。出力 payload の**実表示部分**を見る。

**健康版**: デバッグ指示は**デバッグ環境または feature flag にスコープ**。プロダクションプロンプトは**ユーザーが見る必要のあるものだけ**を要求。

## 13. Cache-busting prefix

**症状**: システムプロンプトが「**Session ID: 7f3c-...**」「**Request timestamp: ...**」のような行で始まる。**残りプロンプトが百万コールで同一**なのに**prefix の cache hit rate がほぼゼロ**。

**なぜ痛いか**: Prefix キャッシングは cached input トークンに**実質コスト・レイテンシ削減**を与え、**厳密 prefix で keyed**。**プロンプト先頭の unique 値が、次 4000 トークンが同一でも、全コールをキャッシュに違って見せる**。**節約を完全に forfeit**。

**どう見つけるか**: システムプロンプトのキャッシュヒット率を check。プロンプト安定性から期待されるよりはるか下なら、**全コールの最初数トークン**を見る。Vary するなら cache-busting prefix。

**健康版**: 揮発メタデータ (ID、タイムスタンプ、per-request flag) は**プロンプト末尾、安定内容の後**に住む。**キャッシュは長安定 prefix を吸収**でき、**短揮発 suffix にだけ full price**。**章全体で最高 leverage 修正の1つ**: 変更は機械的、**節約は immediate**。

```
    cache-busting:                            cache-friendly:
    ┌──────────────────────────────┐          ┌──────────────────────────────┐
    │ [unique session id]          │          │ [stable system prompt]       │
    │ [stable system prompt]       │   →      │ [stable tool definitions]    │
    │ [stable tool definitions]    │          │ [unique session id]          │
    │ [user message]               │          │ [user message]               │
    └──────────────────────────────┘          └──────────────────────────────┘
        cache hit rate: ~0%                       cache hit rate: high
```

## 14. 毎ターン re-summarising

**症状**: エージェントフレームワークが**全ユーザーターン前に「これまでの会話を summarise」ステップ**実行。要約は合理的。**コストはそうではない** — **largely 同じ内容に対する同要約仕事を何度も再実行**して払っている。

**なぜ痛いか**: 要約自体がモデルコール、**履歴長に比例する入力**。毎ターンやる = ターン数で**ほぼ二次的仕事**。**ターン N+1 で要約されるものの大半は既にターン N で要約済み**。**モデルに同素材を indefinitely 再読させて払っている**。

**どう見つけるか**: ユーザーターンあたり**non-user モデルコール**を見る。**新素材量に関わらず毎ターン steady な「summarise」コール**があれば、**過 summarisation**。

**健康版**: 要約は**閾値で実行** — 履歴がトークン数を超える時、または論理境界 (新タスク、新ファイル、新サブ会話) を渡る時。**閾値間は既存要約を再利用**。要約鮮度の小損失と、繰り返し仕事の大削減の交換。

## 診断チェックリスト

自セットアップをスキャン:

1. **難度関係なく per-call コスト高** → kitchen-sink プロンプト (#1)、reasoning leakage (#4)
2. **セッション内で per-step input トークンが時間と共に climb** → 病的履歴成長 (#5)、貼り付け artefact (#7)
3. **新統合追加後に新セッションが顕著に高価** → ツール過剰 (#3)
4. **出力ボリュームの大半が回答ではなく preamble / boilerplate** → polite filler (#6)、忘れデバッグ出力 (#12)
5. **RAG が多チャンク pull するが精度が少数と変わらない** → 過 RAG (#9)
6. **モデル支出がユーザー数より遥か速く成長** → ping-pong ループ (#8)、re-summarisation (#14)
7. **安定プロンプトなのにシステムプロンプトのキャッシュヒット率がほぼゼロ** → cache-busting prefix (#13)
8. **「smart だが遅く高い」とユーザー苦情** → over-sized モデルデフォルト (#10)
9. **「速いが間違い」と苦情** → under-sized デフォルト (#10)、signal を溺らす bloated コンテキスト (#1, #2)
10. **誰も読まない壁テキスト** → table 濫用 (#11)、デバッグ出力 (#12)
11. **プロジェクトの自動生成コンテキストファイルが大** → セクション毎 audit (#2)
12. **単一セッションで同パラメータの同ツールコール複数回** → ループ中 (#8)

## Going deeper

### アンチパターンは互いに乗算

これらは相互作用、相互作用は通常**加算ではなく乗算**。5,000 トークン kitchen-sink、20 統合スキーマ広告、reasoning effort 最大、を**全部持つ**システムを考える。**per-call コストは 3 つの小オーバーヘッドの和ではなく、3 つの乗数の積**。Reasoning は既にプロンプトとツールスキーマで膨れたコンテキストを走り、各 reasoning ステップが高くなる。プロンプトは reasoning の各ステップで replay、膨張がより多く払われる。**コール終了時、同オーバーヘッドに 3 つの違う方法で払った**。

**Good news: 修正時に有利に働く**。プロンプトを半分に削れば reasoning も安く、reasoning 入力が縮むから。Tool を lazy ロードすればプロンプトと reasoning オーバーヘッド両方が落ちる。易タスクで reasoning を下げれば**バルクトラフィックから乗数が完全消失**。**各修正が他を増幅**。

```
    additive view:                multiplicative reality:

    cost = base + a + b + c       cost = base × (1+a) × (1+b) × (1+c)

    1 修正 → a 節約              1 修正 → 比例的節約
    3 全修正 → a+b+c 節約        3 全修正 → 部分の和より遥かに多く節約
```

これが**audit が単一最適化より報われる理由**。1 パスの時間しかなければ、**単一を深堀りせず top 3〜4 アンチパターンに横断**。

### ほとんどは組織問題

これらアンチパターンの**ほとんどはモデルのバグではない**。**組織的**。Kitchen-sink プロンプトは**誰も所有しない**から成長。Tool 過剰は**全チームが自統合を有効にしたい**から。デバッグ出力は**誰も削除しない**から残る。**技術修正は通常 obvious; 難問はプロンプト・ツールリスト・プロダクションプロンプトに owner を置き、誰かが lean に保つ責任を持つこと**。所有なしには、**同アンチパターンが四半期内に再生**する。

> [!IMPORTANT]
> **アンチパターンは除去より予防が容易**。一度システムプロンプトが矛盾ルール 1000 を蓄積すれば、**何かを壊す恐れで誰も触りたがらない**。**レビュー習慣**と**第13章の計測インフラ**を構築し、プロンプトとツールリストが**その状態に達さない**ように。

## よくある誤解 / FAQ

### Q1. 「動いているなら触らない方が良いのでは？」
A. アンチパターンは**沈黙したコスト**を生む。動いていても**請求書とレイテンシで損**している。

### Q2. 「全部直すリソースがない」
A. **チェックリストで上位 3〜4 を特定**し、**audit で複数同時に**。乗算効果で個別深堀りより効く。

### Q3. 「組織が prompt owner を置けない」
A. **計測ダッシュボードを公開**し、**コスト責任者を可視化**。データが自然に owner を呼ぶ。

### Q4. 「自動生成ファイルを削ると壊れる気がする」
A. **小評価セットで AB テスト**。多くの場合性能不変。**確信が証拠で置き換わる**。

### Q5. 「ping-pong ループはエージェントの賢さ次第では？」
A. **ランタイムレベルの hard cap と重複検出**で先に守る。賢さに頼らない。

## まとめ

- **同じ少数のアンチパターン**がほぼ全実デプロイに現れる。名指せば修正できる
- 各アンチパターンは**認識可能症状、具体チェック、健康代替**を持つ。診断をゼロから発明する必要はない
- 最大単一勝利は**機械的**: システムプロンプト trim、ツール lazy load、揮発内容を末尾に、full re-summarisation のような高価仕事繰り返しを止める
- アンチパターンは**乗算的に複利**。3〜4 横断 audit が単一深堀りに勝る
- ほとんどは**技術仮装の組織問題**。プロンプト・ツールリスト・コンテキストファイルに owner なしには再生
- **診断チェックリストを定期レビュー**として — 四半期 1 パスでほとんどのシステムが健康を保つ
