## なぜこの章があるか

前 15 章で、トークンとは実際何か、テキストからどう生成されるか、実ワークロードのどこに蓄積するか、各々が金・レイテンシ・回答品質で何にコストするかの動く理解を構築しました。**コンテキスト窓は機能ではなく予算**、システムプロンプトやツールスキーマのような**隠れ消費者がメッセージ到達前に予算を使う**、コストと品質の関係は**線形ではなく曲がる**ことを見ました。第15章で、**規律ある使用は pennies 節約より、システムが使って気持ち良いと感じるかを決めるリソース (attention、レイテンシ、ヘッドルーム) の保全**であることも見ました。

それで背景は十分。**今、最適化助言を批判的に読むのに必要な全てを持つ**。誰かが clever プロンプトトリック・新フレームワーク・「must-do」パターンを提案する時、**正しい質問**ができる: **どのバケットに影響？保守コストは？実ワークロードへの接触で生き残る？**

本章はチュートリアルではなく**プレビュー**。**将来の最適化フォーカスガイドで学ぶ価値ある技法ファミリ**をカタログし、**遭遇した瞬間に新技法を正しいファミリに slot するメンタルモデル**を与えます。

## メンタルモデル: 全最適化は 3 つの move のいずれか

ファミリを名指す前に、**move を名指す**。**存在する (過去・現在・未発明の) ほぼ全トークン最適化技法は、変装した 3 つの 1 つ**:

1. **より少ないトークンを送る (Send fewer tokens)**: 入力削減。システムプロンプト trim、コンテキストスコープ、未使用ツール定義 drop、履歴要約
2. **より少ないトークンを受ける (Receive fewer tokens)**: 出力削減。terse 回答要求、固定スキーマ、bounded 長、preamble なし
3. **トークンあたり払う額を減らす (Pay less per token)**: 送受トークンに払う価格を変える。安いタスクには安いモデル。**prefix キャッシュにヒットするようプロンプト設計**。合う段に仕事を移す

```
                 ┌─────────────────────────────┐
                 │   THE THREE MOVES           │
                 ├─────────────────────────────┤
   input  ─────► │ 1. Send fewer tokens        │
                 │ 2. Receive fewer tokens     │ ◄───── output
                 │ 3. Pay less per token       │
                 └─────────────────────────────┘
```

新技法に遭遇したら、**まず move を名指す**。「**このトリックはシステムプロンプトを減らす**」 = move 1。「**JSON のみ強制**」 = move 2。「**安定内容を先頭に**」 = move 3。**Move を名指せれば、技法が**自分の**状況に適用されるか、自分にない問題を解くものか判断できる**。

> [!TIP]
> 提案された最適化が 3 つの move のどれにも cleanly map しないなら、**疑え**。**コスト節約コスチュームを着た品質変更**か、**エンジニアリングに dressed up したスタイル好み**である可能性が高い。

## 7 つの技法ファミリ

学ぶ価値ある技法は **7 ファミリ**に clusters。**最小努力で最大リターンのものを最初**に、概ね採用順で:

### 1. Output control (出力制御)

**単一最高 per-token-ROI lever**。**1 度モデルに「terse」と言う**: preamble スキップ、N 語で回答、固定 JSON スキーマ返却、要求なき限り説明省略。**その 1 指示が会話の全将来レスポンスをシフト**、しかも**より高価バケットをシフト**。出力トークンは入力より高価で、**1 つずつ生成されるのでレイテンシも支配**。

正典例: 「**要求なき限り 1 文で回答**」「**このスキーマに合致する JSON で応答、散文なし**」のような単一システムレベルルール。**1 行に収まるルールが、chatty ワークロードで出力ボリュームを半分以上削減**、有用内容の損失なしに。

Output control は**他全てと複利**。**追加コストほぼゼロ、ほぼ全将来変更で生存**。**測定最簡単な勝利の 1 つ** — 第13章の使用メトリックに出力トークンが cleanly 現れる。

---

### 2. Context scoping (コンテキストスコーピング)

**常時オン指示を条件付き・スコープ付き・on-demand ロードに置換**。1 つの巨大指示ファイルを全リクエストに送る代わりに、**多数の小ファイルに分割し、関連ワークフローがアクティブな時のみロード**。第12章の **minimum sufficient context** 原則。

正典例: 単一の数千トークン指示文書 (チームが想像したあらゆるワークフローをカバー) で始まり、**小コアファイル + ダース単位の小・名前付き・on-demand ロードファイル**で終わったプロジェクト。**Per-request input が大幅低下、回答品質も通常改善** — モデルが無関係ルールを sift しないから。

Scoping は**1 度の編集ではなく、継続規律**。指示ファイルは「念のため」新ルールが追加されると bloat に向かって drift。**Scoping 習慣** (四半期レビュー、ruthless 剪定) **が勝利を保つ**。

---

### 3. Prompt compression と phrasing 規律

**より少ないトークンで同じことを言う技** — **しばしばより明確に**。**Declarative > Imperative** (「**Output: JSON**」が「常に JSON 形式で出力するよう必ずしてください、ありがとう」に勝つ)。**Politeness padding 削除**。**死んだルール** (もう起きないシナリオをカバーする指示) 削除。**繰り返しルールを 1 つに collapse**。

正典例: hand-grown 指示ファイルを読み、**全行を声に出し、各々に問う**: **これはまだ true？まだ必要？半分の語で言える？** 不快な仕事、行あたり節約は小さく見えるが、**長寿命プロンプトへの累積効果は実質**。

Compression には**見逃しやすい品質副作用**: **Terser 指示はより信頼的にフォロー**される。礼儀的英語段落で 5 つを頼まれたモデルはしばしば 3 つ; **同じ 5 つを命令動詞の番号付きリストで** = より頻繁に行われる。

---

### 4. Model 選択と tiering

**全タスクが最強モデルを必要としない**。**易タスクを小さく安いモデルにルーティング、フロンティア tier を真に必要なタスクに予約** = **最大ポートフォリオレベル勝利の 1 つ**、**段間価格差が大** (しばしば 1 桁以上、どちら方向にも) だから。

正典パターン: **router または「auto」モード** — 軽量初パスがリクエストを分類、シンプルを小モデルに dispatch、**真に難しいケースのみ escalate**。**多くのワークロードは trivial リクエストの長尾**を持つ — フォーマット、分類、短書き換え — **小モデルが大モデルと区別不能**に処理、コスト fraction で。

Tiering は**レイテンシでも払う**。**小モデルは通常速い**、易ケースが**snappier に感じ、かつ安い**。トリックは**正直な評価**: 「**良いっぽい**」と仮定しない — **コミット前に実トラフィックで測定**。

---

### 5. Caching 規律

**安定 prefix が prefix キャッシュにヒットするようプロンプト設計**。プロバイダが前リクエストで見たのと同じ opening トークンを見れば、**dramatically 少なく課金、または仕事を完全スキップ**。**構造ルールは単純**: **安定内容先頭、揮発内容末尾**。

正典アンチパターン: **タイムスタンプ・リクエスト ID・per-user 変数をプロンプト最先頭**。**位置 1 のその単一 moving トークンが続く全てのキャッシュを invalidate**、**near-free だった prefix を full-price に変える**。

Cache-friendliness は、**長く安定なシステムプロンプトを再利用するワークロード** (ほとんどのプロダクション) で**実効入力コストを 1 桁削減**しうる。**レイテンシも削減**、cached prefix 処理は cold より遥かに速いから。

---

### 6. Tool / MCP 規律

モデルにツールアクセスを与える時 (function call、MCP サーバ、プラグイン)、**全ツールの定義・名前・記述・パラメータスキーマが、使われるかに関わらず全リクエストで送られる**。**Bloated カタログは毎ターン入力を黙って膨張**、第14章の通り、**モデルがより多くのオプションを weigh するのでツール選択品質も劣化**。

正典例: 探索中に 6 ツールサーバ接続、使うのを止めた 4 つを切断せず、**理由不明で per-request input が登るのを見たチーム**。修正: **定期 audit、未使用削除、everywhere に exposing せず必要なワークフローにスコープ**。

**小焦点絞られたツールセットは per-step 安く、ツール選択判断もより良い**。**コストと品質が逆に動かず一緒に動く稀な最適化**。

---

### 7. Workflow 選択

**最高価最適化ミスはタスクに間違った対話モードを使うこと**。**素早い ask** (単一リクエスト・単一レスポンス) = LLM 仕事の最安単位。**Edit モード** (特定変更を target) = 中程度に高価。**Full agent ループ** (モデル計画、ツール呼び、結果観察、反復) = **同じ outcome に 5〜25 倍コスト**しうる、**全ステップが成長コンテキストを再送し、より多く出力するから**。

正典ミス: **タスクが one-shot 質問でも、reflexively に agent loop に手を伸ばす**。正典修正: **モードをタスクに合わせる**。「この関数何する？」 = ask モード、「このファイル全部でこの変数 rename」 = edit モード、「テストスイートが火曜日に失敗し始めた理由を調査して修正提案」 = agent モード。

Workflow 選択が採用順で最後 = **最重要でないからではなく** (リクエストあたり最 impactful 単一判断かも) — **他 6 ファミリが先に整っていることを必要とするから**。**プロンプト・コンテキスト・ツール・モデル選択が規律的になれば、正しいワークフロー選択が他全てを複利する move**。

---

## どこから始めるか

これらファミリを現れる順で採用すれば、**最初 2 つで意味ある節約、最初 4 つで実質節約**を見ます。

| 順 | 📚 ファミリ | 努力 | 典型インパクト |
|-------|----------|--------|----------------|
| 1 | Output control | 極低 | 高 — 全レスポンスに即時影響 |
| 2 | Context scoping | 低〜中 | 高 — 全リクエストの入力を削減 |
| 3 | Prompt compression | 中 | 中 — scoping と複利 |
| 4 | Model tiering | 中 | 高 — 段間価格差大 |
| 5 | Caching 規律 | 低 (構造的なら) | 安定ワークロードで高 |
| 6 | Tool 規律 | 低 | 中 — 品質勝利でもある |
| 7 | Workflow 選択 | 継続 | 判断あたり高 |

**最初 2 ファミリは採用ほぼ無料、即時 payback**。**次 2 つは測定仕事を要するが最大ポートフォリオ節約を unlock**。**最後 3 つは polish** — 重要だが、**基盤が整って初めて意味を持つ**。7 ファミリを超えると、**収穫逓減に素早く当たる**。

## 何から始めない

**驚くほど多くのオンライン助言**が、**reasoning-effort ダイアル、temperature や top-p のような sampling パラメータ、トークンレベル decoding トリックを弄ることから始めよ**と言う。**抵抗せよ**。上の 7 ファミリと比べ、**それら lever は小効果で、コスト節約より品質を害する方法で over-tune しやすい**。

**便利な経験則**: **提案された最適化が、システムプロンプト・ツールカタログ・モデル選択を audit する前に新ダイアル学習を要するなら、それは間違った出発点**。**大勝利は構造に住む、パラメータではない**。**構造仕事が完了し、残コストの実所在を測定が示してから、ダイアルに手を伸ばす**。

> [!WARNING]
> **早すぎるパラメータチューニングは、チームが「最適化した」と自分を確信させながら実節約をテーブル上に残す最一般的方法の 1 つ**。**構造を先に audit**。

## 短い結びの reflection

このガイドの目的は**トークン最適化エキスパートを作ること**ではなかった。**単位に流暢にすること**。**流暢さは、上の全技法と未発明の全技法の前提条件**。なしには、最適化助言は**民間知恵** — 誰かが 1 度効いたと聞いたトリックリスト。**ありで、読む全助言が正しい量の懐疑と正しい量の熱意で着地**。問える: **どの move か、保守コストは、私のワークロードに何を仮定、私自身の測定が confirm するか？**

**最適化は、会計のように、ほとんどが帳簿を読めることについて**。**算術は難しくない**。**規律は、帳簿を保つこと、誠実に読むこと、それが言うことに行動すること**。**今あなたはそれができる**。

## Going deeper: 四半期トークン audit

**最適化は 1 度プロジェクトではない**。**セキュリティレビューや依存性アップグレードと同じく、繰り返しケイデンスの組織実践**として扱う。

**早期勝利を undo する力**がいくつか共謀する。**モデル価格が変わる**、時に下方向で**どの段がどのタスクに正しいかを変える**。**新キャッシング機構が現れ**、去年中立だったプロンプト構造に報酬 (または罰)。**指示ファイルは well-meaning な貢献者が「もう 1 つルール」を追加するので bloat に drift**。**ツールカタログは、人々がサーバ接続し切断を忘れるので成長**。**Workflow が ask-mode から agent-mode に黙ってシフト**、1 チームずつ、請求が気付くまで。

**実用的対抗策 = 四半期トークン audit**:

```
   ┌──────────────────────────────────────────────────┐
   │   QUARTERLY TOKEN AUDIT                          │
   ├──────────────────────────────────────────────────┤
   │ 1. 前四半期使用メトリックを pull                  │
   │ 2. Top 3 コスト drivers を特定                    │
   │ 3. システムプロンプトを声に出して再読            │
   │ 4. ツールリスト — 90 日未使用を削除               │
   │ 5. モデル段 vs タスク難度を再 check              │
   │ 6. Cache-friendly prefix 構造を verify            │
   │ 7. 次四半期に走らせる実験 1 つを記録             │
   └──────────────────────────────────────────────────┘
```

**Audit は重い必要なし**。**プロジェクトあたり四半期 1 時間で、回帰が予算 item になる前に捕まえるのに十分**。**完璧状態到達が目的ではない** (存在しない) — **確立した規律から drift しすぎないようシステムを保つこと**が目的。

**この習慣を構築すれば、7 ファミリはチェックリストを止め、あなたの働き方になる**。

## よくある誤解 / FAQ

### Q1. 「7 ファミリを全部一度にやるべき？」
A. **No**。順に採用し、各々の効果を**測定**してから次へ。一度に全部 = どれが効いたか分からない。

### Q2. 「最適化は本書出版時の話で、未来のモデルには関係ない？」
A. **3 つの move は不変**。新モデル・価格・機構が来ても、**フレームワークは生存**。表面トリックだけ変わる。

### Q3. 「reasoning effort や temperature は触る価値ゼロ？」
A. **構造仕事完了後**は OK。**順序が重要**。ダイアル先行は典型ミス。

### Q4. 「small チームで quarterly audit はやりすぎでは？」
A. **30 分でも価値あり**。Cadence > 厳密性。「絶対に忘れない」が要点。

### Q5. 「次に学ぶべき具体技法は？」
A. **Output control から** — 最低努力、最高即効性。1 ルール追加して 1 週間測定。

## まとめ

- **全最適化は 3 つの move の 1 つ**: より少なく送る、より少なく受ける、トークンあたり少なく払う。**技法評価前に move を名指せ**
- **7 技法ファミリ**がほぼ学ぶ価値全てをカバー: output control、context scoping、prompt compression、model tiering、caching 規律、tool 規律、workflow 選択
- **概ねその順で採用**。最初 2 つはほぼ無料。次 2 つが最大節約 unlock。最後 3 つが polish と複利
- **Reasoning-effort ダイアルや sampling パラメータから始めない**。構造修正がパラメータ修正を矮小化
- **最適化は繰り返し実践、1 度プロジェクトではない**。Audit cadence なしにはコストが drift して戻る
- **単位 (トークン) への流暢さ**が、全将来最適化技法を読める legibility にする。**この流暢さがガイドの目的だった**
